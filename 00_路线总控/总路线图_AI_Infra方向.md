# 总路线图：AI Infra / 云原生资源调度方向

## 一句话定位

**用 RAG/Agent 构造真实 AI workload，用后端系统承载它，用调度器管理它，用监控压测评估它，再逐步迁移到云原生和推理服务平台。**

## 为什么不是纯算法路线

当前更适合从工程系统切入，而不是一开始主攻大模型算法、CUDA 算子或纯理论优化。原因：

- 已有软件工程、后端、数据处理、交付验证和 AI 工具协作经历。
- 市场对 AI 落地工程、RAG/Agent 平台、LLMOps、推理服务和云原生调度需求明确。
- 顾琳老师方向中的云原生、边缘计算、资源调度、性能优化与这条路线高度贴合。
- 通过小型系统和实验可以同时补工程能力与科研能力。

## 总链路

```text
RAG/Agent 应用
-> AI Workload 抽象
-> 任务队列与调度
-> 监控压测与指标
-> Docker/K8s 云原生部署
-> vLLM/Triton 推理服务
-> 金融投研 AI 场景
```

## 核心问题

AI 请求来了以后，系统如何排队、调度、执行、监控，并在延迟、吞吐、成本、稳定性和公平性之间做权衡？

## 能力分层

### 应用层

- 企业知识库 RAG
- 可控 Agent 工作流
- 金融投研/科研资料分析场景

### 服务层

- FastAPI / 后端 API
- 数据库、缓存、异步任务
- 权限、日志、异常处理

### 调度层

- FIFO / Priority / SJF
- 用户配额、限流、失败重试
- 成本感知和负载感知调度

### 平台层

- Docker / Docker Compose
- Kubernetes 基础
- Kueue / Volcano / HPA
- Prometheus / Grafana

### 推理系统层

- vLLM / Ray Serve / Triton
- batching、KV cache、tokens/s
- TTFT、TPOT、P95/P99

### 科研层

- 研究问题定义
- baseline 设计
- 变量控制实验
- 论文阅读与局部复现
- 实验报告和写作

## 目标成果

- 两个月：做出可演示的 RAG/Agent + Scheduler 小平台。
- 四个月：加入压测、监控、异步任务和实验对比。
- 八个月：加入 Kubernetes 和 AI workload 调度思想。
- 十二个月：接入 vLLM 或模拟推理服务，形成科研训练项目。
