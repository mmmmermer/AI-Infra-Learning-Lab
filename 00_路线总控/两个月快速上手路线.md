# 两个月快速上手路线

## 目标

在两个月内做出一个能演示、能形成简历表达草稿、能继续科研化的项目雏形：

**AI Workload Platform for RAG/Agent Services v1**

## 使用前提

这条路线不是零基础第一入口。如果工具链、Python、pytest、路径、虚拟环境和小项目结构还不稳，先走 [[00_路线总控/基础薄弱版12周执行路线|基础薄弱版 12 周执行路线]] 的第 1-2 周。

最低前置：

- 已读 M00/M01 的核心章节。
- 能独立创建 `.venv`、安装依赖、运行 pytest。
- 能写一个 `task_sorter` 小项目并保存实验记录。
- 能解释 pending / queued / running / succeeded / failed 这类任务状态的含义。

不满足这些前置时，不建议直接进入本路线；否则会变成“照着搭平台”，但看不懂为什么这样设计。

## 项目闭环

```text
文档上传
-> 解析切分
-> 向量检索
-> RAG 问答
-> Agent 生成报告
-> 请求进入任务队列
-> 调度 worker
-> 记录延迟/吞吐/失败率
-> Docker 一键启动
```

## 8 周安排

### 第 1 周：后端与任务模型

- FastAPI 最小服务
- `POST /tasks`
- `GET /tasks/{task_id}`
- `GET /workers`
- `GET /metrics`
- 任务状态：pending / queued / running / succeeded / failed / retrying / cancelled

产出：后端服务骨架和 3-5 个测试。

### 第 2 周：调度器 v1

- FIFO
- Priority
- SJF
- worker 分配
- 等待时间和完成时间统计

产出：Mini Scheduler v1。

### 第 3 周：RAG v1

- 文档上传
- 文档切分
- embedding
- 向量检索
- 回答带引用

产出：最小企业知识库问答。

### 第 4 周：RAG 请求接入调度器

- RAG 请求进入任务队列
- 调度后执行检索和回答
- 记录任务耗时
- 简单任务列表页面或 API 查询

产出：RAG workload 调度闭环。

### 第 5 周：Agent 工作流 v1

- 可控 Agent 状态流
- 检索资料
- 生成摘要
- 生成报告
- 人工确认节点先用简单状态模拟

产出：可控 Agent demo。

### 第 6 周：异步任务与缓存

- PostgreSQL task + outbox
- dispatcher + Redis Streams consumer group
- ACK/reclaim/fencing worker
- RQ/Celery 选读对比，不作为 P03 当前主线
- 文档解析异步化
- 报告生成异步化
- 简单缓存

产出：更接近真实后端系统。

### 第 7 周：Docker 与基础监控

- Dockerfile
- docker compose
- 结构化日志
- 基础 metrics endpoint
- 请求耗时统计

产出：本地一键启动。

### 第 8 周：实验与包装

- 压测脚本
- FIFO / Priority / SJF 对比
- P95 延迟、吞吐、队列长度图表
- README
- 架构图
- 项目复盘

产出：可展示项目包。

## 两个月内先不做

- CUDA / 算子优化
- 完整 Kubernetes
- 大模型微调
- 复杂多 Agent
- 从零写模型服务
- 复杂 SpringCloud
