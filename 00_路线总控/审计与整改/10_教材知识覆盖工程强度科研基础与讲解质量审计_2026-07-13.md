## 一、最终判断

当前路线方向正确，已经形成若干质量较高的教学锚点，但全库仍应标记为：

```text
结构化教材草稿 / 局部达到 content-reviewed / 尚未整体达到发布候选
```

这不是因为尚未进行学习者验证，而是作者侧内容本身仍存在四类可独立成立的问题：

1. **知识覆盖不连续**：操作系统、网络请求路径、Python 并发、API 生产契约、分布式系统、数据库基础、可观测性闭环和科研方法没有形成连续的先修链。
2. **路线承诺高于教材正文**：M10 的名称高于真实 vLLM/Triton 实验，F01-F05 的路线关键词高于正文数学与金融覆盖，M11 的单一 RQ01 统计章不足以代表完整研究方法。
3. **概念强度与可执行强度错位**：M06 的 outbox、CAS、lease 设计较强，但 reference 仍有 stale worker 与可靠领取缺陷；M07、M09 的 reference 又明显强于正文讲解。
4. **教学表达不均匀**：部分章节具备直觉、推导、固定输入、中间状态、输出、反例和练习；部分章节仍像字段表、资料汇编或工程设计说明，读者知道“有什么”，却不一定知道“为什么、如何判断、如何失败、如何验证”。

### 1.1 当前最强的内容

- M01 第 2-3 章：从业务需求到数据模型，再到可测试函数，概念递进自然。
- F01 第 1、4、6 章：固定输入、明确计算口径、展示中间值并限制结论外推。
- M03 第 4 章：embedding 与 cosine 有可手算的最小例子。
- M05 第 7 章及 E05/P01：固定 workload、P95/P99、利用率、不变量和边界构成较完整的调度模拟闭环。
- M08 第 7.5-7.6 章：开放环/闭环、offered/achieved load、coordinated omission、阶段和饱和点讲解较强。
- M09 第 2 章：observe-break-diagnose-fix-cleanup 的故障学习路径清楚。
- M10 第 2、4 章：TTFT/TPOT/ITL 口径与 GQA/MQA KV cache 计算较严谨。
- M11 第 5.6 章：实验单位、伪重复、CRN、配对差值、效应量、区间、bootstrap 和 artifact 复算已达到较好的窄域统计推断水平。

### 1.2 最影响自学体验的根因

最影响自学体验的不是单纯缺少练习，而是**同一条路线上的教学承诺不一致**：

```text
有的章节是教材：解释直觉 -> 展开机制 -> 固定示例 -> 反例 -> 独立练习 -> 验收。
有的章节是说明书：术语 -> 字段表 -> 架构图 -> 完整答案 -> 学习检查。
```

后者会制造三种“看懂错觉”：能复述名词但不能诊断故障；能复制完整代码但不能独立变式；能跑 reference 但不能解释结果是否可信。首轮优化必须优先修复这种不均匀，而不是继续增加工具名和资料链接。

## 二、审计方法与证据边界

### 2.1 检查层次

本轮同时使用四层证据：

| 层次 | 检查内容 | 能证明什么 | 不能证明什么 |
|---|---|---|---|
| 路线层 | 路线承诺、模块依赖、状态和目标岗位能力 | 应教什么、先学什么 | 实际是否讲清楚 |
| 章节层 | 202 章人工阅读与结构矩阵 | 知识是否出现、解释是否完整 | 学习者是否真正掌握 |
| 实验层 | E00-E10、GF、P01、P03、RQ01 | 作者提供的任务是否有可执行证据 | 学习者已完成、生产可用 |
| 来源层 | 官方文档、canonical source、版本与状态 | 事实和版本可校准 | 链接存在就等于教材质量高 |

### 2.3 工程、科研和金融强度等级

#### 工程等级

| 等级 | 判定标准 |
|---|---|
| E0 | 能解释术语、组件位置和最小代码片段 |
| E1 | 有确定性、单进程或单组件实验，能观察成功与失败 |
| E2 | 有可靠的多组件服务，覆盖状态、并发、失败恢复与契约测试 |
| E3 | 覆盖部署、安全、可观测性、容量、故障注入和运行边界 |

#### 科研等级

| 等级 | 判定标准 |
|---|---|
| R0 | 描述现象并提出主张 |
| R1 | 固定输入、环境和指标，可复现描述性实验 |
| R2 | 多次重复、受控比较、效应量、区间、稳健性和排除规则 |
| R3 | 有系统相关工作、效度威胁、伦理/治理、可复用 artifact 和可辩护外推 |

#### 金融计算等级

| 等级 | 判定标准 |
|---|---|
| F0 | 术语与单个公式 |
| F1 | 单资产或单模型的可复算计算，口径和边界明确 |
| F2 | 多资产、时间有效、成本/不确定性敏感并有稳健验证 |
| F3 | 研究或生产级治理，覆盖模型风险、数据治理、审计和压力情景 |

这些等级只用于定位缺口。模块可以在某一子领域达到 E2，同时在另一子领域仍停留 E0。

## 四、必须先修的 P0 问题

以下问题会直接教出错误的安全或可靠性模型。在解决前，不应继续以“完整工程闭环”表述相关章节和实验。

### P0-1：M03/P03 RAG 示例仍允许客户端自报身份和权限

- M03 第 8 章的请求示例仍包含客户端 `user_id` 和 `permission_group`。
- M03 第 7 章的代码只展示对内存 chunk 列表的过滤，不能证明权限在向量/关键词检索、打分、缓存和 prompt 之前由存储层强制执行。
- 正确契约应是：认证层建立 server-owned principal，服务端派生 tenant/user/effective ACL，检索查询本身携带强制过滤条件，缓存键和日志也绑定授权范围。

验收不能只看最终答案没有泄露，必须验证未授权文档从未进入候选集、打分集、缓存和 prompt。

### P0-2：E04 身份、审批人和状态契约与 M04 强设计冲突

- E04-01 接收 `user_id/permission_group` 作为工具授权输入。
- E04-03 接收客户端 `reviewer_id`，并把拒绝写成 `cancelled/human_rejected`；M04 第 6 章却要求审批人来自 principal，并使用 `failed/approval_rejected`。
- 应统一 principal、tenant、资源范围、审批对象哈希、策略版本、CAS 和错误码，再提供可执行 runtime。

### P0-3：E06 SQLite reference 缺 stale-worker fencing

`complete_task()` 和 `fail_task()`只按 `task_id + running` 完成状态变更，没有校验 worker owner、claim version 或 fencing token。旧 worker 在 lease 过期并被新 worker 重领后，仍可能提交新 worker 的任务结果。

修复后必须新增：旧 owner complete、旧 owner fail、重复 finalize、lease 过期后重新 claim、heartbeat 与 finalize 竞态测试。

### P0-4：P03 使用破坏性 `BLPOP` 后才做数据库 claim

worker 在 Redis `BLPOP` 删除消息后才尝试数据库 claim。若进程在 pop 与 claim 之间崩溃，消息已经消失，task 仍是 queued，outbox 已是 published，running lease reconciliation 也无法发现它。

应改为 Redis Streams consumer group、processing-list + ack，或数据库可靠队列；在修复前不得声称此路径提供至少一次交付。

### P0-5：M10 发布范围高于证据

E10-01/E10-02 是不加载模型、不使用 GPU、不执行真实 batching/KV 分配的合成模拟；E10-03 仍为 blocked，且没有 Triton 实验。因此当前材料不能被表述为完整“vLLM/Triton 工程教材”。

可选处理只有两种：

1. 完成真实 vLLM 服务、benchmark、取消/过载和资源测量闭环；Triton 另做明确选修实验。
2. 在真实实验完成前，把主模块准确命名为“推理 workload 与指标导论”，将 vLLM/Triton 定位内容标为 design-note/reference。

### P0-6：P01 对单次固定 fixture 使用“显著”措辞

P01 的 worker 数量和综合实验记录把单次确定性 fixture 的数值变化写成“显著降低 P95/P99”。这里没有重复样本、区间或显著性检验，“显著”会被理解为统计证据或可外推效应。

应统一改为“在该固定 fixture 中，P95/P99 数值下降”，或明确引用 RQ01 的多 seed 配对结果及其适用范围。README、实验记录、项目总结和简历稿应同时搜索，不能只修一个句子。

### P0-7：金融路线完成口径与状态表冲突

`金融工程与量化金融支线路线.md` 宣称“已完成学习前建设”“可自学性合格”，但状态表仍将 F00、F01、F03、F04、F06 标为 `draft/unverified`。结构、入口和资料齐备不能推出内容已经审校或可自学。

应改为“结构与入口已建立，内容按模块分批审校”，并把 `content-reviewed`、`reference-verified`、`instructional-ready`、`learner-validated` 分轴记录。

## 六、讲解与文字质量专项判断

### 6.1 “易懂”不等于“少写公式”

自学教材中的易懂应由六件事共同构成：

1. 先说问题和决策，再给定义。
2. 给直觉模型，同时明确直觉会在哪些边界失效。
3. 使用固定、足够小但不失真的输入。
4. 展示关键中间状态，不只给最终代码或最终数字。
5. 对反例解释错误机制和修复后的观察值。
6. 用不同输入做独立练习，答案与题面分离。

当前最常见的解释缺口是：字段表和架构图很多，但缺一条请求、一条任务、一组现金流或一个 seed 从输入到输出的完整追踪。后续每个高风险主章至少增加一个“固定输入 -> 中间状态 -> 实际输出 -> 解释 -> 反例”的 worked example。

### 6.2 需要系统处理的五种弱写法

#### 弱写法 A：名词列表代替机制

典型位置是 M07-M10 的工具定位章。补写时必须回答：组件维护什么状态、输入输出是什么、失败发生在哪里、哪些指标能区分故障。

#### 弱写法 B：伪代码代替可运行教学

M04、M08 tracing 和 M10 serving 尤为明显。伪代码可以解释机制，但必须标为 `pseudocode`，并与最小 runnable reference、运行命令、输出和失败注入配对。

#### 弱写法 C：完整答案紧贴练习

M04、M08、M10 以及 M05 多处练习后立即展示完整实现。应把正文 worked example、独立练习和答案册分开；正文只保留必要提示与验收接口。

#### 弱写法 D：检查表只验证复述

“能解释”“能说明”适合概念检查，但不能独立承担工程验收。至少加入一种产物型任务：修改输入、注入故障、预测输出、比较两种方案、解释异常 artifact 或修复失败测试。

#### 弱写法 E：模块末尾统一来源代替就近证据

安全、统计、版本、金融公式和产品行为应就近绑定 canonical source。M04、M06-M08、M10、F07 等模块的教学章几乎没有章内来源，维护时很难判断哪条主张受哪个版本约束。

### 6.3 章节长度和渐进披露

M05 虽已拆成 13 个文件，但章节仍约 349-845 行，且 509 个代码围栏均未标角色。首轮不应机械删减，而应按学习任务拆为：

```text
概念主章 -> worked example -> 观察实验 -> 独立变式 -> 答案/参考实现 -> 延伸阅读
```

一个主章只承担一个核心决策。字段参考、完整 API、答案和扩展设计移到 reference/appendix，不再挤在概念解释之间。

目标：把内容强弱差异压到可接受范围。

- design-note、reference、appendix 保持各自职责，不强行补成教程。

## 八、统一章节改造模板

每个需要补强的 `instructional` 主章按以下顺序修订：

1. **问题与决策**：本章解决什么工程或研究判断。
2. **目标与先修**：使用可观察动词，列出必需环境和上游概念。
3. **最小心智模型**：定义、机制、状态、边界和一张必要图。
4. **固定 worked example**：输入、设置、中间状态、实际输出和解释。
5. **机制型反例**：错误现象、根因、危险性、修复和回归检查。
6. **观察实验**：允许读者运行给定 reference，重点观察变量关系。
7. **独立变式**：更换输入或约束，正文不直接暴露完整答案。
8. **验收**：命令、断言、artifact 和不变量；不能只写“能理解”。
9. **边界与非目标**：明确教学简化、未实现能力和不能外推的结论。
10. **来源与下一步**：就近 canonical source，下一章依赖明确。

代码围栏必须标明角色：`runnable`、`fragment`、`pseudocode`、`config`、`command`、`output` 或 `expected-error`。产生持久状态的教程必须包含 cleanup；涉及权限、成本、数据删除或外部服务的操作必须在动作前给出 warning。

## 九、验收门禁

### 9.1 单章门禁

- 章节类型、受众、目标、先修、范围和非范围一致。
- 核心主张可追溯，版本和运行环境明确。
- worked example 可由作者执行或复算，输入与输出不依赖未声明文件。
- 反例真实触发目标故障，修复后有回归观察。
- 独立练习更换输入或约束，答案不紧贴题面。
- 安全、统计、金融和状态机结论通过领域复核。

### 9.2 模块门禁

- 章节依赖不存在倒置或隐藏跳步。
- 教材、实验、reference 和项目使用同一身份、状态、字段和错误契约。
- 模块声称的工程等级由至少一个端到端 artifact 支撑。
- 设计稿、blocked 实验、作者 reference、学习者完成和科研结论分别标记。

### 9.3 路线发布门禁

- Wave 0 P0 全部清零。
- 路线关键词与正文首次教学覆盖一致。
- M00-M02 先修链、M03-M10 工程链、M11 科研链和 F00-F07 金融链无断点。
- 自动检查只做语法、链接、结构和回归；最终内容状态由人工语义复核决定。
