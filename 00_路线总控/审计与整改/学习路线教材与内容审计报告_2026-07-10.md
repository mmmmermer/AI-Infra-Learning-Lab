## 一、结论摘要

当前学习库的方向设计是成立的：

```text
RAG / Agent workload
-> 后端与异步任务
-> 调度、监控、容器与 Kubernetes
-> vLLM / Triton 推理系统
-> 科研实验
-> 金融工程与 AI workload 交叉
```

教材的概念边界、官方资料比例、风险提示和非投资建议边界总体较好，不需要推倒重建。

但当前内容不能整体标记为“已完成教材”或直接作为唯一标准开始学习。更准确的状态应是：

> 结构完整、资料较好，但仍处于需要工程验证和专业校订的教材草稿阶段。

主要原因：

1. 多个实验只有设计说明，无法端到端运行。
2. 部分代码语法正确，但业务语义、状态共享、权限和统计方法有错误。
3. Python、vLLM、Kubernetes 等缺少统一版本与运行环境基线。
4. 教材宣称覆盖的主题与实际章节、实验存在差距。
5. AI 参考项目和本人完成状态没有形成可靠的内容边界。
6. M05 等模块篇幅严重失衡，增加了学习与维护成本。

建议在开始正式学习前先完成 P0 内容修订。

## 三、最高优先级问题

### P0-1 Python 版本基线失效

当前默认 `python` 命令仍指向 Python 3.8.6，同时系统另有 Python 3.13。教材仍把 3.8/3.9 作为可继续使用或验收的环境。

- Python 3.8 已于 2024-10 EOL。
- Python 3.9 已于 2025-10 EOL。
- Python 3.10 将于 2026-10 EOL。

影响：新版本 FastAPI、vLLM、Locust、OpenTelemetry 等无法在旧环境可靠安装；不同终端可能执行不同 Python。

修订要求：

- 全库统一 Python 3.12 或 3.13。
- 明确 `py -3.13`、虚拟环境和解释器检查方式。
- P01/P03 更新 `requires-python`。
- 删除以 3.8 为验收目标的内容，仅保留迁移说明。
- 建立锁文件与环境清单。

### P0-2 M02 / E02 不是可累计运行的 FastAPI 项目

三个实验分别创建独立 `FastAPI()`：

- E02-01 固定返回 `task-001`，不保存任务。
- E02-02 使用新的空 `TASKS`。
- E02-03 再创建一个 app 和静态 `METRICS`。

因此无法实现“创建任务 -> 查询状态 -> 指标变化”的教学闭环。

修订要求：提供一个单一项目骨架、单一 app、共享 repository/router，并让 E02-01 至 E02-03 在同一代码库上累计演进；补依赖、启动命令和端到端测试。

### P0-3 M03 / E03 的 RAG 实验结论不可复现

当前允许关键词匹配、手工排序、模板回答和模拟 generation latency。短自造语料不足以证明 chunk size、top-k 对检索质量和延迟的影响。

修订要求：

- 固定小型 corpus 和独立黄金 query/evidence 集。
- 使用确定性 BM25 或锁定版本的 embedding。
- 固定切分器、模型、维度、距离度量和索引版本。
- 实际计时，不填写模拟延迟。
- retrieval 与 generation 分开评价。
- 记录 recall@k、MRR/nDCG、citation precision/recall，而不是只主观判断答案好坏。

### P0-4 RAG 权限过滤顺序不安全

E03-03 把检索前过滤和检索后过滤并列为可选策略。检索后过滤会使无权限文档进入候选、重排、缓存或日志，还会造成先取 top-k 再过滤后的授权召回损失。

修订要求：权限条件必须在检索数据层强制前置；post-filter 只作为故意失败的反例与纵深防御，不得作为正确实现。

### P0-5 M06 缺少数据库与队列一致性模型

教材流程 `create_task -> enqueue -> mark_queued` 会产生双写不一致：

- 数据库有任务但没有入队。
- 消息已入队但状态仍 pending。
- worker 执行后状态更新失败。

修订要求：至少讲清 transactional outbox、pending reconciliation、幂等键、compare-and-set、lease/visibility timeout、heartbeat、重复投递和 worker 崩溃恢复。

### P0-6 M04 会话所有权检查被错误降级

Agent 实验把 `load_for_user(user_id, session_id)` 的所有权校验列为后续增强，意味着基础版本可能仅凭 session_id 读取状态。

修订要求：所有权和租户隔离必须属于最小安全基线；工具参数使用 Pydantic/JSON Schema 严格校验，内部日志保留异常上下文。

### P0-7 E09 / E10 实验文件实际不存在

Kubernetes 与推理服务索引列出 E09-01 至 E09-04、E10-01 至 E10-03，但目录内只有索引文件。

修订要求：正式学习前补齐环境、命令、配置、预期输出、清理步骤与失败诊断；在此之前统一标记为“实验候选”，不能显示为可执行实验。

### P0-8 E07 / E08 依赖尚未实现的 P03

P03 主页明确 API、调度器、RAG、Agent、Compose、metrics 和压测仍未实现，而多个实验直接要求完整 worker 状态流和监控数据。

修订要求：先完成 P03 v0.1 内存版，再做 Docker/worker；数据库和 Redis 版本完成后再进入可观测性实验。当前页面应标为“设计稿”。

### P0-9 P01 / RQ01 研究结论基础不足

- SJF 使用的 estimated duration 同时被当成真实 duration，属于 oracle SJF。
- 只有 52 条固定任务，没有多随机种子和误差模型。
- 52 个样本下 P99 基本等于最大值。
- 队列长度实现可能把刚开始运行的任务仍计入队列。
- 没有置信区间、效应量或公平性分析。

修订要求：增加预测误差、重尾服务时间、突发到达、多个随机种子、common random numbers、置信区间、效应量、公平性和成本指标。当前结果只能称为教学演示，不能称为科研结论。

### P0-10 金融时间序列代码会跨资产计算收益率

F02 对 DatetimeIndex 去重会把同一天的其他 symbol 误删；随后对整列 `pct_change()` 会在资产切换处计算错误收益率。

修订要求：唯一键为 `(symbol, date)`，使用 MultiIndex、groupby 或宽表；补两资产同日数据和资产边界测试。

### P0-11 金融分类标签存在泄漏和假标签

GF07 先执行 `(future_return > 0).astype(int)`，末行 NaN 被变成 0，之后 `dropna(label)` 无法删除；测试集很小，假标签会严重扭曲指标。训练边界也可能使用测试期收益形成标签。

修订要求：先删除 future return 为空的行，再生成标签；按预测 horizon 留 gap；验证集负责选特征和参数，最终测试集保持未触碰。

### P0-12 金融 workload 契约无法重放

F08/GF08 的 input 只有 data source 名称、日期或指标名称，没有不可变数据引用、数据 hash、代码版本、依赖版本、随机种子和 artifact URI。

修订要求：增加 `schema_version`、`input_artifact_uri`、`sha256`、`as_of_time`、`data_version`、`code_ref`、`dependency_lock_hash`、`random_seed`；将输入 `metrics` 改名为 `requested_outputs`，与系统性能指标分开。

### P0-13 久期和凸性实验没有计算久期和凸性

GF05-02 只做收益率上移/下移后的重新定价，没有计算 Macaulay duration、modified duration 或 convexity，但验收仍可通过。

GF05-01 又暴露 payment frequency，却只实现整年年付息；YTM 固定在 `[0, 0.5]` 搜索，不支持负收益率且不验证是否括住根。

修订要求：实现并区分久期与凸性；频率进入现金流；补 clean/dirty price、应计利息、day-count、复利口径；YTM 求解失败必须显式报错。

## 四、重要内容缺口

### 基础与后端

- M00 学习地图中的 uv、requirements、环境变量、端口、进程/线程、数据库/缓存/队列直觉没有完整教材和验收。
- M01 教类型标注但没有 Pyright/mypy 验收；JSON、CLI、logging、模块拆分缺对应实验。
- M02 学习地图包含异步、后台任务和鉴权，但正文和实验没有形成闭环。
- 跨模块任务状态混用 `done` 与 `succeeded`，需要共享枚举和状态机。
- submit time 不应由真实 API 客户端决定；服务端应生成 UTC accepted/created time，模拟时间另命名。

### RAG 与 Agent

- M03 缺相似度度量、embedding 版本/维度、索引重建、词法/混合检索、结构化 chunk 和 claim-level citation。
- M03 资料过度依赖 LangChain/LlamaIndex 文档，应补信息检索与经典 RAG 来源。
- M04 应补 OWASP LLM Top 10、NIST AI RMF/GenAI Profile、prompt injection、工具越权和数据外传。
- M04 高阶 async runtime 内容应放在队列、持久化与并发基础之后。

### 数据库、缓存与异步任务

- `collection_not_found` 等确定性错误不应重试。
- cache key 缺 tenant、文档/索引版本、embedding 模型版本、检索器和过滤条件。
- E06 目前主要是设计表格与伪流程，缺 SQLite/RQ/Redis 可运行实现、建库命令和自动测试。

### Docker、可观测性与 Kubernetes

- 教材中的数据库密码必须明确标注为本地教学值，不能用于部署。
- 镜像和 Python 依赖未锁版本或 digest。
- M08 使用 `request.url.path` 作为 Prometheus label 会制造高基数，应使用 route template。
- 百分位定义在 M08 与 P01 中不一致；需统一 nearest-rank、插值或工具口径。
- M09 只讲 Ingress，未覆盖 Gateway API、controller、Metrics Server/KEDA、resource requests、探针和 kind 镜像加载。
- kind 当前未安装；vLLM 不原生支持 Windows，必须写明 Linux/WSL、GPU、驱动、模型与显存前提。
- M10 必须区分 NVIDIA Triton Inference Server 与 OpenAI Triton kernel language。

### 科研方法

- M11 对单随机种子的依赖过强，且暂缓统计检验，不足以支持科研路线。
- RQ01 prior-art 页面仍为空，需要先界定业务队列、LLM serving scheduler 和 Kubernetes scheduler 的研究层次。
- 应补 Orca、FastServe、PagedAttention/vLLM、Sarathi-Serve、DistServe、Llumnix 等相关工作。

### 金融工程

- F01 缺置信区间、假设检验、回归、矩阵正定性/特征值、约束优化和时间依赖。
- F03 缺 Expected Shortfall/CVaR 和尾部风险出口。
- F04 使用风险中性漂移但没有先建立无套利、复制组合和风险中性概率。
- Monte Carlo 只返回 price/runtime，没有标准误、置信区间、多 seed 或相对 Black-Scholes 误差；全局 random seed 不适合并发 worker。
- F05 需要区分宏观利率序列、par yield、spot/zero curve 和折现因子。
- F06/F07 的数据泄漏检查主要靠人工勾选，缺自动 temporal assertions、walk-forward、pipeline fit 范围和最终 holdout。
- Q 项目的激活门槛低于项目承诺范围，不能只完成部分实验就宣称项目启动条件满足。

## 五、内容状态与结构问题

M05 章节教材约 206 KB、262 个标题和大量细粒度代码块，远高于其他模块。内容没有明显标题重复，但粒度过细，核心路径和扩展材料混在一起。

```text
planned -> draft -> code-reviewed -> executable -> verified
```
