## 一、范围与方法

本轮检查覆盖报告生成前的 287 个 Markdown，其中包含 23 份 `适配教材/章节教材`。
检查分为五层：

1. Markdown 结构、标题、列表和围栏。
2. 中文标点、中英文混排和空格。
3. 技术术语、过期运行时和科研强断言。
4. 外部 URL 可访问性与重定向。
5. 教材目标、前置、概念、示例、练习、验收、来源、边界、重复度和篇幅。

开源工具选择：

| 工具 | 固定版本 | 用途 | 采用判断 |
|---|---:|---|---|
| [markdownlint-cli2](https://github.com/DavidAnson/markdownlint-cli2) | 0.23.0 | Markdown AST 结构规则 | 采用；关闭不适合中文的 80 字符和表格对齐规则 |
| [zhlint](https://github.com/zhlint-project/zhlint) | 0.8.2 | 中文文案排版 | 采用为 advisory；先去除代码、URL 和 WikiLink 目标 |
| [Vale](https://github.com/vale-cli/vale) | 3.15.1 | 自定义术语、版本和强断言规则 | 采用；禁用会误报技术词的英文拼写规则 |
| [Lychee](https://github.com/lycheeverse/lychee) | 0.24.2 | Markdown 外链检查 | 采用；403、协议错误与 404 分开解释 |
| [textlint](https://github.com/textlint/textlint) | 15.7.1 | 可扩展自然语言 lint | 主项目活跃，但本轮不重复引入 |
| `textlint-rule-preset-zh-technical-writing` | 0.0.3 | 中文技术写作 preset | 不采用；仓库已归档且最后发布较久 |

Vale 和 Lychee Windows 二进制均在安装前验证官方 SHA-256；Markdownlint 与 Zhlint
通过固定 npm 版本运行。所有检查均为只读，没有使用 `--fix`。

## 四、主要内容质量问题

### P1-1 M05 已超过适合单文件教材的可维护边界

`M05_任务队列与调度_章节教材.md` 有 7826 行、约 12.2 万字符、515 组代码围栏和
15 个一级标题。Markdownlint 的 14 项 `single-h1` 全来自这个文件。

这不是简单的标题格式问题。它会影响：

- 章节导航、局部复习和移动端阅读。
- WikiLink 精确定位和章节级状态管理。
- diff 审查、重复校订和实验映射。
- 后续把某一章独立降级、重写或复现实验。

建议按现有 15 个一级章节拆成独立文件，保留一个总索引；每章单独记录前置、目标、
示例、练习、验收和来源。拆分前不要只把 H1 机械改成 H2。

### P1-2 教材结构具备练习和边界，但章节标签不统一

23 份教材均能匹配“练习/实验”和“边界/局限”内容，20 份能匹配验收或自测，18 份
能匹配示例。说明教材不是空壳。

但按统一写作规范扫描时：

| 标准环节 | 未匹配文件数 | 解释 |
|---|---:|---|
| 学习目标 | 17 | 多数使用“本模块解决什么”等非统一标题 |
| 前置条件 | 16 | 前置内容常散在导言或学习地图 |
| 核心概念 | 17 | 概念存在，但未形成稳定章节入口 |
| 最小示例 | 5 | 需要人工确认是否只有大段代码、缺少最小例子 |
| 验收/自测 | 3 | 需要优先补齐 |
| 来源/参考 | 4 | 需要确认是否只依赖外部资料索引 |

这些是“结构复核候选”，不是 17 份教材都没有目标或概念。下一轮应统一标题词汇，
使自动检查、学习导航和章节验收能使用同一套契约。

### P1-3 10 份教材正文没有直接外部 URL

零直接外链的文件包括 F00-F08 九份金融教材和 M11 科研方法教材。它们中的部分通过
WikiLink 指向独立资料索引，因此不能直接判定“没有来源”。问题在于概念、公式、版本
边界与具体来源之间缺少正文级映射。

建议每章至少增加一个“本章依据”小节，链接到具体官方文档、教材章节、标准或论文，
而不是只在模块末尾链接整个资料索引。优先处理金融公式、时间序列切分、回测偏差、
债券久期凸性和科研统计方法。

### P2-1 Markdown 结构债务为 130 项

| 规则 | 数量 | 处理建议 |
|---|---:|---|
| trailing spaces | 52 | 可机械清理，但先确认硬换行 |
| lists surrounded by blank lines | 50 | 分批修订模板和高频教材 |
| single H1 | 14 | 通过拆分 M05 解决 |
| fences surrounded by blank lines | 12 | 低风险小修 |
| trailing newline | 1 | 低风险小修 |
| emphasis used as heading | 1 | 人工判断该行是否真是章节标题 |

本轮没有剩余反向链接、无语言围栏或不一致标题风格错误。

### P2-2 中文排版建议需要校准后分批处理

Zhlint 给出 6100 项原始建议，最高频为引号外空格、全半角标点、全角内容间空格和
标点前后空格。其中 M01、M00、M05 章节教材和 M05 交接文档合计 3061 项。

这些结果混合了三类内容：

1. 可修的中英文空格和标点问题。
2. Markdown 表格、引用、路径和技术表达造成的误报。
3. 中文排版流派差异，例如中文引号外是否保留空格。

因此 Zhlint 不应进入阻断式 CI，也不能直接运行 `--fix`。建议先选 M00/M01 各 50 条
人工标注真阳性，再据此增加 case-ignore，只有误报率可接受后才批量处理。

### P2-3 外链存在 11 个“网络不确定”URL

最终没有 404/410。剩余 35 次错误记录由 11 个唯一 URL 产生：

- 28 次为 SEC、DOI/ACM、Columbia、Citadel 等站点的 403 防爬或访问策略。
- 7 次为 FRED HTTP/2 协议错误或缓存错误。
- 19 次重定向均能到达有效页面。

这些 URL 不能删除。建议把 Lychee 结果分成 `confirmed broken`、`access restricted`、
`protocol/network uncertain` 三类，并定期用浏览器或站点 API 人工复核。

## 五、通过项与正面结论

- Python 3.8/3.9 的 20 个 Vale 命中均出现在历史审计、升级说明、兼容性陷阱或禁止
  使用旧解释器的语境，没有发现当前教材重新把它们设为验收基线。
- 两个科研强断言命中分别是“完全正确”的对话表达，以及 RQ01 明确禁止使用的
  “显著优于 FIFO”示例；没有发现缺少证据却直接宣称普遍优越的正文结论。
- `FastAPI`、`GitHub`、`PyTorch`、`PostgreSQL`、`Kubernetes`、`LangChain` 等
  术语大小写规则没有命中错误。
- 未发现正文 TODO/TBD/“待补充/待完善”标记。
- 未发现跨文件长段落机械复制。
- 资料域名以 Python、Kubernetes、MIT OCW、SEC、pytest、FastAPI、Prometheus、
  pandas、Docker 和 vLLM 等官方或高质量来源为主。

## 六、下一轮整改顺序

1. **拆分 M05**：先建立章节索引和跨章依赖，再逐章迁移，不做整文件机械降级标题。
2. **补教材结构契约**：优先补 3 份缺验收、5 份缺最小示例的候选文件，再统一目标、
   前置和核心概念标题。
3. **补正文级来源映射**：优先 F00-F08 和 M11，不重复堆链接，直接映射到公式/方法。
4. **清理 Markdown 低风险项**：围栏空行、尾部空格、列表空行和最后换行。
5. **校准 Zhlint**：人工标注样本后建立 ignore，不把 6100 项原始建议当成 6100 个错误。
6. **持续外链检查**：确认 404/410 阻断；403/协议错误只告警并保留人工复核记录。
