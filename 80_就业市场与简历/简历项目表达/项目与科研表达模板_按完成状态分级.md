# 项目与科研表达模板：按完成状态分级

## 定位

这份文档用于连接 `50_项目产出`、`60_科研训练` 和 `80_就业市场与简历`。

它不是最终简历，也不是包装成果，而是一个表达边界模板：每次写简历 bullet、面试讲法、导师沟通材料前，先判断内容属于哪种状态，再选择对应口径。

核心原则：

- 亲手实现、运行、测试、记录过的内容，才能写成已完成成果。
- AI 辅助生成、参考答案、案例参照，只能写成学习材料、分析材料或复现参照。
- 计划复现、正在学习、正在搭建的内容，只能写成目标、设计、计划和当前进展。
- 没有真实实验记录，就不能写 P95/P99 优化幅度、吞吐、错误率、结论性对比。
- 没有亲手解释过的代码和数据，不能进入个人项目成果表达。

## 表达状态分级

| 状态 | 可以怎么说 | 不能怎么说 | 证据要求 |
|---|---|---|---|
| 已亲手完成 | 实现、复现、运行、测试、记录、分析 | 不能夸大到生产级或企业级 | 代码、运行命令、测试结果、实验记录、问题记录 |
| AI 辅助生成 / 参考答案 | 参考、分析、对照、学习样板、复现目标 | 不能说“我独立完成”“我实现了完整项目” | 来源说明、学习笔记、复现计划 |
| 计划复现 / 正在学习 | 正在搭建、已完成设计、计划验证、准备复现 | 不能说已经优化、证明、落地 | 路线、任务拆解、实验设计、待办清单 |

## P01 Mini Scheduler 表达边界

### 当前状态

P01 当前在学习库中的正确定位是：

```text
M05 教材配套参考答案 / 实验样板 / 后续亲手复现的对照材料
```

它已经有较完整的代码、实验记录和展示素材，但在没有逐章亲手复现、运行测试、解释实验之前，不应直接写成个人已完成项目成果。

### 当前可用简历口径

适合放在学习经历、项目准备或技术积累中：

```text
正在围绕 AI workload 调度复现 Mini Scheduler 学习项目，使用 Python 建模 Task、Worker、Queue 和 FIFO/Priority/SJF 等调度策略，重点训练任务建模、指标统计、实验记录和调度策略对比能力。
```

可用前提：

- 已阅读 P01 项目主页和 M05 教材。
- 已明确 P01 是参考答案或样板，不是当前已完成成果。
- 不写具体优化结论和指标数值为个人成果。

### 复现后可用简历口径

完成亲手复现后才可使用：

```text
复现并扩展 Mini Scheduler 调度模拟器，使用 Python 实现 Task/Worker 建模、FIFO/Priority/SJF/Cost-aware 调度策略、单 worker 与多 worker 执行模拟，并记录 average wait、P95/P99、queue length 和 worker utilization 等指标，用于分析不同策略在尾延迟、资源利用率和公平性之间的取舍。
```

可用前提：

- 亲手复现核心代码，而不是只阅读 AI 生成代码。
- 能本地运行 demo、smoke check 或 pytest。
- 至少完成 E05 调度实验记录。
- 能解释 Task、Worker、Queue、策略函数和指标计算方式。
- 如果使用数值，必须来自自己的实验记录。

### 面试解释口径

当前阶段：

```text
P01 目前是我学习 M05 调度模块时使用的参考项目。我现在重点不是把它包装成完成项目，而是通过逐步复现理解 AI workload 调度中的任务建模、策略比较和指标统计。等我亲手跑完实验并记录结果后，才会把它整理成正式项目经历。
```

复现后：

```text
我先用纯 Python 做调度器，是为了把复杂 AI Infra 问题拆小：不先碰 Kubernetes 或真实 GPU，而是先弄清任务如何进入队列、worker 如何执行、调度策略如何影响等待时间和尾延迟。这个项目的价值在于用实验数据解释策略取舍，而不是简单实现一个排序函数。
```

可用前提：

- 能现场讲清 FIFO、Priority、SJF、Cost-aware 的区别。
- 能解释平均等待和 P95/P99 为什么都需要看。
- 能说明模拟实验不能直接等同真实服务结论。

### 导师沟通口径

```text
我目前在用 Mini Scheduler 作为调度问题的最小实验平台，先复现 FIFO、Priority、SJF 和 Cost-aware 等 baseline，观察它们对 average wait、P95/P99 和 worker utilization 的影响。这个阶段仍是模拟实验，我不会把它当成真实 RAG/Agent 平台结论，后续计划接入 P03 的真实 API/worker 压测来验证。
```

可用前提：

- 已把 P01 与 RQ01 的关系说成“模拟实验层”。
- 不声称已经完成论文级结论。

## P03 AI Workload Platform 表达边界

### 当前状态

P03 是当前学习库的主项目工作台，目前更准确的状态是：

```text
主项目规划 / 工作台已建立 / v0.1 待实现
```

当前不能写成完整平台成果。

### 当前可用简历口径

```text
正在设计面向 RAG/Agent 请求的 AI workload 平台，将 RAG 查询、Agent 报告生成等请求抽象为可排队、可持久化、可监控的 Task，已完成项目范围、API 与数据契约、任务拆解、实验连接和阶段执行说明。
```

可用前提：

- P03 工作台文件已建立。
- 能说明 v0.1 只做 FastAPI、tasks 表或内存替代、mock worker、基础 metrics。
- 不声称系统已经完整实现。

### RAG 任务链路由学习者复现后可用简历口径

```text
设计并实现一个面向 RAG retrieval 与模拟 workload 的任务平台，基于 FastAPI 提供 owner-scoped 任务创建和状态查询接口，将请求抽象为通用 TaskRecord，并通过 PostgreSQL/outbox、Redis Streams 和独立 worker 完成检索或 mock 执行、状态回写与基础 metrics 记录；支持 Docker Compose 本地启动和 queue_wait、runtime、error_type 等指标采集。真实 Agent 工作流尚未接入。
```

可用前提：

- 已实现 `POST /tasks`（`task_type=rag_retrieval`）或等价任务创建接口。
- 已实现 `GET /tasks/{task_id}` 状态查询。
- 已实现 tasks 存储、状态流转和 worker 回写。
- 已完成 E06/E07 至少一轮项目化验证。
- 有 README、运行命令、测试或截图证据。
- 上述实现和验证由学习者本人复现；不能把仓库 reference 直接写成个人成果。

### 压测后可用简历口径

```text
基于 Locust/k6 对 RAG/API/worker 链路进行压测，记录 API latency、queue_wait、task_runtime、P95/P99、throughput、error_rate 和 worker utilization，并分析不同并发量、worker 数量或调度策略下的性能变化。
```

可用前提：

- 已完成 E08 压测实验。
- 指标来自真实压测记录。
- 能区分 API 延迟、排队等待、任务执行时间和总延迟。
- 不虚构优化幅度。

### 面试解释口径

```text
P03 的核心不是做一个泛泛的 RAG demo，而是把 RAG/Agent 请求作为系统 workload 来处理。普通同步 API 只关心一次请求是否返回，而 P03 更关心请求进入队列后如何保存、调度、执行、失败重试、查询状态和监控延迟。第一版我会先用 mock worker 跑通任务生命周期，再逐步替换真实 RAG 和 Agent。
```

可用前提：

- 能说明为什么第一版先 mock。
- 能讲清 M03/M04/M05/M06/M08 分别负责什么。

### 导师沟通口径

```text
我计划把 P03 作为 RQ01 的工程验证平台。第一阶段先实现可观测的 RAG/Agent 任务链路，包括任务入库、队列、worker、状态回写和基础指标；第二阶段再用压测数据比较不同任务类型和调度策略对 P95/P99、queue wait 和 worker utilization 的影响。
```

可用前提：

- 已有 RQ01 实验设计。
- 已明确当前是待验证计划，不是已有研究结论。

## RQ01 科研表达边界

### 当前状态

RQ01 仓库当前可以表达为：

```text
研究问题 + 实验设计 + 30-seed E2 synthetic pilot + 待场景验证假设
```

该 pilot 含 paired bootstrap 和分组伤害分析，但仍是 P01 模拟器上的合成证据，不是 P03
真实服务、RAG/Agent 场景或生产 workload 结论；学习者未独立复现时也不能把 reference 结果写成
个人研究成果。

### 当前可用简历 / 科研兴趣口径

```text
围绕 RAG/Agent 混合请求调度问题，设计对比 FIFO、Priority、SJF、Cost-aware 等策略的实验方案，计划记录 average wait、P95/P99、queue length、throughput 和 worker utilization，分析不同策略在平均效率、尾部延迟和资源利用率之间的取舍。
```

可用前提：

- 已写清研究问题、baseline、自变量、因变量和控制变量。
- 只说“设计”“计划”“待验证”，不说“证明”“发现最终结论”。
- 若提及 E2 pilot，必须明确 `synthetic/reference`，并能展示原始 workload、per-seed 结果和分析脚本。

### 后续有实验记录后可用口径

```text
基于模拟调度实验和 P03 工程压测数据，比较不同调度策略在 RAG/Agent 混合 workload 下对 P95/P99、queue wait 和 worker utilization 的影响，并整理实验记录、失败分析和局限性说明。
```

可用前提：

- `03_实验记录.md` 已有真实记录。
- `04_实验报告.md` 已区分实验观察、结论和局限。
- 能说明模拟实验和真实服务压测之间的差异。

### 导师沟通口径

```text
老师您好，我目前在围绕 RAG/Agent 请求调度做一个入门研究训练。当前问题是：当短问答、长文档总结、报告生成等请求混合到来时，不同调度策略如何影响 P95/P99 尾延迟和 worker 利用率。现在我已经整理了研究问题、baseline、变量和指标，下一步会通过 P01 模拟实验和 P03 工程压测逐步验证。我目前不把初步模拟结果当作最终结论，想请教这个问题是否适合作为系统方向的入门训练，以及实验设计还需要补哪些 baseline 或控制变量。
```

可用前提：

- 能展示 RQ01 的研究问题和实验设计。
- 不虚构论文贡献和实验结论。

## M12 金融/合规场景表达边界

### 当前状态

M12 是 P03 的应用场景，不是金融学教材，也不是投资建议系统。

当前可以说：

```text
将公开金融/合规文档作为 RAG/Agent workload 场景，训练文档解析、证据引用、风险提示、人工确认和任务监控。
```

不能说：

```text
自动荐股、预测股价、生成投资建议、完成金融终端、给出真实投研结论。
```

### 当前可用简历口径

```text
设计金融/合规文档 RAG 场景，将公告、财报、研报或合规材料中的问答、摘要、证据引用和报告草稿生成建模为 AI workload，为 P03 的任务队列、状态流转、延迟监控和人工复核提供场景输入。
```

可用前提：

- 明确这是场景设计或学习场景。
- 不输出投资建议。
- 不虚构公司、数据、财报结论或风险判断。

### P03 场景实现后可用口径

```text
在 P03 中接入金融/合规文档场景，实现文档 metadata、chunk、retrieved_sources、has_citation、risk_note 和 task metrics 记录，使金融文档问答和报告草稿生成能够作为可调度、可追踪、可压测的 workload。
```

可用前提：

- 已完成 E03 或 P03 中的 RAG 字段验证。
- 已有自造测试数据或公开来源数据说明。
- 输出带引用和风险提示。

### 面试解释口径

```text
我使用金融/合规文档不是为了做投资建议，而是因为这类文档对引用、证据、权限、风险提示和人工复核要求高，很适合检验 RAG/Agent 系统是否能做到可追踪和可解释。它在我的路线里是 P03 的场景化 workload。
```

可用前提：

- 能说明 M12 与 M03/M04/P03 的关系。
- 不把场景材料说成真实投研成果。

可用前提：

## 三类口径总模板

### 简历表达模板

已亲手完成：

```text
实现/复现 [项目或模块]，完成 [核心功能]，通过 [测试/实验/压测] 记录 [指标]，用于分析/支撑 [工程目标或研究问题]。
```

AI 辅助生成 / 参考答案：

```text
参考 [项目/案例/样板]，分析其 [结构/字段/实验/测试]，将可学习模式整理为 [个人项目/学习模块] 的复现计划和设计约束。
```

计划复现 / 正在学习：

```text
正在搭建 [项目/实验]，已完成 [范围定义/API契约/实验设计/任务拆解]，计划通过 [具体实验] 验证 [待验证问题]。
```

### 面试解释模板

已亲手完成：

```text
这个项目我亲手做过的部分是 [范围]。我能运行的入口是 [命令/API]，主要指标是 [指标]，遇到的问题是 [失败/限制]，目前不能夸大的地方是 [边界]。
```

AI 辅助生成 / 参考答案：

```text
这部分目前是我的学习参照，不是我独立完成的项目。我使用它的方式是先分析结构，再把核心模式拆成可复现任务，后续在自己的 P03 中亲手实现和验证。
```

计划复现 / 正在学习：

```text
这部分我还没有写成完成成果，目前是设计和计划阶段。我已经明确要做的最小闭环是 [闭环]，验收标准是 [标准]，完成后才会把它写进正式项目经历。
```

### 导师沟通模板

已亲手完成：

```text
我已经完成 [实验/项目] 的第一轮实现和记录，当前观察到 [只说记录事实]。我还没有把它上升为最终结论，因为 [局限]。下一步想请教 [baseline/变量/指标/实验设计问题]。
```

AI 辅助生成 / 参考答案：

```text
我目前参考了 [案例/样板]，主要用于理解工程边界和实验组织方式。它不是我的研究结果，我计划把其中的 [可学习模式] 复现在自己的项目中，再形成可验证数据。
```

计划复现 / 正在学习：

```text
我正在设计 [研究问题/工程项目]，当前已明确问题、baseline、变量和指标，但还没有实验结论。希望请教这个方向是否合理，以及还需要补充哪些控制变量或评价指标。
```

## 禁止表达清单

以下表达在没有真实证据前禁止使用：

- “已证明某调度策略显著优于其他策略。”
- “生产级 AI workload 平台。”
- “企业级 RAG/Agent 平台。”
- “完整云原生推理平台。”
- “真实 DeepSeek/Qdrant/向量数据库主链路已完成。”
- “脱敏合规 AI 案例是我的项目。”
- “自动生成投资建议。”
- “预测股价 / 自动荐股。”
- “优化 P95/P99 达到某个百分比。”
- “支持生产级多租户权限。”
- “论文结果表明……”

## 使用检查表

每次把内容写进简历、面试稿或导师邮件前，逐条检查：

- [ ] 这句话对应的是已亲手完成、AI 辅助参考，还是计划复现？
- [ ] 有没有代码、实验、测试或运行记录支撑？
- [ ] 指标是否来自自己的实验记录？
- [ ] 是否把模拟实验误写成真实服务结论？
- [ ] 是否把案例参照误写成个人成果？
- [ ] 是否把金融/合规场景误写成投资建议或法律意见？
- [ ] 是否明确了可用前提和当前边界？
