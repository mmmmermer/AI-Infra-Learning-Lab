# JD 驱动学习路线系统重构规则

## 定位

这份规则用于把岗位需求转化为学习路线的系统性重构依据。

它不是简历包装工具，也不是看到一个岗位就零散追加几个待办。岗位 JD 的作用是提供外部校准信号：市场反复要求什么能力域，当前学习库是否用合理的课程顺序、教材内容、实验和项目承接了这个能力域。

核心目标：

```text
岗位需求
-> 工程能力拆解
-> 识别背后的能力域
-> 对照权威课程 / 官方文档 / 高质量学习路线
-> 映射 M00-M12
-> 映射 E 实验 / P01 / P03 / RQ01
-> 判断当前路线结构是否合理
-> 系统性重构对应模块、实验、项目和资料索引
```

## 不做什么

- 不因为单个 JD 新增大模块。
- 不把岗位话术直接写成个人能力。
- 不虚构“已掌握”“已完成项目”。
- 不把 Java、Go、K8s、vLLM 等需求直接全部塞进当前阶段。
- 不把金融岗位需求变成投资建议或投研结论。
- 不用零散 checklist 代替教材、实验和项目闭环。
- 不只按关键词匹配学习模块，必须解释能力之间的关系。

## JD 输入优先级

| 来源 | 使用方式 |
|---|---|
| 岗位截图 | 先 OCR 或人工转文字，再拆解 |
| 岗位文本 | 直接拆解 |
| 多个 JD | 汇总共性，优先级高于单个 JD |
| 公司技术博客 / 招聘说明 | 可作为背景，但不能替代 JD |

单个 JD 只生成“候选重构观察”。至少 3 个同类岗位反复出现的要求，或该要求明显影响 P03 主项目能力闭环时，才考虑系统性调整学习路线。

## 能力分类标准

每条岗位要求先归入下面类别：

| 类别 | 对应学习模块 | 典型关键词 |
|---|---|---|
| Python 工程 | M00/M01 | Python、pytest、typing、工程化、脚本、数据处理 |
| 后端 API | M02 | FastAPI、REST、Pydantic、接口、服务化、权限 |
| RAG 工程 | M03 | RAG、embedding、向量数据库、chunk、retrieval、citation |
| Agent 工作流 | M04 | Agent、tool calling、workflow、LangGraph、人工确认 |
| 队列与调度 | M05 | queue、worker、scheduler、priority、latency、resource scheduling |
| 数据库/缓存/异步 | M06 | SQL、PostgreSQL、Redis Streams、异步任务、重试、幂等；Celery/RQ 只作框架对比 |
| Docker/交付 | M07 | Docker、Docker Compose、容器化、部署、环境变量 |
| 监控压测 | M08 | metrics、logs、tracing、Prometheus、Grafana、P95、压测 |
| K8s/云原生 | M09 | Kubernetes、HPA、Service、Ingress、Job、Kueue、Volcano |
| 推理服务 | M10 | vLLM、Triton、Ray Serve、batching、KV cache、TTFT、TPOT |
| 科研实验 | M11 | baseline、实验设计、论文、复现、指标、报告 |
| 金融/场景 | M12 | 财报、公告、研报、合规、证据引用、风险提示 |
| 简历/面试 | 80 就业市场 | 项目表达、岗位匹配、面试题 |

## 覆盖状态

每条能力要求只允许标成四种状态：

| 状态 | 含义 | 处理方式 |
|---|---|---|
| 已覆盖 | 学习模块、实验或项目已明确覆盖 | 不新增内容，只记录岗位证据 |
| 覆盖但薄弱 | 有教材或实验，但缺真实练习记录 | 提高学习/实验优先级 |
| 缺项目证据 | 学过或有教材，但没有项目产出 | 映射到 P01/P03 或实验 |
| 当前不做 | 超出当前阶段或偏离主线 | 放入长期观察，不抢主线 |

## 系统性重构触发条件

只有满足以下条件之一，才进入路线重构：

1. 多个目标岗位反复出现同一能力域。
2. 某个能力是 P03 主项目闭环必须具备的能力。
3. 当前模块虽然有教材，但实验或项目承接明显不足。
4. 当前模块内容顺序与权威课程或工程实践明显不一致。
5. 某能力会影响简历表达真实性，例如没有实验数据却想写性能优化。

不满足触发条件时，只记录为岗位观察，不修改学习路线。

## 系统性重构类型

进入重构后，输出不叫“补丁”，而叫“能力域重构方案”。它可以包含：

1. **模块结构重构**：调整相关 M 模块的小节顺序、学习边界或主线解释。
2. **教材内容重构**：把零散概念改写成可自学讲义，补为什么、工程场景、例子、误区、练习、验收。
3. **实验链路重构**：把岗位能力转成连续实验，而不是单个练习。
4. **P03 项目重构**：调整 P03 的字段、接口、worker、metrics、README 或阶段执行计划。
5. **资料索引重构**：补充或替换为官方文档、顶级高校课程、高质量开源项目资料，并说明每条资料转化到哪里。
6. **学习顺序重构**：调整某能力域在 12 周路线、两个月路线或 P03 阶段计划中的位置。
7. **表达边界重构**：更新简历/面试可用前提，防止路线变化后产生虚假表达。

重构必须保持现有主线：

```text
M00/M01 打基础
-> M05/P01 建调度内核
-> M02/M03/M04/M06/M07/M08 支撑 P03
-> M09/M10/M11/M12 进阶和差异化
```

## 和当前项目的强连接

所有岗位需求都优先尝试映射到 P03，而不是新开项目：

```text
JD 要求 RAG
-> M03 / E03
-> P03 current Task(task_type=rag_retrieval) / retrieved_sources / authorization metadata

JD 要求 Agent
-> M04 / E04
-> P03 planned AgentTask / step_logs / approval_status（v0.3.1 未实现）

JD 要求异步任务
-> M06 / E06
-> P03 tasks 表 / worker / retry / idempotency

JD 要求 Docker
-> M07 / E07
-> P03 api + db + redis + worker compose

JD 要求监控压测
-> M08 / E08
-> P03 latency / queue_wait / P95/P99 / error_rate

JD 要求 K8s/vLLM
-> M09/M10 长期进阶
-> P03 未来迁移路线，不抢第一轮
```

## 权威资料审计要求

任何系统性重构都必须先审计资料来源，优先级如下：

1. 官方文档：Python、FastAPI、PostgreSQL、Redis、Docker、Kubernetes、Prometheus、OpenTelemetry、vLLM、Triton 等。
2. 顶级高校课程：MIT、CMU、Stanford、Berkeley 等公开课程，尤其是系统、数据库、分布式、云计算、机器学习系统相关课程。
3. 高质量工程文档：Google Cloud、AWS、Microsoft、NVIDIA、Ray、LangChain/LangGraph、LlamaIndex 等官方工程文档。
4. 高质量开源项目：能作为结构参照，但不能直接包装成个人项目成果。
5. 技术博客：只做辅助，不作为核心依据。

资料进入路线前必须回答：

- 它支撑哪个能力域？
- 它对应哪个 M 模块？
- 它转化成教材、实验、P03 字段，还是长期观察？
- 是否超出当前阶段？
- 是否和现有路线重复或冲突？

## 输出要求

每次处理一个 JD，输出必须包含：

- 岗位原文或 OCR 文本。
- 岗位族判断。
- 能力需求拆解。
- 与 M00-M12 的映射表。
- 与 E 实验/P01/P03/RQ01 的映射表。
- 当前覆盖状态。
- 权威资料 / 课程 / 文档审计。
- 是否触发系统性重构。
- 能力域重构方案。
- 不进入当前路线的要求。
- 下一步可执行学习动作。

## 质量检查

JD 重构路线前必须检查：

- [ ] 是否只是单个 JD 的个性要求？
- [ ] 是否能映射到现有 M00-M12？
- [ ] 是否能落到 E 实验或 P03 项目？
- [ ] 是否会打乱当前第一阶段学习主线？
- [ ] 是否需要真实项目证据，而不是继续补教材？
- [ ] 是否涉及不能虚构的成果或指标？
- [ ] 是否已经查过权威资料或课程？
- [ ] 是否是系统性重构，而不是零散加 checklist？
