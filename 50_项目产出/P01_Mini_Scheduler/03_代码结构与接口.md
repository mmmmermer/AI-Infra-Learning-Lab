# P01 代码结构与接口

## 设计原则

第一阶段的代码结构要足够清楚，但不要过早复杂化。你现在需要的是一个能支撑学习、测试和实验的小工程，而不是一上来就做企业级框架。

核心原则：

- 数据模型和调度策略分开
- 业务逻辑和命令行 demo 分开
- 指标计算和调度执行分开
- 测试文件只测试规则，不依赖打印输出
- 每个函数先小而明确，后面再组合


补充约束：

- 调度策略必须可切换，FIFO / Priority / SJF / Cost-aware 不能写死。
- 指标必须单独函数化，至少支持平均值、P95、P99 和 worker 利用率。
- 实验口径必须固定优先级方向、时间字段和成本公式。

## 每个文件负责什么

> 环境提示：本项目使用 Python 3.13。先用 `py -3.13 -m venv .venv` 创建环境并激活，再运行代码。旧文件中的 `from __future__ import annotations` 可以保留，但不代表继续支持 Python 3.8/3.9。

### `scheduler/models.py`
负责定义核心数据结构。

```python
from dataclasses import dataclass


@dataclass
class Task:
    id: str
    task_type: str
    priority: int
    estimated_duration: float
    submit_time: float
    token_count: int = 0
    start_time: float | None = None
    finish_time: float | None = None
    status: str = "pending"


@dataclass
class Worker:
    id: str
    available_at: float = 0.0
    current_task_id: str | None = None
    total_busy_time: float = 0.0
```

字段约定：

- `priority=1` 表示最高优先级。
- `token_count` 先用于 Cost-aware 模拟，不要求一开始来自真实 LLM。
- `submit_time`、`start_time`、`finish_time` 是指标计算的基础。

### `scheduler/strategies.py`
负责调度策略排序。这里不负责执行任务，只决定顺序。

```python
from .models import Task


def sort_by_fifo(tasks: list[Task]) -> list[Task]:
    return sorted(tasks, key=lambda task: task.submit_time)


def sort_by_priority(tasks: list[Task]) -> list[Task]:
    return sorted(tasks, key=lambda task: (task.priority, task.submit_time))


def sort_by_sjf(tasks: list[Task]) -> list[Task]:
    return sorted(tasks, key=lambda task: (task.estimated_duration, task.submit_time))


def calculate_cost_score(task: Task) -> float:
    return task.estimated_duration + 0.001 * task.token_count + task.priority * 0.5


def sort_by_cost_aware(tasks: list[Task]) -> list[Task]:
    return sorted(tasks, key=lambda task: (calculate_cost_score(task), task.submit_time))
```

第一版先用 `sorted()`，便于理解和测试；后续如果任务量变大，再把 Priority / SJF / Cost-aware 改成 `heapq` 实现。

### `scheduler/simulator.py`

负责模拟任务执行。第一版可以先做单 worker，再升级多 worker。

```python
from .models import Task, Worker


def run_single_worker(tasks: list[Task], worker: Worker) -> list[Task]:
    completed: list[Task] = []
    current_time = worker.available_at

    for task in tasks:
        start_time = max(current_time, task.submit_time)
        finish_time = start_time + task.estimated_duration

        task.start_time = start_time
        task.finish_time = finish_time
        task.status = TaskStatus.SUCCEEDED

        worker.available_at = finish_time
        worker.total_busy_time += task.estimated_duration
        current_time = finish_time
        completed.append(task)

    return completed
```

### `scheduler/metrics.py`
负责指标计算。不要把指标计算写死在 demo 里，否则后面很难测试。

```python
from .models import Task, Worker


def calculate_wait_time(task: Task) -> float:
    if task.start_time is None:
        raise ValueError(f"task {task.id} has not started")
    return task.start_time - task.submit_time


def calculate_turnaround_time(task: Task) -> float:
    if task.finish_time is None:
        raise ValueError(f"task {task.id} has not finished")
    return task.finish_time - task.submit_time


def average(values: list[float]) -> float:
    if not values:
        return 0.0
    return sum(values) / len(values)


def percentile(values: list[float], percentile_value: float) -> float:
    if not values:
        return 0.0
    ordered = sorted(values)
    index = int((len(ordered) - 1) * percentile_value)
    return ordered[index]


def p95(values: list[float]) -> float:
    return percentile(values, 0.95)


def p99(values: list[float]) -> float:
    return percentile(values, 0.99)


def calculate_worker_utilization(worker: Worker, total_time: float) -> float:
    if total_time <= 0:
        return 0.0
    return worker.total_busy_time / total_time
```

注意：第一版 percentile 用简单实现即可，重点是建立尾延迟口径。后续如果要做更严谨统计，可以再改成 NumPy 或 pandas。

### `examples/run_demo.py`
负责把模型、策略、模拟器和指标串起来。它不是核心逻辑所在地，只负责展示“同一组任务在不同策略下会得到什么结果”。

```python
from scheduler.metrics import (
    average,
    calculate_turnaround_time,
    calculate_wait_time,
    calculate_worker_utilization,
    p95,
    p99,
)
from scheduler.models import Task, Worker
from scheduler.simulator import run_single_worker
from scheduler.strategies import (
    sort_by_cost_aware,
    sort_by_fifo,
    sort_by_priority,
    sort_by_sjf,
)


def build_tasks() -> list[Task]:
    return [
        Task("task-001", "rag_query", 2, 5.0, 0.0, token_count=1200),
        Task("task-002", "agent_tool", 1, 2.0, 1.0, token_count=500),
        Task("task-003", "embedding", 3, 1.0, 2.0, token_count=3000),
        Task("task-004", "long_context", 2, 8.0, 3.0, token_count=8000),
    ]


def run(strategy_name: str) -> None:
    strategies = {
        "fifo": sort_by_fifo,
        "priority": sort_by_priority,
        "sjf": sort_by_sjf,
        "cost_aware": sort_by_cost_aware,
    }

    ordered_tasks = strategies[strategy_name](build_tasks())
    worker = Worker("worker-1")
    completed = run_single_worker(ordered_tasks, worker)

    wait_times = [calculate_wait_time(task) for task in completed]
    turnaround_times = [calculate_turnaround_time(task) for task in completed]
    total_time = max(task.finish_time or 0.0 for task in completed)

    print(f"strategy={strategy_name}")
    print("order=" + ", ".join(task.id for task in completed))
    print(f"average_wait_time={average(wait_times):.2f}")
    print(f"average_turnaround_time={average(turnaround_times):.2f}")
    print(f"p95_wait_time={p95(wait_times):.2f}")
    print(f"p99_wait_time={p99(wait_times):.2f}")
    print(f"worker_utilization={calculate_worker_utilization(worker, total_time):.2f}")
    print()


if __name__ == "__main__":
    for strategy_name in ["fifo", "priority", "sjf", "cost_aware"]:
        run(strategy_name)
```

注意：demo 里不能再出现未定义的 `calculate_completion_time`。本项目统一使用 `turnaround_time = finish_time - submit_time`，中文里可以叫“周转时间”或“完成耗时”，但代码接口统一叫 `calculate_turnaround_time`。

## 第一版接口约定
| 函数 | 输入 | 输出 | 责任 |
|---|---|---|---|
| `sort_by_fifo(tasks)` | `list[Task]` | `list[Task]` | 按提交时间排序 |
| `sort_by_priority(tasks)` | `list[Task]` | `list[Task]` | 按优先级排序 |
| `sort_by_sjf(tasks)` | `list[Task]` | `list[Task]` | 按预计耗时排序 |
| `calculate_cost_score(task)` | `Task` | `float` | 计算成本分数 |
| `sort_by_cost_aware(tasks)` | `list[Task]` | `list[Task]` | 按成本分数排序 |
| `run_single_worker(tasks, worker)` | `list[Task]`, `Worker` | `list[Task]` | 模拟单 worker 执行 |
| `calculate_wait_time(task)` | `Task` | `float` | 计算等待时间 |
| `calculate_turnaround_time(task)` | `Task` | `float` | 计算周转时间 |
| `average(values)` | `list[float]` | `float` | 计算平均值 |
| `p95(values)` | `list[float]` | `float` | 计算 P95 |
| `p99(values)` | `list[float]` | `float` | 计算 P99 |
| `calculate_worker_utilization(worker, total_time)` | `Worker`, `float` | `float` | 计算 worker 利用率 |

## 常见错误

### 错误 1：把排序、执行、打印都写在 `main.py`

这样一开始能跑，但后面没法测试，也不容易扩展。应该把排序放到 `strategies.py`，把执行模拟放到 `simulator.py`，把打印放到 `run_demo.py`。

### 错误 2：函数偷偷修改输入，但自己不知道

如果排序函数直接修改原始列表，测试可能会相互影响。第一阶段建议用 `sorted()` 返回新列表。

### 错误 3：priority 的方向搞反

这里约定数字越小优先级越高，例如 `priority=1` 比 `priority=3` 先执行。一定要在测试里固定这个规则。

### 错误 4：指标写在 print 里

指标必须先是函数，print 只是展示结果。否则后面做实验图和报告时会很痛苦。

## 从 v0.1 到 v0.2 怎么升级

v0.1：纯 Python，单 worker，三种排序策略。

v0.2：多 worker，任务按照 worker 的 `available_at` 分配。

v0.3：加入随机任务分布，生成 FIFO/Priority/SJF 对比实验。

v0.4：接 FastAPI，把任务创建和指标查询服务化。
