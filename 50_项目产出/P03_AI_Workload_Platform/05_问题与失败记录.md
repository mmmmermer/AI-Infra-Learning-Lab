## 2026-07-10 PostgreSQL UUID 与 API 字符串契约不一致

### 现象

第一次真实 Compose 提交返回 HTTP 500；内存测试全部通过。

### 根因

PostgreSQL `UUID` 列经 psycopg 返回 `UUID` 对象，而 `TaskRecord.task_id` 的外部契约是 string。直接把数据库 row 交给 Pydantic 触发类型校验失败。

### 修复

在 repository 的 `_record()` 映射边界统一执行 `str(task_id)`，数据库内部继续使用 UUID，API 继续输出 string。

### 复盘

内存 fake 不能覆盖驱动原生类型差异；真实数据库集成验证不可被 mock 单测替代。

## 2026-07-10 Lease recovery outbox JSON 序列化失败

### 现象

正常任务、失败任务和 dispatcher 积压恢复均成功，但停止 worker 后任务长期停在 `running`，worker 日志反复出现 `UUID is not JSON serializable`。

### 根因

租约恢复从 PostgreSQL 读取 UUID 后，直接放入 `Jsonb({"task_id": row["task_id"]})`。JSON 编码器不接受 UUID，整个恢复事务回滚。

### 修复

outbox JSON payload 只写字符串 task id；数据库外键参数仍使用 UUID。随后从空卷重跑，任务达到 `succeeded`，`retry_count=1`、`delivery_count=2`。

### 复盘

故障恢复分支必须被主动杀进程验证；只测 happy path 无法证明 lease/reconciliation 可用。

## 2026-07-10 压测累计指标被前序故障任务污染

### 现象

Locust 请求统计正常，但 `/metrics.average_runtime_ms` 被前一轮 5 秒 lease 故障任务明显抬高。

### 根因

`/metrics` 是累计聚合，不区分 run id；在同一任务库连续跑不同实验会混合样本。

### 修复

`run_load_smoke.ps1` 默认要求空 task store，否则拒绝运行。reference smoke 从空卷重跑并单独保存 CSV/JSON。

### 复盘

正式 E08/RQ01 还应增加 run id 分区、warm-up 规则和原始任务导出，不能只依赖累计 snapshot。

## 2026-07-11 RAG 权限不能由 input_json 自报

### 现象

若把 `allowed_groups` 直接作为 RAG task 输入，客户端可以把自己声明成
`compliance_private`，异步 worker 又无法区分这是授权结果还是用户伪造字段。

### 根因

业务输入和安全上下文混在同一个未受信任 JSON 中，API、持久化和 worker 之间
没有清晰的 trust boundary。

### 修复

- API bearer token 在服务端映射 principal。
- tasks 表单独保存 tenant、user 和 permission snapshot。
- `input_json` 出现 tenant/user/permission 字段时直接返回 `422`。
- BM25 候选集先按 tenant 和 permission 过滤。
- task 查询按 tenant/user 过滤，非 owner 统一返回 `404`。

### 验证

15 个内存测试覆盖固定 golden queries、公开拒绝、私密授权、跨租户拒绝、空 corpus、owner-scoped
幂等和结果来源持久化；Compose 脚本再次覆盖 PostgreSQL/Redis/outbox/worker
完整链路。

### 复盘

当前 reference token registry 只证明边界放置正确，不是生产鉴权。正式部署仍需
身份提供方、服务端权限查询、token rotation、撤权策略和安全审计。

## 2026-07-11 Windows PowerShell 后台 Locust 进程属性不稳定

### 现象

首次加入队列/资源时序采集后，Locust 实际正常结束，但脚本先后出现
`StartTime` 为空和 `ExitCode` 为空，导致 reference run 被判失败。

### 根因

Windows PowerShell 通过 `Start-Process -PassThru` 配合 stdout/stderr 重定向时，
返回的 process wrapper 没有稳定提供这些属性；同时在主采样循环里同步调用
`docker stats --no-stream`，又会把 500ms 队列采样拖慢到数秒。

### 修复

- 运行起点由脚本显式记录，不依赖 process `StartTime`。
- 改用 `System.Diagnostics.Process` 获取可靠 exit code 和异步 stdout/stderr。
- worker 资源采样放入独立 PowerShell job；主循环只采 API 队列指标。
- 保存 `timeseries.csv` 和 `worker_resources.csv` 两类原始时序。

### 验证

4 秒 instrumentation smoke 得到 12 个队列样本和 3 个资源样本；随后 9 组
随机化重复实验全部完成，且 runner 最终清理了 Compose 容器和 named volumes。

## 2026-07-11 E09 PowerShell 中文路径发生 mojibake

### 现象

kind 验证脚本能够启动，但按硬编码中文路径定位 P03 源码时找不到目录。

### 根因

Windows PowerShell 的脚本编码与当前终端解码组合使中文路径字面量发生 mojibake。

### 修复

脚本从仓库根目录递归查找 ASCII 目录名 `P03_AI_Workload_Platform`，再拼接
`p03_service`，不再依赖中文父目录字面量。

### 复盘

Windows 自动化脚本应尽量用稳定的 ASCII 锚点发现路径，并在执行前用
`Resolve-Path` 验证目标。路径发现成功只证明脚本可移植性提高，不替代其他主机实测。

## 2026-07-11 E09 runAsNonRoot 无法验证镜像用户

### 现象

API Pod 创建后停在 kubelet 校验阶段，提示镜像用户 `appuser` 不是数字，无法确认其
是否为非 root 用户。

### 根因

镜像使用命名用户；Pod 开启 `runAsNonRoot` 后，kubelet 不能仅凭用户名证明 UID 非 0。

### 修复

Pod security context 显式设置 `runAsUser: 10001`、`runAsGroup: 10001` 和
`runAsNonRoot: true`。最终验证 artifact 记录 API 实际 UID 为 `10001`。

### 复盘

容器内存在非 root 用户不等于 Kubernetes 能验证安全上下文。生产环境还需补只读
根文件系统、capability、seccomp、NetworkPolicy 和镜像供应链控制。

## 2026-07-11 E09 readiness probe 默认超时过短

### 现象

API 进程能够启动，但 readiness probe 间歇性超时，Deployment rollout 无法稳定完成。

### 根因

exec probe 默认 `timeoutSeconds: 1`；Python 启动依赖检查在本机 kind 节点上偶尔超过
一秒。

### 修复

为探针增加初始等待，并将 `timeoutSeconds` 调整为 3 秒；随后五个 Deployment rollout
全部通过。

### 复盘

探针参数必须根据启动路径和环境抖动实测，不能把放宽超时当作掩盖依赖失败的手段。

## 2026-07-11 E09 API rollout 后原 port-forward 失效

### 现象

API Deployment 滚动重启后数据库中的任务仍存在，但后续 HTTP 校验因连接拒绝失败。

### 根因

原 `kubectl port-forward` 绑定已删除的旧 API Pod；Pod 替换后该进程正常退出，不会自动
切换到新 Pod。

### 修复

验证脚本加入可重启的 port-forward helper，在 rollout 后等待新 Pod Ready，再重新建立
转发并继续查询原 task。

### 复盘

port-forward 只适合本地诊断，不是稳定入口。此次验证证明 PostgreSQL-backed task 在
API Pod 替换后可查询，不证明 Ingress、负载均衡或高可用入口已经实现。
