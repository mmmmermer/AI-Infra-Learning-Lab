## v0.1 Memory 架构

```text
FastAPI
-> InMemoryTaskStore
-> FIFO deque
-> POST /workers/run-next 手工触发
-> deterministic mock workload
```

此模式默认启用，适合单进程理解状态机。当前完整测试套件为 27 项，跨 memory、PostgreSQL/Redis 语义、RAG 与 fixed sender；进程重启丢数据是 memory 模式有意保留的教学边界。

## v0.2 多服务架构

```text
Client
-> Bearer token -> server-side reference principal
-> FastAPI API
-> PostgreSQL transaction: owner/security snapshot + task(status=pending) + outbox
-> outbox dispatcher lease
-> task(status=queued) + Redis Streams XADD(task_id)
-> independent worker XREADGROUP reserve(task_id)
-> PostgreSQL CAS: queued -> running + claim version
-> deterministic mock or BM25 retrieval workload
-> owner/version/lease-checked CAS: running -> succeeded/failed
-> XACK + XDEL after database finalization
-> GET /tasks/{task_id} / GET /metrics
```

服务：

| 服务 | 职责 | 状态所有权 |
|---|---|---|
| `api` | 输入校验、task/outbox 事务提交、查询 | 无进程内事实状态 |
| `db` | task、outbox、幂等键、租约、结果 | 唯一事实源 |
| `redis` | Streams consumer group 至少一次传递 `task_id`，保留 pending 直到 ACK | 不保存任务真相 |
| `dispatcher` | 租约认领 outbox 并发布消息 | 无 |
| `worker` | CAS 认领、执行、完成/失败、租约恢复 | 无 |

## 一致性设计

### 为什么 API 不双写数据库和 Redis

错误流程：

```text
INSERT task
-> XADD Redis Stream
```

若任一步成功、另一步失败，就会出现数据库有任务但无消息，或消息存在但数据库无任务。

当前流程在一个 PostgreSQL 事务中写 task 和 outbox。dispatcher 在事务外发布 Redis；发布后确认失败可能产生重复消息，因此系统承诺“至少一次投递 + 幂等认领”，不声称恰好一次。worker 用 consumer group reserve 消息；数据库终态写入成功后才 `XACK`。worker 在 reserve 后崩溃时，消息留在 pending entries list，超过 idle 阈值后由其他 consumer `XAUTOCLAIM`。

### Outbox claim

- dispatcher 用 `FOR UPDATE SKIP LOCKED` 选取未发布事件。
- `claimed_at/claimed_by` 构成短租约，避免多个 dispatcher 同时发布。
- Redis 写失败时释放 claim；进程崩溃时等待 claim 过期。
- task 在发布前进入 `queued`，避免 worker 看见消息时 task 仍是 `pending`。

### Worker claim

worker 只有在下面更新成功时才获得执行权：

```sql
UPDATE tasks
SET status = 'running', worker_id = ..., lease_until = ..., version = version + 1
WHERE task_id = ... AND status = 'queued'
RETURNING ...;
```

返回的 `version` 是本次 claim 的 fencing token。heartbeat、成功和失败回写必须同时匹配
`worker_id + version + 未过期 lease`。重复消息、终态任务、旧 claim 和已被其他 worker
认领的任务都无法通过该 CAS。

### Lease recovery

worker 中断后，过期 `running` 任务会在事务内进入 `retrying` 并生成新 outbox。重试耗尽后进入 `failed`。当前 mock workload 最长 5 秒，Compose lease 为 8 秒；长任务 heartbeat 方法已存在，但 worker service 尚未启动周期 heartbeat。

## 当前 workload

允许的 task type：

- `mock_rag`
- `mock_agent`
- `simulated_inference`
- `rag_retrieval`

前三者执行相同的确定性 `mock_reference` 边界，只用于验证任务平台。
`rag_retrieval` 复用 E03 的固定语料和 BM25 方法，先按 `tenant_id` 和
`allowed_permission_groups` 缩小候选集，再建立 BM25 排序。结果保存 extractive
answer、`retrieval_ms`、授权搜索空间大小和完整 source metadata。

身份和权限不从 `input_json` 读取。API 由 bearer token 映射 server-owned
principal，tasks 表持久化 tenant/user/permission snapshot；查询按 tenant/user
过滤。当前 token registry 仅用于本地 reference，不是生产鉴权。

`input_json.force_error=true` 产生不可重试的 `forced_failure`；`sleep_ms` 限制为 `0..5000`。

## Metrics 口径

任务响应计算：

```text
queue_wait_ms = started_at - queued_at
total_latency_ms = finished_at - created_at
```

`GET /metrics` 提供：

- task/status counts。
- Redis `broker_queue_length`。
- `pending_outbox_count`。
- 瞬时 `active_workers`，实际是 running task 数。
- 最近一分钟完成量。
- queue wait 和 runtime 的 average/P95/P99。

`run_load_smoke.ps1` 还导出 500ms `timeseries.csv` 和独立的
`worker_resources.csv`；前者记录 broker/task/outbox 时序，后者记录 Docker
worker 容器 CPU/内存。`run_worker_scaling_repeated.ps1` 随机化组序并计算三次
本机运行上的 95% Student t 区间。

`active_workers` 不是 worker utilization；Redis backlog 是 consumer group 的 `pending + lag`，
也不是数据库全部 queued task 数。
