## v0.3.1 reference 对应的简历表达草稿

设计并实现一个面向 AI workload 的异步任务平台 reference，基于 FastAPI、PostgreSQL、Redis 和 transactional outbox 提供任务创建、状态查询、故障恢复与阶段指标；接入固定语料 BM25 RAG retrieval，按 tenant 和 permission group 前置过滤并持久化来源 metadata。只有学习者亲手复现后才能把这段改成个人完成表述。

## 可拆分亮点

### 工程能力

- 设计 `POST /tasks`（`task_type=rag_retrieval`）、`GET /tasks/{task_id}` 等 owner-scoped 任务 API。
- 实现 `queued/running/succeeded/failed` 状态流转。
- 使用 Docker Compose 串联 API、数据库、Redis、worker。

### AI workload 建模

- 将 RAG 请求建模为 `TaskRecord(task_type=rag_retrieval)`。
- 为后续 Agent 工作流预留通用 task envelope；v0.3.1 只有 `mock_agent` 模拟 workload，尚未实现 AgentTask、step logs 或真实工具执行。
- 支持 mock workload 和固定语料 BM25 retrieval，为后续 embedding、LLM generation 和 Agent 预留接口。

### 调度与指标

- 记录 queue wait、task runtime、total latency、error_type。
- 复用 P01/M05 的调度指标口径，支持后续 FIFO/Priority/SJF/Cost-aware 对比。
- 支持 E08 压测记录和 RQ01 科研训练。

## 面试讲法

### 项目为什么做

RAG 和 Agent 请求不是普通同步 API，它们耗时长、失败路径多、成本不稳定，所以更适合被建模成任务队列里的 workload。P03 的目标就是把这些请求变成可排队、可监控、可压测的工程系统。

### 为什么保留 mock 与 retrieval 两种 workload

mock 用于稳定复现队列、租约和压测；BM25 retrieval 用于验证 tenant、permission、sources 和异步结果持久化。两者都不能替代 LLM generation 或企业级检索评测。

### 不能夸大的地方

- 没有真实压测前，不说优化了多少 P95。
- 只有固定小语料 BM25 reference，不说达到企业级检索或生成效果。
- 没有权限系统前，不说支持生产级多租户。
- 当前只有 disposable kind 功能性 reference、没有 HPA/生产化和真实 vLLM，不说完成云原生推理平台。
