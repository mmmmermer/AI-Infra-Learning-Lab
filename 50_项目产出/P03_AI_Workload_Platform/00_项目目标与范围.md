# P03 项目目标与范围

> 学习者 v0.1 设计页：用于从零拆解第一版目标，不是当前 reference API 或状态真源。当前实现
> 以 `p03_service/README.md` 和 `03_API与数据契约.md` 为准。

## 项目目标

实现一个面向 RAG/Agent 请求的最小 AI workload 平台，用来训练后端 API、任务队列、worker、状态持久化、调度指标、Docker 启动和基础压测能力。

P03 的核心问题是：

```text
AI 请求进入系统后，如何创建任务、排队、执行、记录状态、暴露指标，并用数据解释慢在哪里？
```

## v0.1 范围

v0.1 只做可开工、可运行、可记录的最小版本。

必须包含：

- FastAPI API。
- `POST /tasks` 创建任务；RAG 检索使用 `task_type=rag_retrieval`，业务参数放入 `input_json`。
- `GET /tasks/{task_id}` 查询任务状态。
- tasks 表；如果数据库暂时未接好，可以先用内存 dict 替代，但接口和字段要按数据库方案设计。
- v0.1 先用进程内队列理解任务闭环；工程化升级统一采用 PostgreSQL task + outbox、dispatcher、Redis Streams consumer group 和带 fencing 的 worker，不把 RQ 写成当前 P03 实现。
- 模拟 RAG worker：不要求真实向量库，允许 `RAG_MODE=mock`。
- `mock_agent` 只作为确定性模拟 workload；`AgentTask`、`step_logs`、工具执行和人工审批属于后续 target design，不计入 v0.1 当前能力。
- 基础 metrics：queue wait、task runtime、total latency、error count、worker utilization 的最小记录。
- Docker Compose 启动 api、db、redis、worker 的方案。

## 暂不做

- 不做完整企业知识库平台。
- 不做生产级权限系统。
- 不做复杂文档解析、OCR、完整 PDF/Word 管线。
- 不做真实投资建议、股价预测或金融结论。
- v0.1 核心不做 Kubernetes；后续 E09 已补 disposable kind 功能性 reference，但不纳入 v0.1 完成标准，也不代表生产部署。
- 不做 vLLM/Triton 底层推理优化。
- 不做完整 Prometheus/Grafana 运维平台。
- 不做复杂多 Agent、AutoGPT、强化学习 Agent。
