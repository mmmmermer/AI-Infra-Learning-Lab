# P03 需求与任务拆解

> 学习者 v0.1 设计页：用于从零拆解练习，不是当前 reference API 或状态真源。当前实现以
> `p03_service/README.md` 和 `03_API与数据契约.md` 为准。

## v0.1 用户故事

1. 作为使用者，我可以提交一个 RAG 查询请求，并拿到 `task_id`。
2. 作为使用者，我可以通过 `task_id` 查询任务状态和结果。
3. 作为开发者，我可以看到任务从 `queued` 到 `running` 再到 `succeeded/failed` 的状态变化。
4. 作为开发者，我可以通过日志和 metrics 判断慢在 API、排队、worker 还是 retrieval；generation 是后续扩展，不写入 v0.1 验收。
5. 作为学习者，我可以把 E03/E06/E08 的实验记录落到 P03 上。

## 任务拆解

| 编号 | 任务 | 来源 | 验收 |
|---|---|---|---|
| T01 | 建 FastAPI 项目骨架 | M02 | `/health` 可访问 |
| T02 | 定义 Task 状态模型 | M06/E06 | 有 `pending/queued/running/succeeded/failed` |
| T03 | 实现 `POST /tasks`，以 `task_type=rag_retrieval` 提交 RAG 任务 | M03/E03 | 返回 task 和 `created_new` |
| T04 | 实现 `GET /tasks/{task_id}` | M06/E06 | 返回状态、结果、错误 |
| T05 | v0.1 实现进程内队列；工程化阶段接 PostgreSQL outbox + Redis Streams consumer group | M05/M06 | task 能可靠入队，worker 只在数据库终态成功后 ACK |
| T06 | 实现模拟 RAG worker | M03/E03 | 能回写 answer/retrieved_sources/metrics |
| T07 | 规划 AgentTask/step logs 契约；v0.1 只保留 `mock_agent` 确定性 workload | M04/E04 | 设计边界可解释；不计作 Agent Runtime 已实现 |
| T08 | 记录基础 metrics | M08/E08 | 有 latency、error、queue_wait |
| T09 | 编写 Dockerfile 和 compose | M07/E07 | api/db/redis/worker 可启动 |
| T10 | 写实验记录入口 | E03/E06/E08 | 能沉淀真实执行记录 |

## 优先级

### P0：先跑通最小闭环

- `/health`
- `POST /tasks`
- `GET /tasks/{task_id}`
- 内存 tasks store
- 内存 queue
- 模拟 worker

### P1：工程化闭环

- tasks 表。
- PostgreSQL task + outbox、dispatcher、Redis Streams consumer group 和 CAS/fencing worker。
- Docker Compose。
- 日志和 metrics。
- E03/E06/E08 实验记录。

### P2：展示和场景化

- Agent v1（`AgentTask`、工具执行、`step_logs`、人工审批；planned）。
- 金融/合规文档样例。
- README。
- 简历表达。

## 不进入本轮

- 完整登录和权限系统。
- 生产数据库迁移体系。
- 真实大规模向量库集群。
- Kubernetes。
- vLLM/Triton 推理服务。
- 完整前端 Dashboard。
