# E06-01 任务状态持久化

> 已补可执行参考：`e06_sqlite_reference/`。它在单个 SQLite 事务中创建 task 与 outbox 事件，并通过条件更新保护状态迁移；当前共 29 个测试通过。reference 还验证 schema migration、数据库约束、索引计划、锁竞争恢复、连接关闭、事件来源保护、真实队列退避到期时间、过期 heartbeat 拒绝、投递语义和禁止 update/delete 的事件链。HTTP 认证与 owner/ACL 边界仍由 E03/P03 reference 提供证据。

## 实验目的

本实验用于把 P03 的 RAG/Agent 请求从“内存对象”变成“可保存、可查询、可恢复”的任务记录。

第一轮不追求完整数据库平台，只验证：

```text
POST /tasks（task_type=rag_retrieval）
-> 创建 task
-> 在同一事务写入 tasks 与 outbox
-> 状态从 pending 流转
-> GET /tasks/{task_id} 可查询
```

这一步是 E06 后续异步执行、失败重试、监控压测的地基。

## 数据表/字段

第一轮可以用 SQLite、PostgreSQL 或内存模拟表。重点不是数据库类型，而是字段和状态语义。

### tasks 表最小字段

```sql
CREATE TABLE tasks (
    task_id TEXT PRIMARY KEY,
    tenant_id TEXT NOT NULL,
    user_id TEXT NOT NULL,
    allowed_permission_groups_json TEXT NOT NULL,
    task_type TEXT NOT NULL,
    status TEXT NOT NULL,
    priority INTEGER NOT NULL DEFAULT 5,
    input_json TEXT NOT NULL,
    result_json TEXT,
    error_type TEXT,
    last_error TEXT,
    retry_count INTEGER NOT NULL DEFAULT 0,
    max_retries INTEGER NOT NULL DEFAULT 2,
    idempotency_key TEXT NOT NULL,
    created_at TEXT NOT NULL,
    queued_at TEXT,
    started_at TEXT,
    finished_at TEXT,
    updated_at TEXT NOT NULL,
    UNIQUE (tenant_id, user_id, idempotency_key)
);
```

`input_json` 只保存 `query`、`collection_id`、`top_k` 等业务输入。认证层先验证 bearer token，
再由服务端构造 principal；`tenant_id`、`user_id` 和
`allowed_permission_groups_json` 是服务端写入的 owner/ACL 快照，不能从请求体或
`input_json` 复制，也不能由客户端覆盖。这个最小快照不等于完整权限系统，但必须先把信任边界
放对。

### 状态用途

| 状态 | 用途 | 常见来源 |
|---|---|---|
| pending | 任务已创建，还没进入队列 | API 创建 task 后 |
| queued | 任务已入队，等待 worker | service enqueue 后 |
| running | worker 已开始执行 | worker 读取 task_id 后 |
| succeeded | 任务成功完成 | worker 回写 result_json |
| failed | 任务最终失败 | 超过重试次数或不可恢复错误 |
| retrying | 任务准备重试 | worker 捕获可重试错误 |
| cancelled | 用户或系统取消任务 | API 取消或超时策略 |

第一轮推荐把 `retrying` 当状态，把具体重试事件记录到日志或后续 `task_events` 表。不要把 `retried` 当状态，因为它表示“发生过重试”，不是任务当前在哪里。

### 数据库不变量不能只写在 Python 里

reference 把下面的不变量写入 SQLite schema，并用负向测试证明数据库会拒绝违规数据：

- `status` 只能取已声明状态；
- `0 <= retry_count <= max_retries <= 100`；
- `input_json/result_json` 必须是合法 JSON；
- 终态必须有 `finished_at`；
- queue lease 的 `worker_id` 与 `leased_until` 必须同时为空或同时存在；
- 同一 `(task_id, task_version)` 只能有一个状态事件。

这类约束不是为了替代应用校验。应用校验负责给出友好错误，数据库约束负责在并发、脚本直写或代码缺陷下守住最后一道不变量。

## API 示例

### 创建 RAG 任务

```http
POST /tasks
Content-Type: application/json
Authorization: Bearer reference-public-token
```

```json
{
  "task_type": "rag_retrieval",
  "priority": 5,
  "estimated_duration_ms": 0,
  "idempotency_key": "user-001-rag-risk-q1",
  "input_json": {
    "query": "合规条款样例中要求卖方做什么？",
    "top_k": 3
  }
}
```

请求体的任务信封只能包含服务端契约声明的字段，RAG 业务输入放在 `input_json`。若顶层或
`input_json` 出现 `tenant_id`、`user_id`、`permission_group`、`permission_groups` 或
`allowed_permission_groups`，当前 reference 返回 `422` validation error，且不得创建 task 或
outbox。有效 owner 和权限组只取自认证后的 server-owned principal。

预期响应：

```json
{
  "task": {
    "task_id": "task_001",
    "tenant_id": "tenant-reference",
    "user_id": "user-public",
    "allowed_permission_groups": ["public"],
    "task_type": "rag_retrieval",
    "priority": 5,
    "estimated_duration_ms": 0,
    "idempotency_key": "user-001-rag-risk-q1",
    "input_json": {
      "query": "合规条款样例中要求卖方做什么？",
      "top_k": 3
    },
    "status": "pending",
    "result_json": null,
    "error_type": null,
    "created_at": "2026-07-18T08:00:00Z",
    "queued_at": null,
    "started_at": null,
    "finished_at": null,
    "runtime_ms": null,
    "queue_wait_ms": null,
    "total_latency_ms": null
  },
  "created_new": true
}
```

### 查询任务状态

```http
GET /tasks/task_001
Authorization: Bearer reference-public-token
```

查询同样按 server-owned principal 做 owner scope；其他 tenant/user 对同一 `task_id` 应得到
隐藏式 `404`，不能借错误差异确认任务是否存在。

预期响应：

```json
{
  "task_id": "task_001",
  "task_type": "rag_query",
  "status": "queued",
  "retry_count": 0,
  "created_at": "...",
  "queued_at": "...",
  "started_at": null,
  "finished_at": null,
  "error_type": null,
  "result_json": null
}
```

### 更新状态的最小操作

```text
mark_queued(task_id)
mark_running(task_id)
mark_succeeded(task_id, result_json)
mark_failed(task_id, error_type, last_error)
mark_cancelled(task_id)
```

## 状态流转

第一轮只允许下面几条主路径：

```text
pending -> queued -> running -> succeeded
pending -> queued -> running -> failed
running -> retrying -> queued -> running -> succeeded
pending -> cancelled
queued -> cancelled
running -> cancelled
```

不要允许：

```text
succeeded -> running
failed -> running
cancelled -> running
```

除非后续明确设计“重新提交任务”，那也应该生成新 task。

## 实验步骤

### 步骤 1：创建任务

插入一条 `rag_query` 任务：

```text
task_type = rag_query
status = pending
input_json = query/top_k/collection_id
tenant_id/user_id = server principal owner
allowed_permission_groups_json = server principal ACL snapshot
```

记录 `created_at` 和 `updated_at`。确认 owner/ACL 字段来自测试认证 fixture，而不是从
`input_json` 读取。

### 步骤 2：拒绝伪造身份与权限

分别在请求体中加入 `tenant_id`、`user_id` 和 `permission_group`，每次只注入一个字段。
预期均返回 `422 forged_identity_fields`，并确认 tasks/outbox 行数不增加。再用合法业务请求创建
任务，确认记录中的 owner/ACL 快照仍与 bearer principal 一致。

### 步骤 3：查询任务

执行：

```text
GET /tasks/{task_id}
```

确认能看到 `pending`。

### 步骤 4：模拟入队

调用：

```text
mark_queued(task_id)
```

记录 `queued_at`。

### 步骤 5：模拟 worker 开始执行

调用：

```text
mark_running(task_id)
```

记录 `started_at`。

### 步骤 6：模拟成功完成

写入 result：

```json
{
  "answer": "根据测试文档，卖方需要提供出口合规声明。",
  "retrieved_sources": ["doc_compliance_clause_001_chunk_1"],
  "metrics": {
    "retrieval_ms": 12,
    "generation_ms": 80,
    "token_count": 320
  }
}
```

调用：

```text
mark_succeeded(task_id, result_json)
```

记录 `finished_at`。

### 步骤 7：模拟失败任务

再创建一条任务，模拟失败：

```text
error_type = no_relevant_chunk
last_error = "No chunk matched query under current permission_group."
status = failed
```

### 步骤 8：检查查询计划、连接所有权与锁恢复

调用 `TaskDatabase.explain_claim_plan()`，确认 claim 查询计划包含 `idx_queue_claim`。然后用一个连接执行 `BEGIN IMMEDIATE` 并保持事务，使用 `busy_timeout_ms=20` 的第二个连接尝试写入：第二个写入应在有界时间内报 `database is locked`；释放第一个事务后重试应成功。

这说明 SQLite 是单写者模型，不说明 PostgreSQL 会以相同方式加锁。PostgreSQL 还需要独立检查事务隔离级别、行锁顺序、deadlock victim 和重试边界。

普通事务使用 `TaskDatabase.connection()`；它在提交或回滚后必定关闭连接。只有上面的锁实验使用低层 `connect()` 长时间持有 writer，并在 `finally` 中显式 `close()`。运行 `python -X dev -W error::ResourceWarning -m pytest -q` 后不得出现未关闭数据库警告，测试结束后 SQLite 文件应可立即删除。

真实 PostgreSQL deadlock 演示位于 `e06_sqlite_reference/examples/postgres_deadlock_demo.py`。它要求隔离数据库和 `requirements-postgres.lock`，预期恰好一个事务收到 `40P01`，另一个提交，且 victim 的第一笔更新整体回滚。一次 PostgreSQL 17.9 隔离实例实跑记录位于 `artifacts/postgres_deadlock_reference_2026-07-18.json`；该记录不是 CI 或生产并发证明，换环境后仍需重新运行并保留输出。

### 步骤 9：核对 reference API 下的 append-only 事件链

完成一次“失败一次后成功”的任务，查询 `task_events`，预期得到：

```text
pending(v0) -> queued(v1) -> running(v2)
-> retrying(v3) -> queued(v4) -> running(v5) -> succeeded(v6)
```

再尝试 `UPDATE` 或 `DELETE task_events`，数据库触发器应拒绝。使用旧 `worker_id + claim_version` 伪造失败，任务和事件数都必须保持不变。这个实验只证明 reference 方法与当前 schema trigger 下禁止改写，不证明拥有 DDL/管理员权限的人无法篡改；生产审计还需要独立权限、保留策略或外部不可变存储。

### 步骤 10：验证真实 retry due time

向 `TaskDatabase` 注入 `RetryPolicy`，让暂时性失败进入 `running -> retrying`。失败事务必须同时增加 `retry_count`、计算 `available_at` 并写 retry outbox；dispatcher 只能保留这个到期时间。分别在到期前一微秒和到期时调用 `claim_next()`：前者必须返回空，后者才可领取。再失败一次，确认第二次指数退避大于第一次且仍受 delay cap 限制。lease expiry 的自动恢复也必须推进 `available_at`，不能形成同一时间戳热循环。

## 失败场景

| 场景 | 预期状态 | error_type | 记录重点 |
|---|---|---|---|
| 客户端提交 tenant/user/权限字段 | 不创建任务 | forged_identity_fields | 422；tasks/outbox 行数不变 |
| collection_id 不存在 | failed | collection_not_found | input_json、last_error |
| 权限过滤后无结果 | failed | no_relevant_chunk_after_permission_filter | 服务端 ACL snapshot |
| worker 执行超时 | failed 或 retrying | worker_timeout | retry_count |
| 用户取消任务 | cancelled | user_cancelled | finished_at |
| 重复提交同一请求 | 返回已有 task | duplicate_idempotency_key | idempotency_key |

## 派生指标

任务状态持久化不是只为了“查得到状态”，还要让后续 M05/M08 能计算等待和执行耗时。

第一轮至少记录并计算：

| 指标 | 计算方式 | 用途 |
|---|---|---|
| queue_wait_ms | `started_at - queued_at` | 判断任务排队等待 |
| execution_ms | `finished_at - started_at` | 判断 worker 执行耗时 |
| total_latency_ms | `finished_at - created_at` | 判断用户感受到的总耗时 |
| retry_count | 每次进入 retrying 时加 1 | 判断失败重试成本 |

如果某个时间字段为空，说明任务还没有走到对应阶段，不能强行计算指标。

## 验收标准

- [ ] 能创建 `pending` 任务。
- [ ] 请求体只接受业务字段；伪造 tenant/user/permission 字段返回 422，且不创建 task/outbox。
- [ ] task 的 tenant/user/ACL snapshot 只由 server-owned principal 写入，客户端不能覆盖。
- [ ] 能通过 `GET /tasks/{task_id}` 查询任务。
- [ ] task 查询按 principal 做 owner scope；跨 tenant/user 查询返回隐藏式 404。
- [ ] 能完成 `pending -> queued -> running -> succeeded`。
- [ ] 能完成至少一个失败流转。
- [ ] 能记录 `created_at / queued_at / started_at / finished_at`。
- [ ] 能计算 `queue_wait_ms / execution_ms / total_latency_ms`。
- [ ] 能解释每个状态的用途。
- [ ] 能说明这些字段如何服务 M05 的 queue wait 和 M08 的 latency 指标。
- [ ] 能用数据库负向测试证明非法状态和超额重试被拒绝。
- [ ] claim 查询的 `EXPLAIN QUERY PLAN` 包含预期索引，并能说明该证据只适用于 SQLite reference。
- [ ] 能复现写锁超时、释放锁后恢复，并区分 SQLite 锁竞争与 PostgreSQL deadlock。
- [ ] 普通连接在事务后确定关闭，开发模式无 `ResourceWarning`，数据库文件可立即删除。
- [ ] stale outbox 不能把终态重新打开；事件类型只接受固定来源状态。
- [ ] 真实 queue 的 `available_at` 遵守注入的指数退避、cap 与 jitter，到期前不能领取。
- [ ] `fail_task` 的同一条条件更新同时匹配 status、version、owner 和未过期 lease。
- [ ] 每次成功状态转换恰好追加一个对应版本事件，旧 owner 不能伪造事件。

## 暂时不做

- 不做复杂索引优化，但必须能读取关键 claim 查询的基础查询计划。
- 不做分布式事务。
- 不做生产身份提供方、动态权限管理或实时撤权；本实验只验证最小 server-owned principal、owner scope 和 ACL snapshot 边界。
- 不做带签名或外部 WORM 存储的合规审计链；本实验只用数据库触发器保护状态事件不被更新和删除。
- 不做 Celery/RQ 的全部特性。
