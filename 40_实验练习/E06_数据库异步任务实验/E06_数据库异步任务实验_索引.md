## 定位

E06 用于把 M06 的教材内容落到可观察的小实验上。

它验证三件关键工程问题：

1. 任务状态是否能持久化，API 重启后还能查询。
2. 文档解析、embedding 或 RAG 请求是否能异步执行。
3. Redis 缓存是否能减少重复计算和降低延迟。

实验结果最终要回到 P03：让 RAG/Agent 请求成为可保存、可恢复、可异步执行的 workload。

## 总练习路线

E06 的实验顺序应当形成一条完整工程闭环：

```text
任务入库
-> 状态更新
-> 异步执行
-> 失败重试
-> 查询状态
```

不要先追求多个队列框架。第一轮使用 SQLite reference 的 `tasks + outbox + queue_messages + task_events`，验证数据库不变量、事务、幂等、CAS、lease、heartbeat、reconciliation 和 crash point；再映射到 P03 v0.3.1 的 PostgreSQL + Redis Streams consumer group（`XADD / XREADGROUP / PEL / XAUTOCLAIM / XACK + XDEL`）。RQ/Celery 只作概念对照。

| 阶段 | 推荐落点 | 关键动作 | 验收 |
|---|---|---|---|
| R1 任务入库 | E06-01 | 插入 `tasks` 记录，status = pending | `GET /tasks/{task_id}` 能查到任务 |
| R2 状态更新 | E06-01 | 更新 queued/running/succeeded/failed | 时间字段和 error_type 正确 |
| R3 异步执行 | E06-02 | API 返回 task_id，worker 后台执行 | API 不同步等待长任务 |
| R4 失败重试 | E06-02 | 分别模拟 `collection_not_found` 与 `worker_timeout` | 确定性错误直接 failed；暂时错误才增加 retry_count |
| R4b 崩溃恢复 | SQLite/P03 reference | lease 过期、reconciliation、重新 claim/pending delivery | 错误 worker heartbeat 和旧 claim finalization 都被拒绝 |
| R4c 失效与风暴 | SQLite reliability model + TaskDatabase | 对比三种投递语义与 200-task storm；把确定性 jitter due time 接入真实 queue | 能从 crash point 推导丢失/重复；到期前不可领取；jitter 后峰值下降且重试有上限 |
| R4d 事件审计 | SQLite `task_events` | 查询状态版本链；尝试伪造 update/delete；旧 owner 补写 | 版本连续；每次迁移一个事件；伪造操作均被拒绝 |
| R5 查询状态 | E06-01/E06-02 | 查询单任务、按 status 查询、计算耗时 | queue_wait_ms / execution_ms 可算 |
| R6 缓存与隔离 | E06-03 + E03 cache reference | 使用服务端 ACL fingerprint/version、cache key 和 TTL | 能记录 hit/miss/saved_calls，并证明 public 结果没有 private chunk |

## 同步与异步观察点

做 E06-02 时必须对比同步和异步两种写法：

| 观察项 | 同步执行 | 异步执行 |
|---|---|---|
| API 返回内容 | 直接返回结果 | 返回 task_id |
| 用户等待 | 等 worker 做完 | 轮询 `GET /tasks/{task_id}` |
| 失败记录 | 容易只返回错误响应 | 写入 tasks 表 |
| 重试位置 | 客户端或 API 重试 | worker 根据 retry_count 重试 |
| 和 M05 关系 | 难以统一调度 | 可以进入队列和调度 |

验收时不能只看“最后有没有成功”，还要记录 API 是否快速返回、状态是否可查询、失败是否可复盘。
