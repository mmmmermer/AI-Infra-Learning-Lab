# E06-02 文档解析与 RAG 请求异步化

## 实验目的

本实验用于把耗时任务从同步 API 调用改造成异步执行。

目标闭环是：

```text
API 认证并从服务端 principal 取得 tenant/user/有效 ACL
-> 同一数据库事务写入 task(pending) + outbox
-> API 返回 task_id，dispatcher 后台处理 outbox
-> dispatcher 在事务内 claim outbox，并先把 pending/retrying 条件更新为 queued
-> dispatcher 发布 task_id
-> worker 领取 task_id，并用 CAS + lease claim 为 running
-> worker 执行文档解析或 RAG
-> worker 按 owner + claim_version + 未过期 lease 回写 result_json / error_type
-> GET /tasks/{task_id} 查询状态
```

第一轮使用 `e06_sqlite_reference/` 的 SQLite outbox/queue reference；不再用无法验证崩溃恢复的纯伪队列充当完成实验。P03 v0.3.1 再把同一语义映射到 Redis Streams consumer group，RQ 只作概念对照。

边界要分清：E06 reference 现在已验证 owner/ACL-aware schema v2、bearer fixture、FastAPI create/query、outbox、状态、lease、重试、fencing 和单进程 worker；`tests/test_api_integration.py` 将 API -> SQLite -> worker -> safe cache 串成闭环。它仍不实现真实身份提供方、跨进程 broker、Redis Streams PEL 或生产 RAG 模型；这些仍分别由 E03/P03 reference 或后续外部实验提供证据，不能由 E06 的进程内 TestClient 冒充。

最小执行入口：

```powershell
cd "40_实验练习/E06_数据库异步任务实验/e06_sqlite_reference"
.\.venv\Scripts\python.exe -m pytest -q tests/test_api_integration.py
```

预期 `6 passed`。其中 worker fixture 只消费已由 dispatcher 转为 queued 的 task；测试不会把 `POST /tasks` 的 pending 响应提前写成“已经入队”。

## 数据表/字段

沿用 E06-01 的 `tasks` 表。下表同时列出 E06 可靠性子集与接入 P03 时必须存在的身份字段：

| 字段 | 用途 |
|---|---|
| task_id | worker 只拿这个 id |
| tenant_id / user_id | 由认证后的服务端 principal 写入，不接受客户端覆盖 |
| allowed_permission_groups | 本次提交时的服务端有效 ACL 快照，不放入 `input_json` |
| task_type | `document_parse` / `rag_retrieval` |
| status | pending / queued / running / succeeded / failed / retrying / cancelled |
| input_json | 保存文档路径、query、top_k 等输入 |
| result_json | 保存解析结果或 RAG answer/retrieved_sources/metrics |
| error_type | 失败分类 |
| last_error | 失败详情 |
| retry_count | 当前重试次数 |
| max_retries | 最大重试次数 |
| worker_id / version / lease_until | 隔离并 fence 每次 worker claim |
| queued_at / started_at / finished_at | 计算等待和执行耗时 |

`e06_sqlite_reference/` 使用 `outbox` 与 `queue_messages` 表在一个 SQLite 数据库里模拟可靠投递。P03 把队列替换为 Redis Streams，但任务事实仍在数据库：

```sql
CREATE TABLE outbox (
    event_id INTEGER PRIMARY KEY AUTOINCREMENT,
    task_id TEXT NOT NULL,
    event_type TEXT NOT NULL,
    payload_json TEXT NOT NULL,
    created_at TEXT NOT NULL,
    published_at TEXT
);

CREATE TABLE queue_messages (
    message_id INTEGER PRIMARY KEY AUTOINCREMENT,
    task_id TEXT NOT NULL,
    available_at TEXT NOT NULL,
    leased_until TEXT,
    worker_id TEXT,
    delivery_count INTEGER NOT NULL DEFAULT 0
);
```

SQLite reference 能把 queue row、`queued` 状态和 `published_at` 放在同一事务中。使用外部 Redis 时则必须先在数据库事务里 claim outbox 并完成 `pending/retrying -> queued`，提交后才 `XADD`；否则 worker 可能先看到消息，却仍无法 claim 一个还处于 `pending` 的任务。

映射到 P03 v0.3.1 的 Redis Streams 后，可靠消费顺序必须保持为：

```text
dispatcher: XADD task_id
worker: 先 XAUTOCLAIM 回收超过 idle 阈值的 PEL delivery，再 XREADGROUP 读取新消息
-> delivery 留在 consumer group 的 pending entries list（PEL）
-> worker 用数据库 CAS + lease + claim_version 取得本次写权限
-> 数据库成功提交 succeeded / failed / retrying
-> 最后 XACK + XDEL
```

执行异常、数据库终态更新返回 0 行或旧 worker 已失去 lease 时都不能 `XACK`。该 delivery 应留在 PEL 等待 `XAUTOCLAIM`；数据库中的 lease reconciliation 负责把中断的 `running` 任务转成 `retrying/failed`。这里的 PEL pending message 与任务表的 `status=pending` 是两个不同概念。

## API 示例

### 异步创建 RAG 任务

```http
POST /tasks
Authorization: Bearer reference-public-token
Content-Type: application/json
```

```json
{
  "task_type": "rag_retrieval",
  "idempotency_key": "rag-sources-001",
  "priority": 5,
  "input_json": {
    "query": "RAG 为什么需要 sources？",
    "collection_id": "demo_docs",
    "top_k": 3
  }
}
```

> 上述请求体只有业务输入。`tenant_id`、`user_id`、`permission_group(s)`、`allowed_permission_groups` 等身份或授权字段一律不能由客户端提供；服务端根据 Bearer token 解析 principal，并把当时的有效 ACL 快照写入任务记录。尝试伪造这些字段应以 `422 forged_identity_fields`（或项目统一的等价验证错误）拒绝，而且不得创建 task/outbox。

预期响应必须快速返回，不等待 dispatcher 或 worker 执行。刚提交时应诚实返回 `pending`，不能提前宣称已经入队：

```json
{
  "task": {
    "task_id": "task_rag_001",
    "status": "pending"
  },
  "created_new": true
}
```

查询地址由接口契约确定为 `/tasks/task_rag_001`，不需要假设响应中存在未实现的 `poll_url` 字段。

### 查询任务状态

```http
GET /tasks/task_rag_001
Authorization: Bearer reference-public-token
```

查询也必须使用服务端 principal 做 owner scope；其他 tenant/user 对同一 `task_id` 应得到 `404`，不能据此确认任务是否存在。

运行中：

```json
{
  "task_id": "task_rag_001",
  "status": "running",
  "retry_count": 0,
  "started_at": "...",
  "result_json": null
}
```

完成后：

```json
{
  "task_id": "task_rag_001",
  "status": "succeeded",
  "result_json": {
    "answer": "...",
    "retrieved_sources": ["doc_public_rag_001_chunk_1"],
    "metrics": {
      "retrieval_ms": 15,
      "generation_ms": 90,
      "token_count": 280
    }
  }
}
```

## 状态流转

异步执行的主线：

```text
pending
-> queued
-> running
-> succeeded
```

失败重试主线：

```text
running
-> retrying
-> queued
-> running
-> succeeded
```

最终失败：

```text
running
-> retrying
-> queued
-> running
-> failed
```

`cancelled` 仅是当前 schema 的预留枚举；P03 v0.3.1 没有取消 API，本实验不把下面的迁移计为已实现：

```text
pending / queued / running
-> cancelled
```

## 实验步骤

### 步骤 1：同步版本基线

先记录同步调用的体验：

```text
POST /rag/query  # 同步基线，仅作对照，不是 P03 v0.3.1 当前接口
-> API 内直接执行检索和生成
-> 返回 answer
```

记录：

- api_response_ms
- 是否超时
- 失败是否写入 tasks 表

### 步骤 2：异步创建任务

改成：

```text
POST /tasks
-> transaction: create task(pending) + append outbox event
-> return task_id
-> dispatcher transaction: claim outbox + compare-and-set pending/retrying -> queued
-> XADD task_id
-> acknowledge outbox only after publish succeeds
```

API 响应里只返回 `task_id` 和查询地址。若 dispatcher 在发布前崩溃，过期 outbox claim 可重新取得；若在发布后、确认 outbox 前崩溃，允许重复投递，worker 必须用 CAS 和 fencing 拒绝重复执行。

### 步骤 3：worker 消费 task_id

worker 不从队列里拿完整任务内容，只拿：

```text
task_id
```

然后：

```text
load task from tasks table
XAUTOCLAIM stale PEL delivery, otherwise XREADGROUP new delivery
compare-and-set queued -> running; set worker_id + lease_until; increment and return version
execute task
heartbeat/finalize must match worker_id + claim_version + unexpired lease
database finalization succeeds, then XACK + XDEL; otherwise leave it in PEL
```

这样 worker 重启或失败时，可以从数据库恢复上下文。

### 步骤 4：模拟成功任务

执行一个 `rag_retrieval`：

```text
query = RAG 为什么需要 sources？
top_k = 3
```

worker 回写：

```json
{
  "answer": "RAG 需要 sources 来支撑回答和方便核验。",
  "retrieved_sources": ["doc_public_rag_001_chunk_1"],
  "metrics": {
    "retrieval_ms": 15,
    "generation_ms": 90,
    "token_count": 280
  }
}
```

### 步骤 5：验证确定性错误与可重试错误

设计一个会失败的输入：

```json
{
  "query": "不存在的文档内容是什么？",
  "collection_id": "missing_collection",
  "top_k": 3
}
```

`missing_collection` 是确定性输入/资源错误，第一次执行就必须终止：

```text
status = failed
retry_count = 0
error_type = collection_not_found
new outbox events = 0
```

随后另建一个任务，模拟暂时性 `worker_timeout`。只有这类可恢复错误且 `retry_count < max_retries` 时才允许：

```text
running -> retrying
-> transaction: increment retry_count + append retry outbox
-> dispatcher: retrying -> queued
```

超过最大次数：

```text
status = failed
```

### 步骤 6：查询状态

对每个阶段执行：

```text
GET /tasks/{task_id}
```

确认用户能看到任务状态，而不是只能等最终结果。

## 失败场景

| 场景 | 预期处理 | error_type |
|---|---|---|
| collection_id 不存在 | 直接 failed，不重试 | collection_not_found |
| 客户端提交身份/权限字段 | 请求阶段拒绝，不创建任务 | forged_identity_fields |
| worker 执行超时 | 未到上限进入 retrying；达到上限 failed | worker_timeout |
| RAG 无证据 | failed，或按项目契约以 succeeded + warning metric 表达 | no_relevant_chunk |
| result_json 写入失败 | retrying | result_write_failed |
| 旧 worker 在 lease 过期/重新领取后回写 | CAS 拒绝，不改变新 owner 的结果 | stale_worker_claim |

## 验收标准

- [ ] API 能快速返回 `task_id`，不等待 RAG 执行完成。
- [ ] 请求体不接受 tenant/user/权限字段；服务端 principal 决定 owner 与有效 ACL，伪造字段的请求不创建 task/outbox。
- [ ] API 事务提交后状态为 `pending`；dispatcher 在发布前于数据库事务中 claim outbox 并完成 `pending/retrying -> queued`。
- [ ] 队列只保存 `task_id`，发布成功后才确认 outbox；能解释发布后确认前崩溃为什么会重复投递。
- [ ] 能区分任务表的 `pending` 与 Redis PEL，并用 `XAUTOCLAIM` 恢复未 ACK 的超时 delivery。
- [ ] worker 能用 `task_id` 从 tasks 表读取任务。
- [ ] worker 用 CAS + lease 领取任务，并用 worker_id + claim_version + 未过期 lease 回写 `running / succeeded / failed / retrying`。
- [ ] `GET /tasks/{task_id}` 能查询任务当前状态和最终结果。
- [ ] 能模拟一次 `worker_timeout` 重试，并最终 succeeded 或 failed。
- [ ] 能证明 `collection_not_found` 直接 failed、`retry_count=0` 且不会新增 retry outbox。
- [ ] 能证明过期或已被重新领取的旧 worker 无法完成/失败任务。
- [ ] 能证明只有数据库终态提交成功后才执行 `XACK + XDEL`，条件更新失败或进程异常时消息仍留在 PEL。
- [ ] worker 崩溃后可通过 lease expiry + reconciliation 恢复。
- [ ] 能计算 `queue_wait_ms / execution_ms / total_latency_ms`。
- [ ] 能说明该实验如何把 M05 的 Queue/Worker/Scheduler 接到工程服务。

## 和 M05 的连接

| M05 概念 | E06 工程落点 |
|---|---|
| Task | tasks 表中的一行 |
| Queue | E06 的 queue_messages + lease；P03 的 Redis Streams consumer group/PEL |
| Worker | 后台进程，读取 task_id 并执行 |
| Scheduler | 第一轮可以先 FIFO，后续接 Priority/SJF |
| Metrics | queue_wait_ms、execution_ms、total_latency_ms、retry_count |
