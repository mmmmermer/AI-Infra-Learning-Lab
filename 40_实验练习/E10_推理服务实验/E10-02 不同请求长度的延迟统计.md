# E10-02 不同请求长度的延迟统计

> 状态（2026-07-18）：`内容 content-reviewed / 实现 executable / Reference verified / 教学 partial / 归属 reference / 学习者 not-evaluated`。Python 3.13 下 7 个测试和 demo 已通过；本实验报告合成 workload 的确定性结果，不报告真实模型质量或 GPU 性能。

## 实验问题

在固定单 worker、固定成本 profile 和固定到达序列下，请求长度混合如何影响：

- `queue_wait_ms`
- `ttft_ms`
- `total_latency_ms`
- aggregate output tokens/s
- P95/P99

## workload 形状

参考 smoke workload：

| request_kind | prompt_tokens | output_tokens |
|---|---:|---:|
| short_chat | 300 | 100 |
| rag_answer | 2000 | 300 |
| long_report | 6000 | 1000 |

这只是输入形状，不是自然流量分布。正式对比必须同时保存到达时间、重复次数、随机种子或完整原始任务表。

## 两类实验必须分开

### 隔离服务成本

让请求间隔大于最长服务时间，消除排队，只比较请求长度如何影响合成 service/TTFT。

### 混合队列压力

让到达间隔小于部分任务的服务时间，观察长请求如何把后续请求的 queue wait 和尾延迟推高。

不能用混合队列结果直接声称“长 prompt 自身导致全部 P99”；其中还包含到达顺序和前序任务阻塞效应。

## 百分位方法

参考实现显式使用 nearest-rank：

```text
rank = ceil(p * n)
percentile = sorted_values[rank - 1]
```

注意：

- 30 个请求时 P99 基本等于最大值，只适合 smoke check。
- 研究报告至少应扩大样本，并报告重复运行或置信区间。
- 不同工具的 percentile 插值规则可能不同，跨工具比较前必须统一。
- P99 不能由平均值推导，也不能用单次最慢请求代替而不说明方法。

## 可归因规则

比较请求长度时固定：

- worker 数量
- 调度策略
- profile 参数
- 到达序列
- request count
- 统计公式
- 代码版本

如果比较 FIFO 与 predicted SJF，再单独改变调度策略，并复用同一组任务和到达时间。oracle SJF 不能伪装成可部署策略。
