# GF06-03 数据泄漏检查

## 实验定位

本实验不是继续优化策略，而是检查回测结果是否可能因为未来信息或样本选择而失真。

它服务 F06/F07/M11：

```text
backtest code
-> signal timing check
-> position shift check
-> sample selection check
-> parameter tuning note
-> leakage_flags
-> failure_analysis
```

## 实验目标

- [ ] 能识别 look-ahead bias。
- [ ] 能检查 `signal -> position -> return` 的时间顺序。
- [ ] 能记录 survivorship bias、data snooping、selection bias 的风险。
- [ ] 能写失败分析，而不是只保留好结果。
- [ ] 能把检查结果写入 P03 `backtest_task.result_json.leakage_check`。

## 实验步骤

### 步骤 1：检查 signal 是否使用未来数据

回答：

| 问题 | 记录 |
|---|---|
| signal 使用哪些字段？ |  |
| 这些字段在 signal 生成时是否可见？ |  |
| rolling 是否只用历史窗口？ |  |
| 是否存在负 shift 或未来收益参与特征？ |  |

### 步骤 2：检查 position 是否晚于 signal 生效

合格写法：

```python
df["position_at_open"] = df["signal_at_close"].shift(1).fillna(0)
df["strategy_return"] = df["position_at_open"] * df["open_to_close_return"]
```

危险写法：

```python
df["strategy_return"] = df["signal_at_close"] * df["open_to_close_return"]
```

记录：

```text
position_shift_checked: true/false
position_shift_value: 1
decision_time: close
fill_time: next_open
return_interval: same_day_open_to_close
```

`shift(1)` 不是唯一验收项。若 decision/fill/return interval 没写清，检查仍不通过。

### 步骤 3：运行存活偏差和 best-of-many 固定反例

运行 F06 第 6.4、6.5 节的两个 `runnable` 代码块，并保存：

| 反例 | 必须保存的证据 |
|---|---|
| survivorship | 完整 point-in-time universe、退市收益、survivor-only 对照 |
| data snooping | 200 个候选的训练/测试结果、选择规则、candidate count、test access count |

固定 seed 的精确数值只是回归夹具。验收重点是“完整资产池比幸存资产结果更差”和“测试集
不参与选择”，不是要求每个真实数据集都得到相同方向或幅度。

### 步骤 4：记录样本选择和参数调整

| 检查项 | 说明 |
|---|---|
| date_range_reason | 为什么选这个时间范围 |
| parameter_trials | 尝试了多少窗口或参数 |
| best_result_only | 是否只保存最好结果 |
| candidate_count | 一共生成/比较多少候选 |
| test_access_count | 冻结方案前读取 final test 的次数，目标为 0 |
| data_source_limit | 数据源有什么限制 |

### 步骤 5：写 leakage_flags


## 常见错误

- 只检查代码能不能跑，不检查时间逻辑。
- 只保存最好参数，不记录尝试过程。
- 使用当前成分股回测历史组合，却不说明幸存者偏差。
- 把未来收益作为特征。
- 把泄漏检查当成形式，不写具体风险。
