# E02-04 请求生命周期、可靠性与并发契约

## 实验定位

本实验把前三个 endpoint 从“路由能调用 repository”升级为一条可诊断、可恢复、可防竞态的请求
路径。目标不是模拟生产数据库，而是在确定性内存 reference 中证明这些契约：

- 同一 `request_id` 贯穿入口、认证、service、repository、错误响应与审计日志；
- principal 由服务端 DI 解析，owner 条件在 repository 内成立，日志不出现 bearer/tenant/user 原值；
- 所有预期错误使用 Problem Details 与稳定 `code`，readiness 反映依赖及连接容量；
- `GET /tasks` 以 owner-scoped opaque cursor 做有界分页，跨 owner cursor 不泄露记录；
- 429、依赖不可用和连接容量耗尽都给出恢复路径；
- deadline 过期不提交幽灵写，幂等键控制重复创建，`If-Match` 版本实现 CAS 更新。

## 运行

```powershell
cd e02_service
python -m pytest -q
```

当前 Python 3.13 reference 为 29 项测试通过。重点文件：

- `app/main.py`：middleware、exception handler、薄路由和 app factory；
- `app/dependencies.py`：`AppContainer`、service/rate limiter/request context DI；
- `app/service.py`：授权、deadline、domain error 到 API error 的翻译；
- `app/repository.py`：owner scope、连接 lease、幂等映射与原子 CAS；
- `tests/test_lifecycle_reliability.py`：16 个累计生命周期、可靠性、分页与并发场景。
- `tests/test_openapi.py`：2 个媒体类型、响应模型与 JSON Pointer 完整性场景。

## 一次请求如何穿过四层代码

```text
middleware: validate/generate request_id, compute deadline once
-> security dependency: credential -> immutable Principal
-> route: validate HTTP headers/body, consume mutation rate limit
-> service: scope + business orchestration
-> repository: connection lease + owner-scoped atomic operation
-> response/error handler: status + machine code + same request_id
```

测试向 `GET /tasks/not-visible` 发送 `X-Request-ID: trace-e02-001`，再从日志记录的结构化字段收集
`stage`。验收集合必须包含 `entry/security/service/repository/error_response`，响应 header 和 body 也必须
是同一个 ID。日志只保存 `sha256(tenant:user)` 的 12 字符 `subject_ref`，不记录 Authorization、
fixture bearer、tenant 或 user 原值。

未知异常同样不能直接使用 `logger.exception()` 输出异常 message，因为下游 SDK 的异常文本可能含 URL
凭据、SQL 参数或输入片段。reference 只记录受控的异常类型，以及 `文件名:行号:函数名` 栈帧位置；
既保留定位证据，又不把异常值或源码行复制进日志。

跨租户证据必须使用真正不同的服务端 principal，不能把同一 tenant 内 Alice/Bob 的 owner 隔离改名为
跨租户测试。`carol-fixture` 由服务端映射到 `tenant-other`；它读取 Alice 的已知 task ID 时得到与随机
未知 ID 同构的 404，列表为空、metrics 全零，错误正文和审计日志均不出现双方 tenant/user/credential
原值。客户端仍不能在请求体选择 tenant。

反例是在每层各自生成 trace ID：日志看似都有 ID，却无法连接同一次请求。另一个反例是把完整 headers
作为异常上下文输出，这会把 bearer token 直接写进日志。

## 机器可判定错误

所有预期错误使用 `application/problem+json`，OpenAPI 对每个 4xx/5xx（包括通用 500）也只声明这一
媒体类型，不能让文档写 `application/json` 而运行时返回另一种契约：

```json
{
  "type": "urn:e02:problem:version_conflict",
  "title": "Version conflict",
  "status": 412,
  "detail": "The task is currently at version 2.",
  "code": "version_conflict",
  "request_id": "trace-e02-001"
}
```

客户端分支依据是 HTTP status 与稳定 `code`，不是匹配英文 detail。422 的 `invalid_params` 只返回
字段位置与错误类型，不回显原始输入，避免把 secret-like 值复制到错误响应。

不能把 `ProblemDetails.model_json_schema()` 原样塞进任意 OpenAPI response 节点：Pydantic 可能生成
`#/$defs/...`，该 JSON Pointer 会从整份 OpenAPI 文档根解析，而不是从被内联的小 schema 解析。
reference 在嵌入前展开这些局部定义；测试递归遍历整份 `/openapi.json` 的每个本地 `$ref`，并从文档
根实际解析目标。只断言媒体类型正确，仍可能留下文档工具无法解析的悬空引用。

## Owner-scoped cursor pagination

`GET /tasks?limit=2&cursor=...` 的 limit 限制为 1-100，返回 `items` 与 `next_cursor`。repository
先按 principal 过滤，再按 task ID 稳定排序、定位 cursor 和切页。Alice 的 cursor 对 Bob 与任意坏
cursor 一样不能暴露 Alice 的记录或位置；不能先全局分页后过滤 owner。当前 base64url cursor 只是
教学不透明值，但仍严格限制为无填充 Base64URL，并要求 decode 后重新编码得到完全相同的 canonical
字符串；非法字符、多余 `=` 和非 canonical pad-bit 别名统一返回 400。它没有签名与过期；生产 cursor
还需绑定 owner、查询条件和版本。

## Liveness、readiness 与依赖容量

`GET /livez` 只证明应用进程能响应，不访问 repository。`GET /readyz` 尝试取得一次非阻塞 connection
lease：依赖被标记 unavailable 或 semaphore 容量耗尽时返回 503、稳定 code、`Retry-After: 1`；恢复
依赖或释放 lease 后再次返回 200。

连接容量测试使用 `TaskRepository(pool_size=1)`，先持有唯一 lease，再从 TestClient 请求 readiness。
它是确定性 pool-exhaustion fixture，不代表已经测试真实数据库连接池、排队超时或连接泄漏。

另一条测试让依赖恰好在 commit hook 内变为 unavailable：创建必须返回 503 且记录数保持 0；依赖恢复
后同一 app 再次创建成功。repository 因而既在取得 lease 时检查可用性，也在持锁提交前复查，不能把
“入口时可用”误当成整个写操作必然可用。

反例是 liveness/readiness 都固定返回 200。这样依赖故障时负载均衡器仍会继续发送新流量，放大失败。
另一个反例是 dependency 失败后永久把进程标成不健康；readiness 应在依赖恢复后自动恢复。

## 限流必须告诉客户端何时恢复

写端点按 `subject_ref` 使用固定窗口 limiter。测试注入 manual clock 和容量 1：第一次创建成功，第二次
返回 429、`Retry-After` 与 `retry_after_ms`，时钟推进一个窗口后第三次成功。manual clock 避免 sleep
导致慢测和抖动。

固定窗口是教学 reference，存在窗口边界突发，不是完整分布式限流器。生产环境还需共享状态、时钟
语义、代理身份边界和容量策略；不能从单进程测试声称多副本全局限流完成。

## Deadline：检查必须靠近提交点

客户端可发送 `X-Request-Deadline-Ms: 10`，middleware 在入口用 monotonic clock 计算一次绝对
deadline，后续只传播，不逐层重置预算。service 在进入操作前检查；repository 在持锁、真正修改
dict 前再次检查。

创建无幽灵写测试的 `before_commit` hook 把 manual clock 推进 20ms：请求得到 504
`deadline_exceeded`，并断言全局记录数仍为 0；随后用同一幂等 key 重试得到首次 201，证明超时路径
也没有遗留半条幂等映射。更新测试在版本 1 的 task 上用相同方式耗尽 deadline，断言 priority 和
version 都保持原值。若只在路由开始处检查，预算可能在等待连接或执行业务期间耗尽，随后仍提交写入；
客户端会认为失败并重试，从而产生重复记录。

这个内存原子段不能替代数据库事务。生产实现应让 deadline/事务取消与 commit 边界协同，并在无法
撤销的外部副作用上使用幂等键或 outbox。

## 幂等创建

客户端为同一逻辑创建发送稳定 `Idempotency-Key`：

1. 首次相同 key/payload 返回 201、`Idempotency-Replayed: false`。
2. 重复相同 key/payload 返回 200、同一 task ID、`Idempotency-Replayed: true`。
3. 同 key/不同 payload 返回 409 `idempotency_key_conflict`。

repository 在同一锁内检查 owner-scoped key、payload fingerprint、写 task 与写幂等映射，因此并发重复
不会跨过检查各写一次。自动测试用 barrier 同时放行两个相同请求，必须得到一个 201、一个 replay 200、
相同 task ID 且总数为 1。key 的作用域包含 tenant/user；Alice、同租户 Bob 与另一租户 Carol 使用
同一 key 时各自首次创建，不会互相重放或阻塞。

## 版本与 CAS 并发更新

创建记录从 `version=1` 和 `ETag: "1"` 开始。更新使用：

```http
PATCH /tasks/{task_id}
If-Match: "1"
Content-Type: application/json

{"priority": 4}
```

repository 在 owner 检查和锁内比较 expected/current version，成功后写入 `version=2`。测试让两个线程
同时以 `If-Match: "1"` 更新：恰好一个返回 200，另一个返回 412 `version_conflict`；最终只有 version
2，不出现“最后写入静默覆盖前一次成功”。

## 独立练习与验收

1. 为成功创建发送自定义合法 request ID，断言四层日志和响应一致；再发送含空格 ID，断言 400。
2. 用 `pool_size=1` 复现容量耗尽，释放 lease 后不重启 app 即恢复 readiness。
3. 创建 5 条任务，以 limit 2 完成 2/2/1 分页；另一个 owner 看不到记录或 cursor 位置。
4. 同一 idempotency key 并发发两次相同 payload，断言 task_count 仍为 1。
5. 读取 version 2 后用 `If-Match: "2"` 再更新到 version 3；旧 `"1"` 必须继续 412。
6. 捕获日志并断言不含 fixture bearer、Authorization、tenant/user 原值和完整请求 body。

完成口径是测试能分别定位每种失败，且每个恢复路径也有正向断言。只看 `/docs`、只测最终 status，或
用 sleep 猜并发顺序，都不足以证明无副作用、恢复和 CAS。
