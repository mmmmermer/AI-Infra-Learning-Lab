# E01-04 并发选择与取消

## 实验定位

本实验不是“把所有函数改成 async”。目标是先识别等待与计算的性质，再选择 thread、process 或
`asyncio`，最后证明取消/超时后资源已释放。它是 FastAPI、worker 和并发 API 的直接前置。

## 先做选择，不先选语法

| 工作负载 | 首选 | 原因 | 典型反例 |
|---|---|---|---|
| 现有同步文件/网络 I/O | thread | 等待可重叠，且同步库无需重写 | 为一次小文件读取启动进程 |
| CPU 密集 Python 计算 | process | 多进程隔离解释器锁与故障 | 在线程池里跑长时间纯 Python 循环 |
| 大量原生可等待 I/O | asyncio | 单线程协作调度大量等待 | 在 event loop 直接调用 `time.sleep()` |

`choose_concurrency_model()` 给出模型和理由。选择不是永久标签：如果库从同步 API 换成原生 async，
边界也会变化；若任务需要共享大量可变内存，process 的序列化和一致性成本必须计入。

## Worked example：阻塞 I/O benchmark

`benchmark_blocking_io()` 对同一组四项输入分别顺序执行和四线程执行，并断言结果顺序相同、线程
耗时更短。这个小计时只证明“sleep 型等待可重叠”，不代表线程对 CPU 工作也更快，更不能作为容量
规划。正式 benchmark 还需预热、重复样本、分位数、资源监控和固定机器环境。

## 反例：阻塞事件循环

`loop_progresses_during_direct_call()` 先安排一个 loop callback，再直接调用同步函数；函数返回前 callback
不可能得到调度，因此契约稳定返回 `False`。`loop_progresses_while_call_is_offloaded()` 用线程启动信号和
释放门闩固定观察窗口：worker 已启动但尚未释放时，event loop 仍能执行 callback，因此返回 `True`。
`finally` 会释放并等待受 `shield` 保护的 worker，避免 helper
被取消后底层线程仍继续运行。整个断言只比较事件顺序，不比较毫秒数，也不会因机器负载变化而误报。

不要把 `async def` 当成自动并发。函数内部没有 `await`，或在 `await` 之间执行长 CPU 循环，同样会
阻塞 loop。`to_thread` 适合迁移阻塞 I/O，不会把 CPU 密集 Python 代码自动变成多核并行。

## 取消不是调用 `cancel()` 就结束

reference 的资源探针遵守以下顺序：

```text
create task -> wait until resource acquired -> cancel/timeout -> await task -> finally release
```

显式取消和 `asyncio.timeout()` 两条路径都必须得到 `acquired == released == 1`。反例是调用
`task.cancel()` 后马上返回：取消只发出请求，`finally` 要到 task 再次得到调度并被等待后才有确定
证据。吞掉 `CancelledError` 后继续无限循环也会破坏结构化并发。

## 独立练习

1. 把 thread benchmark 输入改为 8 项并保留结果顺序断言，记录而不硬编码绝对耗时。
2. 新增一个 event-loop heartbeat，在阻塞反例中观察一次明显停顿，在 `to_thread` 变式中继续增长。
3. 给资源探针新增 `active` 计数，取消后断言回到 0。
4. 写出三个场景的选择理由：同步数据库驱动、纯 Python 特征计算、原生 async HTTP 客户端。

## 验收

- [ ] 能根据等待/计算性质选择 thread、process、asyncio，而不是根据函数是否写了 `async`。
- [ ] 线程 benchmark 保持输出一致，且不把一次计时外推为生产性能结论。
- [ ] event-loop blocking 反例与 `to_thread` 修复都由自动测试区分。
- [ ] 显式取消和 timeout 后 task 已被 await，资源获取/释放计数相等。
- [ ] 进程池运行的是模块顶层可序列化函数，并与顺序函数结果一致。
