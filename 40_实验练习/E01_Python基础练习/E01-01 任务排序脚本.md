# E01-01 任务排序脚本

## 实验定位

这是 P01 Mini Scheduler 的第一个动手实验。它的目标不是写一个复杂系统，而是先把调度器最核心的动作练出来：给一批任务排序。

后续 FIFO、Priority、SJF、P95 延迟、worker 利用率，本质上都建立在这个问题上：任务来了以后，系统先处理谁。

## 实验目标

完成一个 `task_sorter` 脚本，支持三种排序方式：

- FIFO：按提交时间排序
- Priority：按优先级排序
- SJF：按预计耗时排序

## 输入数据

先用 list + dict 表示任务，不急着上类。

```python
tasks = [
    {"id": "task-001", "submit_time": 0, "priority": 2, "estimated_duration": 5},
    {"id": "task-002", "submit_time": 1, "priority": 1, "estimated_duration": 2},
    {"id": "task-003", "submit_time": 2, "priority": 3, "estimated_duration": 1},
]
```

字段含义：

| 字段 | 含义 | 调度意义 |
|---|---|---|
| `id` | 任务编号 | 用来观察排序结果 |
| `submit_time` | 任务提交时间 | FIFO 需要它 |
| `priority` | 优先级 | Priority 需要它 |
| `estimated_duration` | 预计耗时 | SJF 需要它 |

## 实验步骤

### 第 1 步：写 FIFO 排序

> 环境要求：本实验以 Python 3.13 验收。使用 `py -3.13 -m venv .venv` 创建环境；激活后 `python --version` 必须显示 3.13。

```python
def sort_by_fifo(tasks: list[dict]) -> list[dict]:
    return sorted(tasks, key=lambda task: task["submit_time"])
```

FIFO 的含义是 first in, first out。谁先提交，谁先处理。

### 第 2 步：写 Priority 排序

```python
def sort_by_priority(tasks: list[dict]) -> list[dict]:
    return sorted(tasks, key=lambda task: (task["priority"], task["submit_time"]))
```

这里约定：数字越小，优先级越高。`priority=1` 比 `priority=3` 先执行。

加上 `submit_time` 是为了处理同优先级任务：同优先级时，先来的先执行。

### 第 3 步：写 SJF 排序

```python
def sort_by_sjf(tasks: list[dict]) -> list[dict]:
    return sorted(tasks, key=lambda task: (task["estimated_duration"], task["submit_time"]))
```

SJF 是 shortest job first。预计耗时越短，越先执行。它常常能降低平均等待时间，但可能让长任务一直被推迟。

### 第 4 步：打印结果

```python
def print_order(name: str, sorted_tasks: list[dict]) -> None:
    order = [task["id"] for task in sorted_tasks]
    print(f"{name}: {order}")
```

建议输出：

```text
fifo: ['task-001', 'task-002', 'task-003']
priority: ['task-002', 'task-001', 'task-003']
sjf: ['task-003', 'task-002', 'task-001']
```

### 第 5 步：加入最小 logging

```python
import logging

logging.basicConfig(level=logging.INFO, format="%(levelname)s:%(message)s")

logging.info("loaded %s tasks", len(tasks))
```

日志的目的不是好看，而是以后排查问题。调度器里至少应该记录：输入任务数量、使用的策略、输出顺序。

## 常见错误

### priority 方向写反

如果你写成 `reverse=True`，可能会让数字大的任务先执行。除非你明确约定“数字越大越优先”，否则本路线统一使用数字越小越优先。

### 把所有逻辑写在 for 循环里

能跑不代表结构好。排序规则应该是函数，因为后面 pytest 要直接测试函数。

### 修改了原始任务列表

建议用 `sorted()`，不要一开始就用 `tasks.sort()`。`sorted()` 返回新列表，更不容易影响后续测试。
