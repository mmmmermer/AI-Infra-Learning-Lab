# E05-03 高峰负载下的 P95 延迟实验

> 状态：`draft / executable reference / unverified research`。当前 52 个固定任务只适合教学演示。nearest-rank P99 在 52 个样本下基本等于最大值，不得据此形成科研结论。后续研究版需要更多任务、多随机种子、预测误差、重尾和突发负载，并报告置信区间与效应量。

## 实验定位

这一步要回答的问题是：

```text
当任务突然变多时，不同调度策略的尾延迟会怎么变化？
```

平均值经常会掩盖问题，所以这里要重点看 P95 / P99。

## baseline

- FIFO
- Priority
- SJF
- Cost-aware

## 实验目标

- 构造低负载和高峰负载两种任务流。
- 比较 FIFO / Priority / SJF / Cost-aware 在尾延迟上的差异。
- 观察队列长度在高峰期间如何变化。
- 解释为什么平均值不足以描述用户体验。

注意：本实验里的 Cost-aware 只作为第四种对照策略。权重公式、权重敏感性和分组影响放到 E05-04 与章节教材第 8 章，不在本实验里展开。

## 任务流设计
当前 P01 已经加入两个确定性任务流，代码位置：

```text
mini_scheduler/scheduler/workloads.py
```

### 低负载

低负载任务流由 `build_low_load_tasks()` 生成：

- 任务数量：24 个。
- 到达方式：每 4.0 个时间单位到达一个任务。
- 任务类型：`rag_query`、`agent_tool`、`embedding`、`long_context` 混合。
- 任务耗时：1.0、1.5、2.0、3.0、5.0 循环出现。
- 预期现象：worker 大多数时间不会长时间排队，四种策略差异应当较小。

低负载用于回答：

```text
系统没有明显拥塞时，策略差异是否真的重要？
```

### 高峰负载

高峰负载任务流由 `build_peak_load_tasks()` 生成：

- 总任务数量：52 个。
- warmup：8 个任务，较平稳到达。
- burst：36 个任务，在 `submit_time=20.0` 之后以 0.25 的间隔密集到达。
- cooldown：8 个任务，峰值后逐渐恢复。
- 任务类型：`agent_tool`、`rag_query`、`embedding`、`long_context`、`batch_job` 混合。
- 任务耗时：短任务和长任务混合，最长 8.0。
- 优先级：`priority=1/2/3` 混合。

高峰负载用于回答：

```text
当任务到达速度明显超过 worker 处理速度时，平均等待时间和 P95/P99 会如何分化？
```

## 推荐指标

| 指标 | 用途 |
|---|---|
| 平均等待时间 | 总体参考 |
| 最大等待时间 | 最坏情况 |
| P95 等待时间 | 尾部体验 |
| P99 等待时间 | 更极端的尾部 |
| 队列长度变化 | 拥塞强度 |

## 推荐步骤
1. 先运行低负载和高峰负载对比：

```bash
python examples/run_high_load_experiment.py
```

2. 观察低负载下四种策略是否差异很小。
3. 观察高峰负载下平均等待时间、最大等待时间、P95、P99 和最大队列长度。
4. 对比 FIFO、Priority、SJF、Cost-aware 的结果。
5. 单独解释：为什么某个策略平均值更好，但 P95/P99 不一定更好。
6. 写出一个后续优化假设，例如：

```text
如果后续第 10 章给低优先级任务加入 aging 机制，Priority 的 P95/P99 可能下降。
```

7. 将稳定结论同步到 P01 项目实验记录。

## 结果观察

你应该重点看这些现象：

- 高峰期是不是把尾延迟明显拉高了。
- SJF 是否更照顾短任务，但让长任务更痛苦。
- Priority 是否让高优先级任务更稳定，但把低优先级任务挤到后面。
- FIFO 是否在高峰下显得最“平均”，但不一定最舒服。

| Priority | 0.17 | 1.00 | 1.00 | 1.00 | 1 | 0.61 |

高峰负载结果：

| 策略 | 平均等待时间 | 最大等待时间 | P95 | P99 | 最大队列长度 | worker 利用率 |
|---|---:|---:|---:|---:|---:|---:|
| FIFO | 48.71 | 101.45 | 97.70 | 101.45 | 31 | 0.97 |
| Priority | 37.02 | 121.45 | 112.95 | 121.45 | 27 | 0.97 |
| SJF | 26.04 | 121.45 | 108.45 | 121.45 | 27 | 0.97 |
| Cost-aware | 26.51 | 121.45 | 108.45 | 121.45 | 27 | 0.97 |

### 初步结论

低负载下，四种策略表现几乎一样。这说明：当任务到达速度低于系统处理能力时，调度策略的差异不明显，worker 利用率也只有 0.61。

高峰负载下，差异开始显现：

- FIFO 的平均等待时间更高，但最大等待时间和 P99 反而低于 Priority/SJF/Cost-aware。
- SJF 和 Cost-aware 的平均等待时间明显更低，说明短任务被更快处理。
- Priority/SJF/Cost-aware 的最大等待时间达到 121.45，说明某些任务被长期推迟。
- 最大队列长度从低负载的 1 上升到高峰负载的 27-31，说明拥塞是尾延迟上升的直接原因。

这组结果说明：

```text
优化平均等待时间不等于优化尾延迟。高峰负载下，策略会改变谁受益、谁被推迟；因此必须同时看 average、P95/P99、max wait 和队列长度。
```

## 常见错误

- 只看平均等待时间，不看尾部。
- 任务流太小，看不出差异。
- 低负载和高峰负载的任务集合不是同一尺度。
- 忘了记录队列长度变化。
