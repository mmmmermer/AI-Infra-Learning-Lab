## 实验目标

- 给任务增加成本字段。
- 设计一个简单成本分数。
- 用成本分数排序任务。
- 和 FIFO / Priority / SJF 做对比。
- 解释成本感知调度的收益和风险。

## 成本字段

建议先使用这些字段：

| 字段 | 含义 | 为什么有用 |
|---|---|---|
| `estimated_duration` | 预计执行时间 | 表示任务占用 worker 的时间 |
| `token_count` | 预计 token 数 | 近似表示 LLM/RAG 请求成本 |
| `priority` | 业务优先级 | 表示任务重要程度 |
| `submit_time` | 到达时间 | 防止长期等待任务被忽略 |

## 最小成本分数
第一版先定义可审查的公式形状：

```text
cost_score = w_duration * f_duration(task)
           + w_token * f_token(task)
           + w_priority * f_priority(task)
```

`f_*` 不是预先给定答案。学习者必须决定方向、尺度和归一化方式，并说明当前约定
`priority=1` 最高时，priority 项如何影响选择顺序。

当前代码中提供了四组权重预设：

| 预设 | duration_weight | token_weight | priority_weight | 设计意图 |
|---|---:|---:|---:|---|
| default | 1.0 | 0.001 | 0.5 | 默认平衡版 |
| duration_dominant | 1.5 | 0.0005 | 0.3 | 更重视执行耗时，接近 SJF 思路 |
| token_dominant | 0.7 | 0.003 | 0.3 | 更重视 token 成本 |
| priority_dominant | 0.8 | 0.0005 | 2.0 | 更重视业务优先级 |

代码位置：

```text
mini_scheduler/scheduler/strategies.py
```

核心函数：

```python
calculate_cost_score(task, weights)
sort_by_cost_weights(tasks, weights)
```

这个公式不是标准答案。它的学习价值在于：你可以改变权重，观察平均等待时间、P95/P99、最大等待时间和队列长度如何变化。

## 推荐实现方式
当前实现不直接把所有任务全局排序后再执行，因为那会忽略任务到达时间。正确做法是：

```text
每一轮只从 submit_time <= current_time 的已到达任务中，根据 cost_score 选择下一个任务。
```

代码里通过 `run_single_worker(..., selector=...)` 支持自定义选择器：

```python
completed = run_single_worker(
    tasks,
    worker,
    selector=lambda available_tasks: sort_by_cost_weights(available_tasks, weights)[0],
)
```

这样既能测试不同成本权重，又不会让 worker 为了未来任务空等。

运行入口：

```bash
cd AI-Infra-Learning-Lab/50_项目产出/P01_Mini_Scheduler/mini_scheduler
python examples/run_cost_weight_experiment.py
```

## 实验步骤
1. 复用前面实验的 `Task` 模型。
2. 确认任务包含 `estimated_duration`、`token_count`、`priority`、`submit_time`。
3. 定义 `CostWeights`，把 duration、token、priority 三个权重显式写出来。
4. 写 `calculate_cost_score(task, weights)`。
5. 写 `sort_by_cost_weights(tasks, weights)`。
6. 构造专门的权重敏感任务流：`build_cost_sensitivity_tasks()`。
7. 分别跑 default、duration_dominant、token_dominant、priority_dominant 四组权重。
8. 对比平均等待时间、最大等待时间、P95、P99、最大队列长度。
9. 写出权重变化带来的收益和风险。

当前运行命令：

```bash
python examples/run_cost_weight_experiment.py
```

## 结果观察

重点看：

- Cost-aware 是否减少了高成本任务对队列的阻塞。
- 低成本短任务是否明显更快。
- 高成本长任务是否被推迟过久。
- priority 在成本公式里权重是否过大或过小。

## 记录模板
### 本轮任务流

代码入口：

```text
mini_scheduler/scheduler/workloads.py
build_cost_sensitivity_tasks()
```

任务流设计为 30 个密集到达任务，包含几类冲突任务：

| task_type | 特征 | 用途 |
|---|---|---|
| short_high_token | 执行短，但 token 很高 | 测 token 权重是否会推迟它 |
| long_low_token | 执行长，但 token 很低 | 测 duration 权重是否会推迟它 |
| urgent_medium | 高优先级，中等耗时 | 测 priority 权重是否会提前它 |
| low_priority_short | 低优先级，但很短 | 测 priority 是否牺牲短任务 |
| batch_heavy | 低优先级，耗时和 token 都高 | 测高成本任务是否被后置 |
| cheap_medium | 中等耗时，token 很低 | 测 token 主导时是否提前 |

### 本轮运行结果

命令：

```bash
python examples/run_cost_weight_experiment.py
```

| 权重预设 | 平均等待时间 | 最大等待时间 | P95 | P99 | 最大队列长度 | worker 利用率 |
|---|---:|---:|---:|---:|---:|---:|
| default | 41.97 | 104.40 | 99.60 | 104.40 | 24 | 0.92 |
| duration_dominant | 37.30 | 104.40 | 99.60 | 104.40 | 24 | 0.92 |
| token_dominant | 48.63 | 110.80 | 110.60 | 110.80 | 24 | 0.92 |
| priority_dominant | 46.63 | 104.40 | 99.60 | 104.40 | 27 | 0.92 |

### 初步结论

这组结果说明：

- duration_dominant 的平均等待时间最低，说明在当前任务流里，优先处理短耗时任务能改善整体平均等待。
- token_dominant 的平均等待、P95、P99、最大等待时间都更差，说明过度强调 token 成本可能会把某些任务推得太靠后。
- priority_dominant 的最大队列长度更高，说明强行照顾优先级可能让队列更不均衡。
- 四组权重的 worker 利用率相同，说明差异不是资源是否忙，而是谁被安排在前面、谁被推迟。

这一轮最重要的结论是：

```text
Cost-aware 调度不是写一个公式就结束，而是要通过权重实验观察策略偏向。不同权重会改变收益对象，也会改变尾延迟风险。
```

### 按 task_type 分组观察

同一条命令现在还会输出 `task_type_breakdown`：

```bash
python examples/run_cost_weight_experiment.py
```

#### duration_dominant

| task_type | count | 平均等待时间 | 最大等待时间 | P95 |
|---|---:|---:|---:|---:|
| batch_heavy | 5 | 94.80 | 104.40 | 104.40 |
| cheap_medium | 5 | 9.60 | 13.20 | 13.20 |
| long_low_token | 5 | 59.40 | 73.00 | 73.00 |
| low_priority_short | 5 | 0.40 | 0.40 | 0.40 |
| short_high_token | 5 | 32.40 | 40.80 | 40.80 |
| urgent_medium | 5 | 27.20 | 32.80 | 32.80 |

解释：duration 主导非常照顾短任务，`low_priority_short` 几乎立刻执行；但 `batch_heavy` 和 `long_low_token` 被明显推迟。

#### token_dominant

| task_type | count | 平均等待时间 | 最大等待时间 | P95 |
|---|---:|---:|---:|---:|
| batch_heavy | 5 | 90.80 | 100.40 | 100.40 |
| cheap_medium | 5 | 9.60 | 13.20 | 13.20 |
| long_low_token | 5 | 35.40 | 49.00 | 49.00 |
| low_priority_short | 5 | 0.40 | 0.40 | 0.40 |
| short_high_token | 5 | 88.40 | 110.80 | 110.80 |
| urgent_medium | 5 | 67.20 | 72.80 | 72.80 |

解释：token 主导会照顾 token 少的任务，例如 `long_low_token` 比 duration 主导下更早；但 `short_high_token` 虽然执行很短，因为 token 很高，被严重推迟。这就是 token 成本权重过高的风险。

#### priority_dominant

| task_type | count | 平均等待时间 | 最大等待时间 | P95 |
|---|---:|---:|---:|---:|
| batch_heavy | 5 | 94.80 | 104.40 | 104.40 |
| cheap_medium | 5 | 23.60 | 27.20 | 27.20 |
| long_low_token | 5 | 55.40 | 69.00 | 69.00 |
| low_priority_short | 5 | 35.40 | 35.40 | 35.40 |
| short_high_token | 5 | 64.40 | 80.80 | 80.80 |
| urgent_medium | 5 | 6.20 | 11.80 | 11.80 |

解释：priority 主导显著改善 `urgent_medium`，但会牺牲 `low_priority_short`。这说明业务优先级能保护关键任务，但如果没有 aging 或最大等待保护，低优先级任务会被系统性推后。

### 分组结论

总表只能告诉你哪个权重整体更好，分组表能告诉你谁被牺牲。当前实验最重要的工程结论是：

```text
Cost-aware 调度必须做分组分析。否则平均等待时间看起来变好时，可能只是把某些任务类型的等待时间转移到了尾部。
```



### Aging / 最大等待保护实验

在分组分析之后，下一步不是只说“某类任务被牺牲了”，而是尝试保护它。

当前新增了 aging 配置：

| 配置 | 含义 |
|---|---|
| no_aging | 不做等待保护 |
| gentle_aging | 等待越久，成本分数略微下降 |
| strong_aging | 等待超过阈值后启用硬最大等待保护 |

运行命令：

```bash
python examples/run_aging_experiment.py
```

总表结果：

| 配置 | 平均等待时间 | 最大等待时间 | P95 | P99 | 最大队列长度 | worker 利用率 |
|---|---:|---:|---:|---:|---:|---:|
| no_aging | 48.63 | 110.80 | 110.60 | 110.80 | 24 | 0.92 |
| gentle_aging | 48.63 | 110.80 | 110.60 | 110.80 | 24 | 0.92 |
| strong_aging | 42.66 | 104.40 | 100.80 | 104.40 | 26 | 0.92 |

分组对比重点：

| task_type | no_aging 平均等待 | strong_aging 平均等待 | 变化 |
|---|---:|---:|---|
| short_high_token | 88.40 | 52.40 | 明显改善 |
| batch_heavy | 90.80 | 68.80 | 改善 |
| urgent_medium | 67.20 | 65.20 | 平均略改善，但 P95 变差 |
| low_priority_short | 0.40 | 4.00 | 被轻微牺牲 |

结论：

```text
Aging / 最大等待保护可以降低极端等待，尤其能改善被 token 成本压制的任务；但它不是免费午餐，会改变原本的优先顺序，使部分原本很快的任务等待增加。
```

这一步把实验从“比较策略”推进到了“提出改进策略”。

## 常见错误

- 成本公式太复杂，自己也解释不清。
- 把 token 数权重设得过大，导致其他因素失效。
- 忘了和 FIFO / Priority / SJF 做 baseline 对比。
- 只说“成本更低”，但没有看长任务是否被饿死。
