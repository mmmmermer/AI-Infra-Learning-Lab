# E00-06 进程、信号、权限与请求路径诊断

## 实验定位

这个实验补齐“脚本能运行”和“服务可被运维”之间的基础层。完成后，你不仅要看到报错，还要能回答：

- 哪个 PID 在运行，父进程是谁，环境变量从哪里来，退出码代表什么；
- 服务收到终止请求后是否执行清理；
- 写文件前如何识别权限失败，并在修复权限后重试；
- 一次请求停在 DNS、TCP、TLS 还是 HTTP，为什么 HTTP 500 不是“网络不通”。

## 实验一：观察进程边界

`run_child_process()` 使用 `Popen` 取得启动器 PID，捕获 stdout/stderr，并保留退出码。测试让
`service_worker.py --exit-code 7` 明确失败，同时只传入非秘密环境变量 `DEMO_MODE=test`。

`communicate(timeout=...)` 抛出 `TimeoutExpired` 时不会自动停止子进程。reference 必须先 `kill()`，
再调用一次 `communicate()` 排空管道并回收进程，最后重新抛出超时。回归 worker 会先写 PID、延迟写
完成 marker；验收要求超时后 marker 不存在且该 PID 已不再运行。

通常 worker 就是直接子进程：worker PID 等于 `Popen.pid`，worker PPID 等于测试进程 PID。Windows
虚拟环境还可能先运行 `Scripts\python.exe` 重定向器，此时 `Popen.pid` 属于启动器，真正 worker 的
PPID 必须等于这个启动器 PID。测试只接受这两种精确关系，而且仅在确认命令指向带 `pyvenv.cfg` 的
Windows 虚拟环境解释器时才允许重定向器拓扑；它不会把任意不一致的 PID 当作合法结果。

记录以下证据：

| 证据 | 解释 |
|---|---|
| child PID | OS 为本次进程分配的身份，不等于端口或任务 ID |
| parent PID | 谁启动了它；服务管理器、shell 和测试进程会形成不同进程树 |
| exit code 7 | 进程已结束且主动报告失败；不能只看“没有 traceback” |
| stdout/stderr | 两条独立通道；日志内容不能包含凭据或完整环境变量 |

反例是只运行 `python service_worker.py` 后看见一行输出，就声称服务生命周期正确。启动成功没有证明
它能接收终止、清理资源或返回可解释的非零退出码。

## 实验二：终止与清理

POSIX 上，父进程发送 `SIGTERM`，worker 的 handler 设置停止事件，主流程随后输出 cleanup 并以 0
退出。Windows 的 `os.kill(pid, SIGTERM)` 是无条件终止，不会调用同一个 Python handler；本
reference 在 Windows 子进程内部 `raise_signal(SIGTERM)`，只验证 handler 和 cleanup 语义，不伪装成
POSIX 外部投递。容器部署的 Linux `SIGTERM` 必须在 Linux/容器门禁中另行复验。

正确顺序是：

```text
ready -> termination requested -> stop accepting work -> release resources -> exit
```

反例是在 handler 里执行长时间阻塞 I/O 或直接 `kill -9`。前者可能超过优雅终止预算，后者根本不给
应用清理机会。handler 应只设置停止状态，把可失败的清理留给主控制流。

等待 worker 输出 `ready` 也必须受同一个总 deadline 约束，不能对 `readline()` 做无限阻塞，再在
ready 之后重新获得一整份 timeout。silent worker 与超过预算才 ready 的 worker 都必须有界失败、
被终止回收，且不能留下延迟副作用。

## 实验三：权限失败先于写入

`ensure_writable()` 对现存目标检查自身，对尚不存在的目标检查父目录。测试先注入拒绝结果，断言
`PermissionError`，再注入允许结果证明恢复。这个注入测试证明控制流，不证明本机 ACL 已正确配置。

真实练习另建临时目录，不要改仓库或系统目录权限：

1. 查看目标及父目录权限；Windows 使用 `Get-Acl`，Linux 使用 `ls -ld`。
2. 只移除临时目标的写权限并记录一次失败。
3. 恢复原权限，再运行同一命令并成功。
4. 删除临时产物，确认没有以管理员身份长期绕过权限边界。

## 实验四：四层请求路径

`trace_request_path()` 固定执行 `resolve -> connect -> handshake -> exchange_http`。每个测试只让一层
失败，后续层不得执行：

| 故障 | 可见证据 | 不应下的结论 |
|---|---|---|
| DNS | `gaierror`，没有 TCP 观察 | “服务返回了 500” |
| TCP | 已解析地址，连接拒绝或超时 | “证书坏了” |
| TLS | TCP 已建立，握手/证书失败 | “HTTP 路由不存在” |
| HTTP | 前三层均成功，有合法 4xx/5xx 状态 | “网络不可达” |

注入式阶段让实验离线且确定，但不包含真实解析器缓存、代理、证书链、SNI、重传或抓包。真实环境需要
结合 `Resolve-DnsName`/`nslookup`、`Test-NetConnection`/`ss`、`openssl s_client` 或平台等价工具，
并保存命令、时间、目标和原始错误；不能只保存一张“访问失败”截图。
