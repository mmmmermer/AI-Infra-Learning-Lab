# 知识卡片编写模板

## 使用定位

知识卡片不是学习模块的替代品，也不是第一次学习的入口。

正确顺序是：

```text
先读 10_学习模块里的学习地图和适配教材
-> 做对应实验或项目练习
-> 再把真正理解过、踩过坑、用过的知识点沉淀成卡片
```

如果一张卡片只有“一句话定义 + 链接”，它只适合做复习索引，不适合帮助学习。重要知识点应该写成“小讲义卡片”。

## 小讲义卡片结构

````markdown
# 知识点名称

## 一句话解释

用自己的话解释这个概念，不追求术语漂亮，先追求能听懂。

## 为什么要学

说明它解决什么问题，以及为什么会在当前路线中反复出现。

## 放在系统里的位置

用一段流程说明它在 P01/P03/RQ01 或某个实验中的位置。

```text
输入
-> 这个知识点发挥作用的环节
-> 输出
```

## 直觉解释

用一个生活类比或工程场景解释它，不要只给定义。

## 最小例子

写一个最小代码、表格、字段或流程。

## 常见误区

- 误区 1：
- 误区 2：
- 误区 3：

## 和其他概念的关系

| 相关概念 | 关系 |
|---|---|
|  |  |

## 对应教材、实验和项目

- 教材：
- 实验：
- 项目：

## 自测问题

- 我能不能不用术语解释它？
- 我能不能说出它解决什么工程问题？
- 我能不能写出一个最小例子？
- 我能不能说出一个常见错误？
````

## 不同类型卡片怎么写

### 概念卡片

重点写清：

- 它是什么。
- 它不是什么。
- 为什么项目里需要它。
- 初学者最容易误解哪里。

适合例子：`RAG`、`Agent`、`queue`、`worker`、`baseline`。

### 技术机制卡片

重点写清：

- 输入是什么。
- 中间发生了什么。
- 输出是什么。
- 哪些地方会失败。

适合例子：`embedding 检索`、`任务状态流转`、`Docker network`、`KV cache`。

### 指标卡片

重点写清：

- 指标定义。
- 怎么计算。
- 什么时候会误判。
- 在实验报告里怎么解释。

适合例子：`P95/P99`、`吞吐`、`错误率`、`worker utilization`。

### 问题与坑卡片

重点写清：

- 现象。
- 原因。
- 排查步骤。
- 修复方法。
- 如何下次避免。

适合例子：`pytest 找不到模块`、`Docker 容器连不上 db`、`虚拟环境混乱`。

## 质量标准

一张重要卡片写完后，至少应该满足：

- [ ] 不是照抄资料定义。
- [ ] 有自己的解释。
- [ ] 有最小例子。
- [ ] 有常见误区。
- [ ] 能连接到教材、实验或项目。
- [ ] 能帮助未来复习或面试解释。
