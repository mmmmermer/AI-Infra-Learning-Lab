## baseline

| 策略 | 角色 | 可部署性 |
|---|---|---|
| FIFO | 最小基线 | 是 |
| Priority + aging | 业务优先级和等待保护基线 | 是，需固定规则 |
| predicted SJF | 使用提交时可用估计值 | 是，取决于预测器 |
| oracle SJF | 使用真实服务时间 | 否，只作上界/诊断 |
| cost-aware | 多因素启发式 | 是，但权重必须预注册 |
| Random | 实现 sanity check | 不作为主要竞争 baseline |

所有结果中把历史别名 `sjf` 展示为 `predicted_sjf`。

## 因子矩阵

| 因子 | 第一轮水平 |
|---|---|
| load | 低、中、高；用目标 utilization 或到达率定义 |
| service distribution | 轻尾、lognormal、有限均值的重尾 |
| arrival pattern | 平稳 Poisson、on/off burst |
| prediction error | 0、低、中、高；用显式乘性误差模型 |
| task mix | short/medium/long 的固定比例 |
| workers | 1、2、4 |
| strategy | FIFO、priority+aging、predicted SJF、oracle SJF、cost-aware |

第一轮不要全因子爆炸。先固定 workers=1 和三档 prediction error，验证实现后再扩展。

## workload 生成

每个 seed 生成一次 workload manifest：

```text
task_id
task_class
submit_time
actual_duration
estimated_duration
priority
token_count_or_cost_proxy
seed
generator_version
```

预测误差示例：

```text
estimated_duration = actual_duration * multiplicative_error
```

误差分布、偏差和截断范围必须预注册。不能为了让某策略获胜而按结果调参数。

## common random numbers

同一个 seed 生成的任务表必须原样复制给所有策略：

```text
seed 17 workload -> FIFO
                 -> predicted SJF
                 -> oracle SJF
                 -> priority + aging
```

这样每个策略差异是配对差异，不被另一组随机任务噪声污染。策略执行不得修改共享输入对象；每次运行使用深拷贝或重新反序列化。

## 到达模型与 coordinated omission

模拟器使用开放环 submit time，任务到达不应因为系统变慢而自动停止。P03 压测也要使用固定到达计划或支持 open-loop 的工具，避免只在前一个请求完成后才发送下一个请求，从而低估排队和尾延迟。

## 指标

### 主要指标

- overall P95/P99 queue wait
- per-class P95/P99 queue wait
- p95/p99 slowdown
- SLO violation rate

### 次要指标

- average/max queue wait
- turnaround
- throughput
- worker utilization
- max/area-under-curve ready queue length
- per-class completed share 和 service share

### 预测器指标

- MAE / median absolute error
- multiplicative error distribution
- rank correlation between estimate and actual duration
- underestimation/overestimation by task class

没有预测器指标，就无法解释 predicted SJF 的结果。

## 百分位与样本单位

- 单次 run 内 percentile 固定为 nearest-rank，或在实现前统一改成另一种明示方法。
- 统计推断的独立样本单位是 seed/run，不是同一 run 内的每个 task。
- 小样本 P99 只作 smoke check；正式实验需要足够任务量和多 seed。
- warmup、measurement 和 cooldown 必须分段，warmup 不进入主要统计。

## 重复与停止规则

1. 先用 30 个固定 seeds 做 pilot，只检查方差、失败率和 CI 精度。
2. 在看策略显著性结论前，按 CI 半宽目标预注册最终 seed 数，最大 100。
3. 不因为结果“刚好显著”提前停止。
4. 任何 seed 排除必须基于预注册的系统故障规则，并保留原始记录。

## 统计分析

对每个条件和每个 seed，计算 `strategy - FIFO` 的配对差异：

- 报告平均配对差异、配对中位数和相对变化率。
- 对 seed 级配对差异做 95% bootstrap CI，并记录 bootstrap seed 和方法。
- 分布明显偏斜时，以 median difference 和 percentile CI 为主。
- 若做显著性检验，仅作为次要证据，并对多策略/多指标使用 Holm 校正。
- 同时报告实际效应大小，不能只报告 p-value。

## 公平性与伤害检查

每个总体 improvement 都必须同时给出：

- short/medium/long 分组结果。
- max wait 和 p99 slowdown。
- 超过等待阈值的比例。
- 最差任务类别相对 FIFO 的变化。
- aging 开启/关闭对比。

如果整体平均改善但长任务伤害明显，结论必须写成 tradeoff，而不是“策略更优”。

## 可重放 artifact

每次 run 保存：

```text
config.json
workload.csv
per_task_results.csv
summary.json
environment.json
git_commit.txt
stdout.log
errors.csv
```

`environment.json` 至少含 Python、依赖、OS、CPU、worker 数、代码版本和 percentile 方法。P03 阶段再增加 Docker image digest、服务配置和硬件信息。
