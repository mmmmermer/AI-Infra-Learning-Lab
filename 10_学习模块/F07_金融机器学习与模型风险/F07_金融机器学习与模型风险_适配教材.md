## 本模块最值得理解的难点

F07 的难点不是模型不够复杂，而是金融 ML 很容易“看起来准确，但其实错了”。

| 难点 | 为什么难 | 工程痛点 | 后续影响 |
|---|---|---|---|
| label 很容易设计错 | 未来收益、方向、风险事件含义不同 | label 和 feature 时间错位会泄漏 | 评测结果失真 |
| NaN 可能被静默改写 | rolling、lag 和未来标签天然产生边界缺失 | 填 0 或全表 dropna 都可能改变样本语义 | 标签、样本量和指标失真 |
| 时间序列不能随机打乱 | 未来样本混进训练集会污染实验 | 常规 ML 示例默认 shuffle | 模型看似泛化，实际不可用 |
| `gap=horizon` 不是防泄漏开关 | gap 按行计数，不理解标签时间区间、缺失交易日或 pipeline | 参数写了但信息窗口仍可能重叠 | final test 仍被污染 |
| baseline 容易被忽略 | 复杂模型不一定超过简单规则 | 没有 baseline 就无法判断价值 | 面试和报告缺可信对照 |
| 指标可能误导 | 类别不平衡时 accuracy 很虚 | 只看一个指标容易误判 | 需要 confusion matrix 和失败样例 |
| 阈值也会过拟合 | 在 test 上挑 0.4/0.5/0.6 等于反复看答案 | final test 不再是未见数据 | 指标虚高且无法复核 |
| 标准化也会泄漏 | 用全样本 fit scaler 等于看了未来 | pipeline 写错不一定报错 | 结果虚高 |
| 样本外失效常见 | 市场环境会变化 | 一段时间有效不代表长期有效 | 必须写 limitations |

学习 F07 的核心习惯是：

```text
模型结果必须和 label 定义、feature 时间口径、切分方式、baseline、失败样例、限制说明一起出现。
```

## 一条贯通学习线

F07 不按“模型列表”来学，而按“一个金融预测实验如何被证明没有乱来”来学：

```text
问题定义
-> label
-> feature
-> dataset cleaning / NaN boundary
-> time-based split
-> baseline
-> challenger model
-> validation-only threshold
-> evaluation
-> stability / calibration
-> leakage_check
-> monitoring contract
-> model_card
-> P03 model_risk_task
```

这条线的重点是：金融 ML 的核心不是模型复杂度，而是实验可信度。只有 label、feature、时间切分、baseline、失败样例和模型卡都能讲清楚，模型指标才有学习价值，也才可能进入后续项目表达。

## 可迁移的原则

1. 时间切分优先于随机切分。金融 ML 的样本有时间顺序，随机打乱常常会把未来信息泄漏到训练集。
2. baseline 是模型价值的底线。复杂模型如果不能超过简单规则或朴素预测，就不能被包装成有效模型。
3. 模型风险要写进输出。数据泄漏、样本外失效、类别不平衡、概念漂移和不可解释性，都必须进入模型卡和项目表达边界。
4. `gap`、Pipeline 或库函数只能执行你写出的规则，不能替你定义信息可见时间。每个 label 和 feature 都要能回答“用了哪个时间区间的信息”。
5. validation 用于选择，final test 只用于冻结后的最终评估。看过 final test 再改特征、阈值或参数，旧 final test 就已经参与了开发。

## 踩坑现场

> 你把全样本数据先标准化，再切 train/test，测试集效果明显提升。问题是 scaler 已经看过未来测试数据。正确做法是只在训练集 `fit`，再对验证集和测试集 `transform`，并在 GF07 实验记录里写明 pipeline 顺序。

## 第 1 章：金融 ML 不是先上模型

### 1.1 本章解决什么问题

普通机器学习教程常常从模型开始：逻辑回归、随机森林、神经网络。但金融 ML 第一优先级不是模型，而是问题定义：

```text
我要预测什么？
预测发生在什么时候？
当时能看到哪些数据？
用什么 baseline 比较？
怎么证明没有泄漏？
```

如果这些问题不清楚，模型越复杂，越容易制造漂亮但无效的结果。

### 1.2 金融直觉

假设你想预测“明天收益率是否为正”。这不是一句口号，而是一组严格定义：

```text
label_t = 1 if return_{t+1} > 0 else 0
features_t = only information available at time t
prediction_time = end of day t
evaluation_time = day t+1
```

如果特征里不小心包含了 `return_{t+1}` 或用未来全样本统计量处理了数据，模型可能看起来很准，但本质上已经作弊。

### 1.3 工程场景

F07 未来可以连接 P03：

```text
model_risk_task:
  input_json:
    label_definition
    feature_list
    split_rule
    baseline_rule
    model_config
  result_json:
    metrics
    confusion_matrix
    leakage_check
    failure_examples
    limitations
```

### 1.4 常见错误

- 一上来就训练复杂模型。
- 不写 label 定义。
- 随机切分金融时间序列。
- 没有 baseline。
- 只报告最好的模型结果。
- 把分类准确率写成投资能力。

## 第 2 章：label、feature 和时间口径

### 2.1 本章解决什么问题

F07 的第一个关键是把 label 和 feature 分清楚。

| 对象 | 含义 |
|---|---|
| feature | 模型输入，只能使用预测时可见的信息 |
| label | 模型要预测的目标，通常来自未来一段时间 |
| prediction_time | 做出预测的时间 |
| horizon | 预测多远之后的结果 |

### 2.2 最小 label 设计

用明日收益方向作为教学 label：

```python
df["return"] = df["adjusted_close"].pct_change(fill_method=None)
df["future_return"] = df["return"].shift(-1)
df = df.dropna(subset=["future_return"]).copy()
df["label"] = (df["future_return"] > 0).astype("int8")
```

注意：必须先删除未来收益为空的行，再生成标签；否则 NaN 比较结果会被转成 0，制造假标签。`future_return` 只能用于构造 label，不能作为 feature。

这里还必须写清预测发生在什么时候。本教材的最小例子假定：在交易日 `t` 收盘后形成特征并做预测，因此 `adjusted_close_t` 已经可见；标签使用下一交易日收益，信息区间是：

```text
feature information: <= close(t)
label information:   (close(t), close(t+1)]
```

如果预测发生在 `t` 日开盘前，`adjusted_close_t` 尚不可见，特征必须再向后移动。`shift(1)` 是否足够，取决于 prediction time，而不是取决于代码是否运行。

多期标签不能把“第 h 期的单期收益”误写成“未来 h 期累计收益”。以未来 `horizon` 个交易观测为例：

```python
horizon = 5
df["future_return_h5"] = (
    df["adjusted_close"].shift(-horizon) / df["adjusted_close"] - 1
)
df["label_end_time"] = df.index.to_series().shift(-horizon)
```

此时 `horizon=5` 表示接下来 5 个数据行/交易观测，不是 5 个自然日。周末、节假日和停牌会让日历跨度变化；若数据还缺了本应存在的交易日，标签含义会继续漂移。因此必须记录：

```text
prediction_time
label_start_time
label_end_time
horizon_value
horizon_unit: trading_observations / calendar_days
trading_calendar_or_missing_session_note
```

当 `horizon > 1` 时，相邻样本的标签区间会重叠。例如 `t` 的 5 日标签和 `t+1` 的 5 日标签共享大部分未来价格。这不会自动让标签“错误”，但会让相邻样本相关，也会影响切分边界和不确定性解释。

### 2.3 最小 feature 设计

只使用历史可见特征：

```python
df["return_lag_1"] = df["return"].shift(1)
df["return_lag_2"] = df["return"].shift(2)
df["rolling_vol_3"] = df["return"].rolling(3).std().shift(1)
```

这里 `shift(1)` 的意思是：今天预测时，只使用昨天及以前可见的信息。

### 2.4 可执行 dataset 清洗和 NaN 边界

lag、rolling 和未来标签天然会产生缺失值：

```text
lag / rolling -> 数据开头缺失
future label  -> 数据结尾缺失
原始字段      -> 中间可能有真正的数据缺失
```

错误版：

```python
df = df.dropna()  # 无关字段缺失也会删行，样本可能被静默改变
df[features] = df[features].fillna(0)  # 把“历史不足”伪装成真实的 0
df["label"] = (df["future_return"] > 0).astype("int8")  # 末尾 NaN 会变成 0
```

正确的第一轮做法是：先定义模型真正需要的列，只对这些列处理 NaN/inf；未来收益为空的行先删除，再创建标签；然后用断言验证时间顺序、唯一索引和必需字段。

下面是可从仓库根目录独立运行的最小例子，只依赖 F02/GF finance reference 已使用的 pandas/numpy：

```python
import numpy as np
import pandas as pd


raw = pd.DataFrame(
    {
        "date": [
            "2024-01-02", "2024-01-03", "2024-01-04", "2024-01-05",
            "2024-01-08", "2024-01-09", "2024-01-10", "2024-01-11",
            "2024-01-12", "2024-01-16", "2024-01-17", "2024-01-18",
        ],
        "adjusted_close": [100, 101, 99, 102, 103, 101, 100, 106, 99, 108, 98, 110],
        "optional_note": [None] * 12,
    }
)


def build_model_dataset(raw: pd.DataFrame, horizon: int = 2):
    required_columns = {"date", "adjusted_close"}
    missing_columns = required_columns - set(raw.columns)
    if missing_columns:
        raise ValueError(f"missing columns: {sorted(missing_columns)}")
    if horizon < 1:
        raise ValueError("horizon must be at least 1")

    work = raw.copy()
    work["date"] = pd.to_datetime(work["date"], errors="raise")
    work["adjusted_close"] = pd.to_numeric(work["adjusted_close"], errors="coerce")
    work = work.sort_values("date")
    if work["date"].duplicated().any():
        raise ValueError("duplicate prediction_time")
    if (
        work["adjusted_close"].isna().any()
        or not np.isfinite(work["adjusted_close"]).all()
        or (work["adjusted_close"] <= 0).any()
    ):
        raise ValueError("adjusted_close must be finite and positive")
    work = work.set_index("date")

    returns = work["adjusted_close"].pct_change(fill_method=None)
    work["return_lag_1"] = returns.shift(1)
    work["return_lag_2"] = returns.shift(2)
    work["rolling_vol_3"] = returns.rolling(3).std().shift(1)
    work["future_return"] = (
        work["adjusted_close"].shift(-horizon) / work["adjusted_close"] - 1
    )
    work["label_end_time"] = work.index.to_series().shift(-horizon)

    features = ["return_lag_1", "return_lag_2", "rolling_vol_3"]
    required_for_model = [*features, "future_return", "label_end_time"]
    dataset = (
        work.replace([np.inf, -np.inf], np.nan)
        .dropna(subset=required_for_model)
        .copy()
    )
    dataset["label"] = (dataset["future_return"] > 0).astype("int8")

    assert dataset.index.is_monotonic_increasing
    assert dataset.index.is_unique
    assert not dataset[features + ["label"]].isna().any().any()
    assert "future_return" not in features
    return dataset, features


dataset, features = build_model_dataset(raw, horizon=2)
print(f"rows={len(dataset)}")
print(f"range={dataset.index.min().date()}..{dataset.index.max().date()}")
print(f"label_counts={dataset['label'].value_counts().sort_index().to_dict()}")

assert len(dataset) == 6
assert dataset["label"].tolist() == [0, 1, 0, 1, 0, 1]
```

预期输出：

```text
rows=6
range=2024-01-08..2024-01-16
label_counts={0: 3, 1: 3}
```

`optional_note` 全部为空，但它不是模型必需字段，所以不能用全表 `dropna()` 删除所有样本。清洗规则必须和 feature contract 绑定，而不是“看到 NaN 就全部删除或填 0”。

### 2.5 工程痛点

最常见的错误是把“未来信息”藏在特征工程里：

- 使用 `future_return` 当 feature。
- rolling 特征没有 shift。
- 对所有 NaN 统一填 0，伪造本不存在的历史窗口或未来标签。
- 对整张表 `dropna()`，让无关字段改变训练样本。
- 用全样本均值、标准差做标准化。
- 用全样本排序/分位数生成特征。
- 先全样本选特征，再切分训练测试。

## 第 3 章：时间切分和 TimeSeriesSplit

### 3.1 本章解决什么问题

金融数据有时间顺序。训练集必须早于测试集，不能像普通表格任务那样随便随机打乱。

错误做法：

```text
random shuffle -> train/test
```

更合适的第一轮做法：

```text
early period -> train
prediction-horizon gap
middle period -> validation
prediction-horizon gap
latest period -> final test
```

### 3.2 最小切分

```python
horizon = 2
# model_df 由上一节同一个 build_model_dataset 处理完整 GF07 数据得到。
# 6 行 fixture 只验收清洗逻辑，不足以再切 train/validation/final test。
if len(model_df) < 30:
    raise ValueError("need a longer time series for three-way split with gaps")

train_end = int(len(model_df) * 0.6)
validation_end = int(len(model_df) * 0.8)

train = model_df.iloc[:train_end].copy()
validation = model_df.iloc[train_end + horizon : validation_end].copy()
test = model_df.iloc[validation_end + horizon :].copy()
```

validation 用于选择特征、阈值和参数；final test 在方案冻结后只使用一次。样本不足时应扩大数据或停止评测，不得删除 gap 来凑样本。

切分后不能只检查索引位置，还要检查训练标签的信息结束时间没有进入后续集合：

```python
def assert_no_label_overlap(train_part, evaluation_part) -> None:
    if train_part.empty or evaluation_part.empty:
        raise ValueError("split is too small")
    if train_part["label_end_time"].max() >= evaluation_part.index.min():
        raise ValueError("training labels overlap evaluation prediction times")


assert_no_label_overlap(train, validation)
assert_no_label_overlap(validation, test)
```

这里第二个断言把 validation 看成已经冻结的模型选择阶段。若 validation 的标签区间延伸进 final test，阈值和参数选择仍然间接看到了 final test 时间段。

### 3.3 `gap=horizon` 为什么不是自动防泄漏

`TimeSeriesSplit(gap=horizon)` 的 `gap` 只表示：在训练索引尾部和验证索引开头之间排除多少行。它不知道这些行代表交易观测、自然日、分钟 bar 还是多个资产。

第一轮至少检查四件事：

| 问题 | `gap=horizon` 会自动解决吗 | 需要额外做什么 |
|---|---|---|
| 训练 label 的未来区间进入 validation | 仅当 horizon 确实按等价的数据行定义时可能解决 | 保存 `label_end_time` 并做时间断言 |
| 节假日、停牌或缺失交易日 | 不会；gap 只数行 | 校验交易日历，记录缺失 session 和 horizon unit |
| 全样本 scaler、选特征或挑阈值 | 不会 | 只在 train/validation 阶段 fit/选择 |
| 多资产同一日期被拆到两侧 | 不会 | 先按唯一 prediction date 切分，再映射回各资产行 |

例如，“未来 7 个自然日收益”不能直接使用 `gap=7` 行；7 行可能跨越两周，也可能因为缺失数据跨得更久。反过来，“未来 5 个有效交易观测”可以按行定义，但必须先确认数据没有意外缺少交易 session。

重叠标签还意味着 validation 内相邻样本并不独立。`gap` 保护的是集合边界，不会让集合内部样本突然独立，也不会替你计算正确的置信区间。

### 3.4 TimeSeriesSplit 的直觉

`TimeSeriesSplit` 用于做多个按时间推进的训练/验证切分：

```text
train: early window 1 -> test: next window
train: early window 1+2 -> test: next window
train: early window 1+2+3 -> test: next window
```

第一轮只理解思想，不需要复杂调参。

### 3.5 最小代码

```python
from sklearn.model_selection import TimeSeriesSplit

tscv = TimeSeriesSplit(n_splits=3, gap=horizon)

for train_idx, validation_idx in tscv.split(model_df):
    train_fold = model_df.iloc[train_idx].copy()
    validation_fold = model_df.iloc[validation_idx].copy()
    assert_no_label_overlap(train_fold, validation_fold)
    print(
        train_fold.index.min(),
        train_fold["label_end_time"].max(),
        validation_fold.index.min(),
        validation_fold.index.max(),
    )
```

`TimeSeriesSplit` 用于开发期的多个 train/validation folds，不替代最后一段从未参与选择的 final test。官方文档还要求样本等间隔，才能让不同 fold 的指标覆盖可比较的时间长度；金融“日频表”只有在交易 session 完整、每行语义一致时，才可以近似理解为等间隔交易观测。

### 3.6 工程痛点

随机切分可能让未来样本进入训练集。模型看到未来市场环境后，再去预测过去或中间样本，评测就会虚高。

另一个隐蔽错误是看到代码里已有 `gap=horizon` 就停止检查。gap 值可能和 label 的真正信息区间不一致，或者全样本标准化、阈值选择和特征筛选仍然在泄漏。

切分记录必须包含：

```text
split_rule
train_date_range
validation_date_range
test_date_range
horizon_gap
horizon_unit
label_end_time_rule
label_overlap_assertion
n_splits
shuffle: false
final_test_frozen: true
```

- [ ] 能说明 TimeSeriesSplit 的作用。

## 第 4 章：baseline 先于复杂模型

### 4.1 本章解决什么问题

如果没有 baseline，你不知道复杂模型到底有没有价值。

第一轮 baseline 可以很简单：

- 永远预测多数类。
- 预测明天方向等于今天方向。
- 使用简单逻辑回归作为基础模型。

### 4.2 金融直觉

如果市场多数时候上涨，一个总是预测上涨的模型可能有不错 accuracy。复杂模型必须超过这种简单规则，并且解释它在哪里失败。

### 4.3 最小 baseline

```python
majority_class = train["label"].mode()[0]
validation["baseline_pred"] = majority_class
```

或者：

```python
validation["baseline_pred"] = (validation["return_lag_1"] > 0).astype("int8")
```

### 4.4 challenger model

第一轮 challenger 可以用逻辑回归：

```python
from sklearn.linear_model import LogisticRegression

features = ["return_lag_1", "return_lag_2", "rolling_vol_3"]

model = LogisticRegression()
model.fit(train[features], train["label"])
validation["model_score"] = model.predict_proba(validation[features])[:, 1]
```

### 4.5 阈值只能在 validation 选择

分类模型通常先输出 score/probability，再用阈值转成类别：

```text
model_score >= threshold -> predict 1
model_score < threshold  -> predict 0
```

`0.5` 不是永远正确的业务阈值；但尝试 `0.4/0.5/0.6` 后选 final test 上 F1 最高的一项，同样属于 selection leakage。

错误版：

```python
for threshold in candidate_thresholds:
    test_pred = (test_score >= threshold).astype("int8")
    # 错误：用 final test 标签选出最好的 threshold
```

修正版流程：

```text
train: fit model / scaler
validation: choose features, parameters and threshold
freeze: feature contract + cleaning + model + threshold + metric code
final test: evaluate exactly once
```

下面用固定 score 演示 validation-only threshold selection，不依赖模型训练即可运行：

```python
def binary_f1(labels: list[int], predictions: list[int]) -> float:
    pairs = zip(labels, predictions, strict=True)
    counts = list(pairs)
    tp = sum(y == 1 and pred == 1 for y, pred in counts)
    fp = sum(y == 0 and pred == 1 for y, pred in counts)
    fn = sum(y == 1 and pred == 0 for y, pred in counts)
    denominator = 2 * tp + fp + fn
    return 0.0 if denominator == 0 else 2 * tp / denominator


def apply_threshold(scores: list[float], threshold: float) -> list[int]:
    return [int(score >= threshold) for score in scores]


validation_labels = [0, 0, 1, 1, 1, 0]
validation_scores = [0.10, 0.40, 0.45, 0.55, 0.65, 0.80]
candidate_thresholds = [0.40, 0.50, 0.60]

validation_f1 = {
    threshold: binary_f1(
        validation_labels,
        apply_threshold(validation_scores, threshold),
    )
    for threshold in candidate_thresholds
}
selected_threshold = max(candidate_thresholds, key=validation_f1.get)

# 从这一行开始 threshold 已冻结；不能根据 final test 结果重新选择。
test_scores = [0.30, 0.44, 0.60, 0.70]
test_predictions = apply_threshold(test_scores, selected_threshold)

print(validation_f1)
print(f"selected_threshold={selected_threshold:.2f}")
print(f"frozen_test_predictions={test_predictions}")

assert selected_threshold == 0.40
assert test_predictions == [0, 1, 1, 1]
```

预期输出：

```text
{0.4: 0.75, 0.5: 0.6666666666666666, 0.6: 0.4}
selected_threshold=0.40
frozen_test_predictions=[0, 1, 1, 1]
```

真实模型在冻结后才运行：

```python
test["baseline_pred"] = majority_class
test["model_score"] = model.predict_proba(test[features])[:, 1]
test["model_pred"] = (test["model_score"] >= selected_threshold).astype("int8")
```

如果看完 final test 后修改任何 feature、NaN policy、scaler、模型参数、阈值或指标实现，这段 final test 已经参与开发。旧 test 应降级为 validation，并另取更晚、未见过的时间段做新 final test。

### 4.6 工程痛点

模型比较必须同口径：

- 同一个 train/validation/final test。
- 同一个 label。
- 同一个冻结后的 final test。
- 同一组评估指标。
- 同样记录限制。

否则 baseline vs model 没有意义。

- [ ] 能写出一个 baseline。
- [ ] 能说明 challenger 是什么。
- [ ] 能只用 validation 选择 threshold，并通过 `selected_threshold=0.40` 的断言。

- [ ] 能确保 baseline 和 model 用同一冻结 final test 评估。

## 第 5 章：指标、混淆矩阵和失败样例

### 5.1 本章解决什么问题

F07 不要求你掌握所有评估指标，但必须知道单一 accuracy 经常不够。

第一轮记录：

- accuracy。
- precision。
- recall。
- F1。
- confusion matrix。
- failure examples。

### 5.2 最小代码

```python
from sklearn.metrics import accuracy_score, precision_score, recall_score, f1_score, confusion_matrix

y_true = test["label"]
y_pred = test["model_pred"]

metrics = {
    "accuracy": accuracy_score(y_true, y_pred),
    "precision": precision_score(y_true, y_pred, zero_division=0),
    "recall": recall_score(y_true, y_pred, zero_division=0),
    "f1": f1_score(y_true, y_pred, zero_division=0),
    "confusion_matrix": confusion_matrix(y_true, y_pred, labels=[0, 1]).tolist(),
}
```

固定 `labels=[0, 1]` 是结果契约的一部分。若某个时间段只出现一个类别，输出仍保持 `2 x 2`
形状；样本类别缺失应另作为诊断字段报告，不能靠矩阵形状变化暗示。

### 5.3 失败样例

```python
failure_examples = test[test["label"] != test["model_pred"]].head(5)
```

失败样例比一个总分更有学习价值。你要观察：

- 模型在哪些日期错了。
- 错误是否集中在波动较大的阶段。
- label 是否本身噪声很大。
- 特征是否太弱。

### 5.4 时间分段稳定性

一个 final test 总分可能把时间变化平均掉。第一轮至少把 final test 按事先定义的连续时间段拆开，例如 early/middle/late 或按季度，分别报告样本数、类别比例、F1、错误类型和 score 分布。

| segment | rows | positive_rate | baseline_f1 | model_f1 | failure_note |
|---|---:|---:|---:|---:|---|
| early | 实际值 | 实际值 | 实际值 | 实际值 | 主要错误 |
| middle | 实际值 | 实际值 | 实际值 | 实际值 | 主要错误 |
| late | 实际值 | 实际值 | 实际值 | 实际值 | 主要错误 |

时间段必须在看结果前固定，不能看到某段特别好以后才把它单独切出来。每段样本太少时只写“诊断性观察”，不要把波动解释成市场规律。

第一轮的稳定性检查只回答：指标是否集中依赖某一时间段、类别比例是否变化、失败是否成簇出现。它不证明模型未来稳定，更不构成收益预测。

### 5.5 概率校准的最小边界

`model_score=0.8` 不会自动表示“事件有 80% 概率发生”。校准关注的是：预测分数相近的一组样本，其真实正类比例是否与预测分数大致一致。

在冻结 final test 上可以做只读评估：

```python
import pandas as pd


test["score_bin"] = pd.cut(
    test["model_score"],
    bins=[0.0, 0.4, 0.6, 1.0],
    include_lowest=True,
)
calibration_table = (
    test.groupby("score_bin", observed=True)
    .agg(
        rows=("label", "size"),
        mean_score=("model_score", "mean"),
        observed_positive_rate=("label", "mean"),
    )
)
brier_score = ((test["model_score"] - test["label"]) ** 2).mean()
```

必须同时报告每个 bin 的 `rows`。样本很少时，校准表只用于发现明显问题。不要在 final test 上拟合 Platt scaling、isotonic regression 或重新划分 bin 后再宣称校准改善；那又会把 final test 变成开发集。

第一轮只做 reliability table、Brier score 和限制说明。复杂校准器、嵌套交叉验证和金融决策效用函数暂缓。

### 5.6 工程痛点

金融 ML 的指标容易被误解：

- accuracy 高不代表能赚钱。
- 分类方向正确不代表收益风险好。
- 小样本指标不稳定。
- 某个时间段表现好不代表长期稳定。
- score 排序能力好不代表概率已经校准。
- overall 指标稳定不代表每个时间段或任务子群都稳定。

- [ ] 至少保存 3 个失败样例。
- [ ] 能按预先固定的连续时间段报告 rows、类别比例和模型/baseline 指标。
- [ ] 能解释 `model_score=0.8` 为什么不自动等于 80% 真实概率。
- [ ] 能在不重新拟合 final test 的前提下生成 calibration table 和 Brier score。
- [ ] 能说明指标不是投资承诺。

## 第 6 章：数据泄漏和模型风险

### 6.1 本章解决什么问题

F07 的核心是模型风险。模型风险不是“模型效果差”这么简单，而是模型因为数据、假设、切分、指标或使用场景不当而产生误导。

### 6.2 常见泄漏

| 泄漏类型 | 例子 |
|---|---|
| look-ahead leakage | 用未来收益做 feature |
| target leakage | feature 直接或间接包含 label |
| scaling leakage | 在 train/test 切分前用全样本 fit scaler |
| selection leakage | 根据测试集结果反复调参 |
| feature leakage | rolling 特征没 shift |
| label-window leakage | 训练 label 的信息结束时间进入 validation/final test |
| cleaning leakage | 用全样本统计量填补缺失，或把未来标签 NaN 静默转成 0 |

### 6.3 正确使用 scaler 的最小原则

错误：

```python
scaler.fit(df[features])
```

更合理：

```python
scaler.fit(train[features])
train_scaled = scaler.transform(train[features])
validation_scaled = scaler.transform(validation[features])
test_scaled = scaler.transform(test[features])
```

更稳的工程方式是把 imputer、scaler 和 model 放进同一个 scikit-learn `Pipeline`，并只对 train 调用 `fit`。但 Pipeline 仍然不知道 label 的信息区间，也不会阻止你在 final test 上挑 threshold。

### 6.4 模型风险记录

```yaml
model_risk:
  look_ahead_check: {status: pass|fail|not_run, evidence: artifact-or-test-id}
  label_information_interval: {status: recorded|missing, evidence: dataset-card-field}
  label_overlap_assertion: {status: pass|fail|not_run, evidence: test-id}
  random_shuffle_used: actual_boolean
  scaler_fit_scope: train|full_sample|not_applicable
  nan_policy: {status: recorded|missing, evidence: dataset-card-field}
  threshold_selected_on: validation|other|not_run
  final_test_status: frozen_unused|used_once|contaminated|not_run
  baseline: {status: defined|missing, evidence: report-section}
  failure_examples: {status: recorded|missing, evidence: artifact-uri}
  temporal_stability: {status: checked|not_run, evidence: artifact-uri}
  calibration: {status: checked|not_run, evidence: artifact-uri}
  monitoring_contract: {status: defined|missing, evidence: model-card-section}
  limitations:
    - small sample
    - noisy labels
    - no trading execution
    - no investment advice
```

### 6.5 上线监控的第一轮边界

离线 final test 通过不代表模型上线后会一直保持同样表现。第一轮只设计监控 contract，不要求搭建生产模型平台：

| 监控层 | 第一轮字段 | 为什么需要 |
|---|---|---|
| 输入质量 | schema、required feature 缺失率、inf 比例、时间新鲜度、取值范围 | 先发现数据管道坏了 |
| 分布 | feature summary、model score 分布、predicted positive rate | 发现输入或输出明显漂移 |
| 延迟与失败 | runtime、timeout、error rate、data rows | 区分模型质量和服务故障 |
| 延迟标签 | actual label 到达率、按时间段 F1、confusion matrix、Brier score | 标签成熟后再检查真实表现和校准 |
| 版本 | data/feature/model/threshold/calibration version | 让变化能追溯 |

金融标签通常延迟到达。预测 `t+horizon` 收益方向时，在 `label_end_time` 到来前不能计算这条预测的准确率。每条记录至少保留：

```text
prediction_id
prediction_time
label_end_time
feature_version
model_version
threshold
model_score
predicted_label
actual_label: pending / 0 / 1
label_available_at
```

错误做法是只监控实时 accuracy，却把尚未成熟的标签当 0，或只统计已经快速到达的标签。修正版是先监控输入和 score，等 label window 完成后按 `prediction_id` 回填真实标签，再按固定时间段计算性能和校准。

分布变化只是排查信号，不自动等于模型失效；整体分布没变也不保证特定子群没有问题。报警阈值应从冻结的 validation/reference period 预先定义，不能每天看图后临时改规则。

第一轮不做自动重训练、复杂漂移检测、在线 A/B 交易策略或生产告警平台，只把字段、延迟标签流程、阈值来源和人工复核动作写进模型卡。
