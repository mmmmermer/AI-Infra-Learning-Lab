# M08 监控压测与可观测性学习地图

## 怎么读这个模块

把 M08 当成“系统慢在哪里，怎么用数据说清楚”的训练，不要当成 Prometheus/Grafana 工具清单。

阅读主线是：请求进来，产生日志，记录 metrics，压测制造负载，观察 average/P95/P99/吞吐/错误率/队列长度/worker utilization，最后写报告解释瓶颈。

第一轮不追求完整监控平台，只追求能把“感觉慢”变成一张可信的实验记录表。

## 在总路线中的位置

M08 负责把“系统感觉慢”变成可观测、可压测、可解释的数据证据。

它承接 M05/P01 的调度指标，把 average、P95/P99、queue length、worker utilization 从模拟实验迁移到 P03 的真实 API、RAG、任务队列和 worker 服务中。

主线是：

```text
M05 调度指标
-> P01 调度实验样板
-> P03 API / RAG / worker metrics
-> E08 压测实验
-> 性能报告和瓶颈解释
```

## 要解决的问题

- 系统慢在哪里：API、排队、worker、retrieval 还是 generation？
- 高峰负载下 P95/P99 怎么变化？
- 吞吐、错误率、队列长度和 worker 利用率怎么一起解释？
- M05 的 average/P95/P99/utilization 如何变成真实服务指标？
- 压测报告应该如何证明一个工程结论？

## 学习目标

- [ ] 能解释 logs、metrics、tracing 的区别。
- [ ] 能记录 request_id、task_id、status、error_type、duration_ms。
- [ ] 能设计 P03 最小 metrics：请求耗时、吞吐、错误率、队列长度、worker utilization。
- [ ] 能解释 P95/P99 为什么比平均值更能暴露尾部问题。
- [ ] 能用 Locust 或 k6 设计一轮 API/RAG/任务队列压测。
- [ ] 能把 M05 调度指标迁移到 P03 真实服务监控。
- [ ] 能写出压测报告和瓶颈判断。

## 核心内容

| 内容 | 学到什么程度 | 对应出口 |
|---|---|---|
| logging | 会写结构化日志和关键事件 | P03 排错 |
| metrics | 会设计 Counter、Gauge、Histogram | `/metrics` |
| tracing 基础 | 会拆一次 RAG 请求的阶段 | P03 链路分析 |
| P95/P99 | 会解释尾延迟 | E08-02 |
| 吞吐 / 错误率 | 会判断处理能力和稳定性 | E08 报告 |
| 队列长度 | 会判断任务是否堆积 | P03 queue |
| worker utilization | 会判断 worker 是否忙满 | M05/P03 |
| 压测设计 | 会固定变量、制造负载、记录结果 | E08-01 |
| 报告模板 | 会把指标写成工程结论 | README / 科研记录 |

## 推荐学习顺序

1. 读 [[M08_监控压测与可观测性_适配教材|M08 适配教材]] 第 1-3 章，理解日志、metrics 和 tracing 的最小边界。
2. 读第 4-5 章，把 average、P95/P99、吞吐、错误率、队列长度、worker utilization 讲清楚。
3. 对照 M05/P01，理解模拟调度指标如何迁移到 P03。
4. 完成 E08-01：用 Locust 压测 RAG/API 请求。
5. 完成 E08-02：记录 P95/P99、吞吐、错误率和队列长度。
6. 完成 E08-03：展示队列长度和 worker utilization。
7. 用教材报告模板写一份性能与瓶颈解释。

## 检查标准

- [ ] 能用压测工具生成负载。
- [ ] 能记录并解释 average、P95/P99。
- [ ] 能解释吞吐、错误率、队列长度和 worker utilization。
- [ ] 能通过指标判断一个瓶颈。
- [ ] 能说明 M08 如何连接 M05、P01 和 P03。
- [ ] 能输出一份压测与监控报告。
