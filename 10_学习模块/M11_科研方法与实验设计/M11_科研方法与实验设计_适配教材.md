## 可迁移的原则

1. 研究问题必须来自可观察的工程现象。RQ01 不是“研究 AI 调度”这种大口号，而是围绕 RAG/Agent 请求类型、调度策略、尾延迟和 worker 利用率提出可验证问题。
2. baseline 是科研训练的地基。FIFO、Priority、SJF 或 Random 这类简单策略不是低级内容，而是判断新策略是否有价值的参照。
3. 没有真实实验数据时，只能写假设和设计，不能写结论。P01/P03/M05/M08 的指标没有跑出来之前，导师材料和简历表达都必须保持“待验证”口径。
4. 请求数不等于独立重复数。同一 seed/run 内的任务共享同一 workload 和系统状态；RQ01 第一轮以 seed/run 为推断单位，先算 seed 级配对差值，再讨论效应和不确定性。

## 踩坑现场

> 你跑了一次实验，发现 SJF 的平均等待时间更低，于是想写“SJF 优于 FIFO”。这还不够。你需要检查 P95/P99、任务类型分布、worker utilization、重复运行稳定性和控制变量，否则这个结论可能只是某一组负载下的偶然现象。

## 第 1 章：从工程现象到研究问题

### 1.1 为什么要学

工程项目里经常出现这样的现象：

```text
高峰负载下，系统变慢了。
SJF 平均等待更低，但 P99 可能更差。
worker 增加后 P95 下降，但利用率也下降。
```

这些现象本身还不是研究问题。研究问题需要更清楚：

- 在什么场景下？
- 比较什么方法？
- 用什么指标判断？
- 控制哪些条件？
- 预期能解释什么机制？

### 1.2 RQ01 的问题形式

当前 RQ01 可以表述为：

```text
在 RAG/Agent 请求类型差异较大的混合负载下，不同调度策略如何影响 P95/P99 尾延迟、平均等待时间、吞吐和 worker 利用率？
```

这个问题比“研究 AI 调度”更好，因为它限定了：

| 维度 | RQ01 的限定 |
|---|---|
| 场景 | RAG/Agent 混合请求 |
| 方法 | FIFO / Priority / SJF / Cost-aware |
| 指标 | P95/P99、平均等待、吞吐、worker utilization |
| 系统 | P01 模拟调度器 -> P03 AI Workload Platform |
| 目标 | 解释尾延迟和资源利用率取舍 |

### 1.3 工程问题和研究问题的区别

| 工程问题 | 研究问题 |
|---|---|
| 怎么实现一个调度器？ | 哪种调度策略在何种负载下改善或恶化尾延迟？ |
| 怎么让 worker 更多？ | worker 数量变化如何影响 P95 和利用率的取舍？ |
| 怎么让系统更快？ | 瓶颈来自排队、执行时间、任务长度分布还是 worker 数量？ |
| 怎么接入 RAG/Agent？ | RAG/Agent 请求类型差异如何改变调度策略效果？ |

### 1.4 常见错误

第一个错误：问题太大。

```text
研究 AI Infra 调度优化
```

这太大。第一轮要收敛到 RQ01。

第二个错误：问题不可验证。

```text
让系统更智能
```

这没有明确指标。

第三个错误：结论先行。

```text
证明 SJF 比 FIFO 更好
```

这会诱导你忽略 P99 变差等反例。更好的说法是“比较 SJF 与 FIFO 在不同指标上的取舍”。

### 1.5 小练习

把下面工程现象改写成研究问题：

```text
高峰负载下，Priority 的 P99 比 FIFO 更高。
```

参考方向：

```text
在高低优先级任务混合到达的高峰负载下，Priority 调度是否会改善高优先级任务等待时间，同时恶化整体 P99 或低优先级任务等待？
```

### 1.6 本章检查标准

- [ ] 能区分工程问题和研究问题。
- [ ] 能把 RQ01 写成可验证问题。
- [ ] 能避免“证明某策略一定更好”的结论先行。
- [ ] 能说明 RQ01 和 P01/P03 的关系。

## 第 2 章：假设，先写可被验证或否定的判断

### 2.1 为什么要学

假设不是结论。假设是实验前的可检验判断。

例如：

```text
在高峰负载下，SJF 可能降低平均等待时间，但可能提升长任务的 P99 等待时间。
```

这句话可以被数据支持，也可以被数据否定。

### 2.2 RQ01 可以有哪些假设

第一轮可以写三个弱假设：

| 编号 | 假设 |
|---|---|
| H1 | 高峰负载下，调度策略差异比低负载下更明显。 |
| H2 | SJF / Cost-aware 可能降低平均等待时间，但可能牺牲部分长任务或高 token 任务的尾延迟。 |
| H3 | 增加 worker 数量可以降低 P95/P99，但可能降低 worker utilization。 |

注意：这些是假设，不是已经完成的科研结论。

### 2.3 假设必须对应指标

| 假设 | 需要指标 |
|---|---|
| 高峰负载下策略差异更明显 | average wait、P95、P99、queue length |
| SJF 降低平均等待但牺牲长任务 | average wait、P99、按 task_type 分组指标 |
| worker 增加降低 P95 但降低利用率 | worker count、P95、utilization、throughput |

### 2.4 常见错误

第一个错误：把假设写成口号。

```text
我们的系统效果更好。
```

第二个错误：没有反例空间。

```text
Cost-aware 一定优于 FIFO。
```

第三个错误：假设和指标脱节。

如果假设谈公平性，就要记录不同 task_type 或 priority group 的指标。

### 2.5 小练习

为 RQ01 写一个关于 Priority 的假设，要求同时包含：

- 适用场景。
- 可能收益。
- 可能代价。
- 需要指标。

### 2.6 本章检查标准

- [ ] 能写出可验证假设。
- [ ] 能区分假设和结论。
- [ ] 能为每个假设指定指标。
- [ ] 能保留假设被否定的可能性。

## 第 3 章：baseline，先有参照物再谈改进

### 3.1 为什么要学

没有 baseline，就不知道“改进”到底改了什么。

M05 中 FIFO 是最重要的第一版 baseline，因为它简单、稳定、容易解释。

RQ01 第一轮 baseline 可以是：

| baseline | 作用 |
|---|---|
| FIFO | 先到先服务，最容易解释 |
| Random | 随机选择，作为弱参照 |
| Priority | 业务重要性参照 |
| SJF | 平均等待优化参照 |

不是所有 baseline 都代表“差方法”。baseline 的作用是建立比较口径。

### 3.2 RQ01 的 baseline 设计

第一组实验建议：

```text
固定任务流
固定 worker 数
固定指标计算方式
比较 FIFO / Priority / SJF
```

第二组再加入：

```text
Cost-aware
worker 数量变化
task_type 分组分析
```

### 3.3 baseline 要公平

公平比较至少要求：

- 同一批任务。
- 同一 worker 数。
- 同一随机种子。
- 同一指标计算方式。
- 同一实验时长。
- 同一任务到达规则。

### 3.4 常见错误

第一个错误：拿不同负载下的结果比较策略。

这样结论不成立。

第二个错误：只比较自己喜欢的方法。

如果没有 FIFO 或 Random，读者不知道你的方法强在哪里。

第三个错误：baseline 没有解释。

报告里要说明为什么选 FIFO/Random/Priority/SJF。

### 3.5 小练习

为 RQ01 第一轮实验写 baseline 说明：

```text
选择 FIFO 作为主要 baseline，因为它按到达顺序执行，逻辑简单、可解释性强，适合衡量其他策略在平均等待、尾延迟和公平性上的收益与代价。
```

### 3.6 本章检查标准

- [ ] 能解释 baseline 的作用。
- [ ] 能说明为什么 FIFO 适合作为第一 baseline。
- [ ] 能列出公平比较的条件。
- [ ] 能避免只比较自己喜欢的方法。

## 第 4 章：变量，实验一次只改变少数东西

### 4.1 为什么要学

实验要能解释原因，就不能让太多东西同时变化。

RQ01 至少要区分：

- 自变量：你主动改变的东西。
- 因变量：你观察的结果。
- 控制变量：你保持不变的条件。

### 4.2 RQ01 变量表

| 类型 | 示例 |
|---|---|
| 自变量 | 调度策略、worker 数量、任务类型比例、并发量 |
| 因变量 | 平均等待、P95/P99、吞吐、队列长度、worker utilization、失败率 |
| 控制变量 | 任务生成规则、随机种子、执行速度模型、实验时长、指标口径 |

### 4.3 第一轮变量控制

第一轮建议简单：

```text
只改变调度策略
固定任务流
固定 worker 数
固定随机种子
固定指标
```

第二轮再改变 worker 数量。

第三轮再改变任务类型比例。

### 4.4 和 M08 的关系

M08 压测也强调实验条件：

- 并发数。
- 请求类型。
- worker 数量。
- 运行时长。
- 预热时间。
- 数据规模。

如果这些不记录，P95/P99 无法解释。

### 4.5 常见错误

第一个错误：策略、worker 数和任务分布一起变。

这样无法知道结果变化来自哪里。

第二个错误：随机种子不固定。

如果任务流每次不同，策略对比会混入噪声。

第三个错误：指标口径变化。

例如一组用 waiting_time，另一组用 turnaround_time，却放在同一张表里比较。

## 第 5 章：指标，平均值不能代表尾部体验

### 5.1 为什么要学

RQ01 的关键词是“尾延迟”。所以不能只看平均等待。

至少要记录：

- average waiting time。
- P95 waiting time。
- P99 waiting time。
- max waiting time。
- throughput。
- queue length。
- worker utilization。
- error rate。

如果接入 P03，还要记录：

- queue_wait_ms。
- task_runtime_ms。
- p95_total_latency。
- retrieval_ms。
- generation_ms。
- token_count。

### 5.2 指标和研究问题的关系

| 研究关心 | 指标 |
|---|---|
| 整体效率 | average wait、throughput |
| 最差体验 | P95/P99、max wait |
| 资源使用 | worker utilization |
| 是否堆积 | queue length |
| 稳定性 | error rate、timeout |
| 成本 | token_count、estimated_cost |
| 公平性 | 按 task_type / priority 分组指标 |

### 5.3 P95/P99 怎么写进报告

不要只写：

```text
SJF 更快。
```

要写：

```text
在高峰任务流下，SJF 的平均等待时间低于 FIFO，但 P99 或 max wait 可能更高。这说明 SJF 改善了部分短任务体验，但可能牺牲长任务尾部等待。该现象需要进一步通过 task_type 分组指标验证。
```

注意这里使用“可能”和“需要验证”，因为研究报告不能夸大。

### 5.4 常见错误

第一个错误：只报平均值。

平均值可能掩盖长尾。

第二个错误：不报 worker utilization。

延迟下降可能是资源增加导致的，不一定是策略更好。

第三个错误：没有分组指标。

整体变好可能掩盖某类任务变差。

### 5.5 小练习

读下面结果：

```text
FIFO average=48.71, P99=101.45
SJF average=26.04, P99=121.45
```

写两句话解释：

- SJF 改善了什么？
- SJF 可能牺牲了什么？

### 5.6 本章检查标准

- [ ] 能解释 average、P95/P99、max wait 的区别。
- [ ] 能说明 worker utilization 为什么必须报告。
- [ ] 能把指标和研究问题对应起来。
- [ ] 能避免“单指标胜利”。

## 第 5.5 章：Rubric 和人工评测，把主观判断变成可复核记录

### 5.5.1 为什么要学

P95/P99、queue length、worker utilization 能说明系统是否慢、是否堵、是否忙。

但 RAG/Agent 还有另一类问题：

```text
回答看起来很流畅，但有没有证据？
引用存在，但是否真的支撑结论？
Agent 调用了工具，但这一步是否合理？
金融数字看起来正确，但单位、时间、口径是否一致？
```

这些不能只靠延迟指标判断，需要 Rubric 和人工评测。

Rubric 不是“凭感觉打分”，而是把质量判断拆成稳定维度，让不同时间、不同样例、不同评审者尽量按同一规则判断。

### 5.5.2 Rubric 在 P03 里的位置

P03 v0.3.1 当前没有 `AgentTask`、`EvaluationTask` 或 `ReviewRecord` 实体。下面是 post-v0.3.1 /
vNext planned 的轻量质量复核链路，不应当作当前 reference 已实现能力：

```text
Task(task_type=rag_retrieval) output / planned AgentTask output
-> output
-> EvaluationTask
-> Rubric
-> ReviewRecord
-> quality metrics
```

Rubric 的作用是回答：

- 这个输出要评什么？
- 每个维度怎么判？
- 哪些错误必须记录？
- 什么情况下需要人工复核或重新生成？

### 5.5.3 RAG 输出质量 Rubric

第一版 RAG Rubric 可以只保留 5 个维度：

| 维度 | 问题 | 记录字段 |
|---|---|---|
| 引用存在 | 回答是否提供 sources？ | `has_citation` |
| 引用支撑 | sources 是否真的支持关键断言？ | `has_citation_support` |
| 无来源断言 | 回答里有多少 unsupported claims？ | `unsupported_claim_count` |
| 事实错误 | 是否存在明显事实错误或幻觉？ | `hallucination_type` |
| 边界说明 | 是否说明不确定性、限制或风险？ | `risk_note_included` |

一个简单评分规则：

| 分数 | 含义 |
|---|---|
| 5 | 关键断言都有来源支撑，无明显错误 |
| 4 | 大部分断言有来源支撑，存在轻微遗漏 |
| 3 | 有引用，但部分关键断言支撑不足 |
| 2 | 引用弱或不相关，存在明显 unsupported claims |
| 1 | 无引用或大量幻觉，不可接受 |

总分只能作为摘要，不能替代可观察锚点。第一轮至少为每个维度固定离散锚点，例如：

| 维度 | 0 分 | 1 分 | 2 分 |
|---|---|---|---|
| 引用支撑 | 无 locator、引用矛盾或不支持关键断言 | 只支持部分关键断言，或 locator 不完整 | 所有 material claims 都有可解析且直接支持的 locator |
| 事实正确 | 存在改变结论的事实错误 | 有局部错误，但不改变主要事实 | 在给定 reference 内未发现事实错误 |
| 边界说明 | 把推断写成事实或给出越界建议 | 有提示但遗漏关键限制 | 事实、推断、不确定性和使用边界均分开 |

`material claim`、`directly supports` 和“改变结论”必须在 Rubric 说明里给正反例。若引用矛盾、
捏造金融数字或给出被禁止的投资建议，应记录 critical failure，不能靠其他维度平均成高分。

### 5.5.4 Agent 过程质量 Rubric

Agent 不是只看最终回答，还要看过程。

第一版 Agent Rubric 可以记录：

| 维度 | 问题 | 记录字段 |
|---|---|---|
| 步骤完整性 | 是否按固定 workflow 执行？ | `missing_step_count` |
| 工具选择 | 是否调用了正确工具？ | `expected_tool` / `actual_tool` |
| 工具参数 | 参数是否完整、合法、必要？ | `tool_call_error` |
| 失败处理 | 工具失败后是否进入重试、降级或人工确认？ | `review_status` |
| 日志可复盘 | step_logs 是否足以复盘过程？ | `step_review_note` |

常见错误类型：

| error_type | 说明 |
|---|---|
| `wrong_tool` | 该检索时调用了生成，或该确认时直接结束 |
| `wrong_args` | 工具参数缺失、格式错误或范围不合理 |
| `missing_tool` | 应该调用工具但没有调用 |
| `unnecessary_tool` | 调用了和任务无关的工具 |
| `step_order_error` | 步骤顺序不合理 |

### 5.5.5 金融内容核验 Rubric

金融场景最容易出现的问题不是“回答不流畅”，而是数字和口径错。

第一版只检查五类：

| 维度 | 检查问题 | 常见错误 |
|---|---|---|
| 来源 | 是否来自指定公告、财报、研报或 filings？ | 来源缺失、引用错文档 |
| 时间 | 年份、季度、报告期是否一致？ | 把 2023 年和 2024 年混用 |
| 单位 | 元/万元/亿元、USD/CNY 是否一致？ | 单位放大或缩小 |
| 指标口径 | revenue、net income、operating cash flow 是否混淆？ | 利润和现金流混用 |
| 边界 | 是否避免投资建议和确定性结论？ | 输出荐股、预测股价 |

### 5.5.7 评审校准与一致性

一套 Rubric 写得很详细，也不代表不同评审者会自然得到相同结果。正式评测前先抽取覆盖正常、
边界和失败情况的 calibration set，由至少两名评审者独立、尽量盲化地评分：

```text
冻结 rubric_version 和 reference
-> 随机化样例顺序并隐藏策略名称
-> 两名评审者独立给 dimension_scores 与错误类型
-> 报告逐维度一致率和分歧表
-> 对有序评分补 weighted kappa 等一致性统计及样本数
-> 讨论分歧并形成 adjudication 记录
-> 修改锚点后重新校准，再冻结主评测版本
```

一致率高不证明评分正确，kappa 也会受类别极不平衡和小样本影响，因此必须同时保存 confusion
table、原始评分与分歧原因。主评测若只有一名评审者，至少对预先固定的审计子集做双评；若
中途修改 Rubric，旧结果必须按新版本重评，或与新结果分开报告，不能直接合并。

### 5.5.8 规则迭代怎么写

评测后不要只留下分数，还要记录规则是否需要改。

例如：

```text
问题：多个回答都有 sources，但 sources 只支持事实背景，不支持最终判断。
规则迭代：新增 has_citation_support 字段，区分“有引用”和“引用真正支撑关键断言”。
```

规则迭代记录可以包含：

| 字段 | 说明 |
|---|---|
| 问题现象 | 评测中反复出现什么问题 |
| 原规则盲区 | 原 Rubric 为什么覆盖不了 |
| 新规则 | 增加或修改的判断标准 |
| 影响范围 | 影响 RAG、Agent、金融还是代码质检 |
| 下一轮验证 | 用哪些样例检查规则是否有效 |

### 5.5.9 常见错误

第一个错误：把 Rubric 写成主观形容词。

```text
回答质量好 / 一般 / 差
```

这不可复核。要写成可观察维度：

```text
关键断言是否有来源支撑；是否存在 unsupported claims；是否有风险提示。
```

第二个错误：把模型自评当最终评测。

模型可以辅助评测，但第一轮必须保留人工复核和 reviewer_note。

第三个错误：把质量指标和性能指标混在一起。

P95 低不代表回答正确；rubric_score 高也不代表系统吞吐好。

第四个错误：没有记录规则变化。

如果规则改了，前后实验结果不能直接比较，必须说明版本。

第五个错误：只给最终总分，不保存逐维度评分和评审者分歧。

这样无法区分“模型质量波动”和“评审规则不稳定”，也无法复算一致性。

## 第 5.6 章：最小统计推断，从多 seed 到可复核区间

### 5.6.1 为什么要学

前面几章已经让你能够提出问题、固定 baseline、控制变量并计算 P95/P99。但下面这句话仍然不能由单次运行支持：

```text
predicted SJF 的 P99 比 FIFO 更差。
```

单次运行可能刚好抽到一组特别难或特别容易的任务。即使一次 run 内有 500 个任务，这 500 个任务也共享同一个随机 workload、同一个队列演化和同一组系统条件，不能自动变成 500 次独立重复。

第一轮最小统计推断只回答四个问题：

1. 什么才是独立重复的实验单位？
2. 如何让不同策略在同一随机 workload 上公平配对？
3. 策略差异有多大，区间有多宽？
4. 做了很多条件和指标比较后，哪些结论仍然只是探索性观察？

本章不要求你推导复杂统计理论，也不把置信区间变成“显著性比赛”。目标是让 RQ01 的实验结论可以复算、可以质疑、可以明确限定边界。

### 5.6.2 实验单位和伪重复

RQ01 的数据有三层：

```text
实验条件，例如 high prediction error
-> seed/run，例如 seed 17
-> 同一 run 内的 500 个 measurement tasks
```

第一轮把 `seed/run` 作为统计推断的实验单位：

| 层级 | 用途 | 能不能直接当独立重复 |
|---|---|---|
| task | 计算该 run 的等待分布、P95/P99 和分组指标 | 不能；同一 run 内共享 workload 和队列状态 |
| seed/run | 产生一个策略级 P95/P99 等摘要 | 可以作为第一轮推断单位，但 seeds 必须按预先固定的生成规则独立产生 |
| condition | 表示 prediction error、worker 数等实验条件 | 不能；它是要比较或分层报告的条件 |

一个典型反例是：

```text
30 seeds x 每个 seed 500 个任务 = 15,000 个独立样本
```

这叫伪重复。正确口径是：每个策略和实验条件有 30 个 seed 级重复；500 个 measurement tasks 用于估计每个 seed 内的等待分布。

真实 P03 服务实验的单位可能是独立启动的 run、固定回放批次或独立时间窗口，不能机械地把这里的 seed 口径照搬过去。无论选择什么单位，都要在看结果前写清楚，并说明单位之间为什么可以近似独立。

#### 固定 trace 与固定 window 回答不同问题

服务实验必须在看结果前选择 estimand，不能跑完后按有利结果切换：

| 设计 | 分母与结束条件 | 优点 | 主要风险 |
|---|---|---|---|
| fixed trace | 各策略回放同一组带计划到达时间的任务，直到全部到达终态 | 比较同一批任务的完整结果 | 各策略运行时长不同，不能直接把完成数/总时长当同窗吞吐 |
| fixed window | 各策略观察相同时间窗，窗口结束停止新到达 | 适合容量、backlog 和稳定窗口问题 | 窗口末尾未完成任务属于 right-censored，完成者子集会偏向短任务 |

固定 window 不能静默删除未完成任务、把其延迟填成 0，或只对已完成请求算 P99 后声称代表全部
到达任务。第一轮有两种诚实做法：把 fixed trace 作为主要完整延迟比较；或在固定 window 后
单独 drain，分别报告窗口内 offered/started/completed、窗口末 backlog、censored 数量/比例，
以及 cool-down 完成结果。需要直接估计删失延迟分布时再使用生存分析方法，不能手工补值。

### 5.6.3 多 seed、common random numbers 和配对设计

固定一个随机种子只能帮助复现同一次运行，不能证明结果对随机 workload 稳定。多 seed 的作用是重复抽取 workload，观察策略差异是否在不同运行中保持方向和量级。

RQ01 使用 common random numbers，简称 CRN：同一个 seed 生成的同一份 workload 原样交给所有策略。

```text
seed 17 workload -> FIFO
                 -> predicted SJF
                 -> oracle SJF

seed 18 workload -> FIFO
                 -> predicted SJF
                 -> oracle SJF
```

这样比较的是同一 workload 上的策略差异。对每个 seed 先计算：

```text
delta_seed = metric(strategy, seed) - metric(FIFO, seed)
```

对等待时间这类“越低越好”的指标：

| 配对差值 | 含义 |
|---:|---|
| `< 0` | 该策略相对 FIFO 降低了该指标 |
| `= 0` | 该 seed 下没有差异 |
| `> 0` | 该策略相对 FIFO 增加了该指标 |

CRN 的价值是减少“两个策略恰好抽到不同任务”带来的噪声。它不意味着每个 task 变成独立样本，也不能修复错误的 workload 模型、遗漏的 baseline 或不真实的系统假设。

常见错误是把所有 FIFO 结果放一组、所有 SJF 结果放另一组，却丢掉 seed 对应关系。这样会浪费配对信息，也更难解释差异来自策略还是 workload。

### 5.6.4 先报告效应量，再谈区间

效应量回答“差多少”，而不是只回答“方向是否稳定”。第一轮至少报告：

```text
mean paired delta
median paired delta
mean relative change
原指标单位
```

例如：

```text
mean paired delta = +52.15 time units
mean relative change = +51.43%
```

这表示在当前实验条件和 30 个 seeds 上，该策略相对 FIFO 的 P99 平均增加约 52.15 个模拟时间单位。它不是“未来所有 workload 都会增加 52.15”，也不是生产系统毫秒结论。

相对变化率必须和绝对差一起报告。baseline 很接近 0 时，相对变化会非常大或没有稳定含义；只报百分比容易夸大一个绝对值很小的变化。

协议必须预先定义零基线规则，例如 `abs(baseline) < epsilon` 时把 relative change 记为 null，
只报告原单位绝对差和两边原值。不能用任意小常数替换 0 后继续算百分比，也不能从汇总表中
删除这些 runs。

还要区分统计不确定性和工程意义：区间没有跨 0，不代表变化一定值得部署；区间跨 0，也不等于已经证明“完全没有影响”。工程判断还需要预先定义可接受的最小收益、最大伤害、SLO 和资源成本。

### 5.6.5 95% CI：配对 t 区间和 percentile bootstrap 的边界

第一轮把 95% confidence interval 理解为：如果按同一 protocol 反复抽取独立 runs，并每次用同一方法构造区间，长期来看约 95% 的区间会覆盖对应的真实参数。不要写成“真实效应有 95% 概率落在这一个已经算出的区间里”。

配对 t 区间用于 seed 级配对差值的均值：

```text
mean_delta +/- t_critical * stdev(deltas) / sqrt(seed_count)
```

它更适合：

- 研究目标是平均配对差值。
- seed 级差值近似独立。
- 差值分布没有极端偏斜或少数异常 run 完全支配结果。
- seed 数、排除规则和主要指标在看结果前已经固定。

percentile bootstrap 的第一轮做法是：从完整的 seed 级配对差值中有放回抽取同样数量的差值，重复 5,000 次，每次计算均值或中位数，再取 bootstrap 统计量的 2.5% 和 97.5% 分位数。

```text
deltas = [delta_seed_17, ..., delta_seed_46]
-> 按 seed 级差值重采样
-> 每次计算 mean(deltas_sample)
-> 取 bootstrap means 的 P2.5 / P97.5
```

它更适合做当前 RQ01 pilot 的可解释区间，因为 seed 级尾延迟差值可能偏斜。但 bootstrap 不是自动正确：

- 必须重采样完整的 seed 级配对差值，不能分别打乱两个策略，也不能把 task 行当独立样本。
- 很少的 seeds、强依赖、选择性排除和错误 workload 都不会被 bootstrap 修好。
- percentile bootstrap 对偏差和极端偏斜也可能表现不好；第一轮把它作为预先指定的简单方法，不宣称适用于所有分布。
- 必须保存方法、resample 次数和 bootstrap seed，保证区间可以复算。

RQ01 pilot 以 seed 级 paired percentile bootstrap 为主要区间；配对 t 区间只作为敏感性检查。若两种方法方向明显冲突，应先检查差值分布、异常 runs 和样本量，而不是挑选更喜欢的区间。

### 5.6.6 P95/P99 为什么需要足够的 run 内样本和多 seed

P95/P99 有两层不确定性：

1. 每个 run 内只有有限任务，算出的 P95/P99 本身会波动。
2. 不同 seeds 生成不同 workload，各 run 的 P95/P99 也会波动。

RQ01 pilot 每个 seed 有 500 个 measurement tasks。nearest-rank P99 大致落在排序后的第 495 个位置，也就是最慢的少数任务之一。若只看 long 类，单个 seed 可能只有约 75 个 long tasks，此时 long P99 常常非常接近该组最大值，更不稳定。

第一轮使用固定的两步口径：

```text
先在每个 seed 内，用同一种 percentile 方法计算该 run 的 P95/P99
-> 再对各 seed 的策略配对差值做区间估计
```

不要把 30 个 seeds 的所有 task 无条件拼成一个大列表再算 P99。池化会改变每个 run 的权重，掩盖 run 间波动，也会丢掉配对结构。

正式报告必须同时记录：

- 每个 seed 的 measurement task 数。
- 分任务类型的样本数。
- percentile 定义，例如 nearest-rank。
- seed 数和 seed 列表。
- run 失败和排除规则。

任务很少时，P99 只适合 smoke check。增加 task 数可以改善 run 内尾部分位数的分辨率；增加独立 seeds 才能改善对 run 间变化的认识，两者不能互相替代。

### 5.6.7 多重比较边界

RQ01 pilot 同时有三档 prediction error、多个策略和多个指标。如果看到哪一格好看就只报告哪一格，即使每个区间单独看似合理，也会增加偶然发现和选择性叙述风险。

第一轮按下面三层管理：

| 层级 | 做法 |
|---|---|
| primary | 在看结果前固定一个主要条件、baseline、策略和指标，并完整报告效应量与 CI |
| secondary | 预先列出少量公平性、伤害和资源指标，作为主要结论的必要补充 |
| exploratory | 其余切片明确标记为探索性，只生成下一轮假设，不包装成已确认结论 |

95% CI 并不会自动解决多重比较问题。如果后续对多策略、多条件和多指标做正式假设检验，应预先定义 comparison family，并使用 Holm 等校正方法；校正后的 p-value 仍然不能替代效应量、CI、失败分析和场景边界。

第一轮不要求手推 Holm 公式。你只需要做到：不事后挑指标、不隐藏反方向结果、把 primary/secondary/exploratory 写进 config 或实验设计，并在报告中保留完整比较表。

### 5.6.8 最小可复算例子

下面的代码只使用 Python 3.13 标准库，从仓库根目录运行。它读取 RQ01 的 `per_seed_summary.csv`，计算 high prediction error 条件下 `predicted_sjf - fifo` 的 P99 seed 级配对差值，并复算配对 t 区间和 percentile bootstrap CI。

> 这是对现有 AI 参考 artifact 的复算示例，不代表学习者已经完成独立实验。学习者自己的完成证据必须来自亲手运行、保存的新 artifact 和复盘记录。

```python
from csv import DictReader
from hashlib import sha256
from math import ceil, sqrt
from pathlib import Path
from random import Random
from statistics import fmean, stdev


matches = list(Path.cwd().rglob("rq01_e2_pilot_20260711/per_seed_summary.csv"))
assert len(matches) == 1, matches
with matches[0].open(encoding="utf-8", newline="") as handle:
    rows = list(DictReader(handle))

CONDITION = "high"
BASELINE = "fifo"
STRATEGY = "predicted_sjf"
METRIC = "p99_wait_time"


def values_by_seed(strategy: str) -> dict[int, float]:
    return {
        int(row["seed"]): float(row[METRIC])
        for row in rows
        if row["prediction_error"] == CONDITION and row["strategy"] == strategy
    }


fifo = values_by_seed(BASELINE)
predicted = values_by_seed(STRATEGY)
seeds = sorted(set(fifo) & set(predicted))
deltas = [predicted[seed] - fifo[seed] for seed in seeds]
assert len(deltas) == 30

mean_delta = fmean(deltas)
standard_error = stdev(deltas) / sqrt(len(deltas))

# 仅用于这个固定 30-seed 教学例子：df=29 的双侧 95% t 临界值。
t_critical_df29 = 2.045229642
t_ci = (
    mean_delta - t_critical_df29 * standard_error,
    mean_delta + t_critical_df29 * standard_error,
)


def nearest_rank(values: list[float], probability: float) -> float:
    ordered = sorted(values)
    index = ceil(probability * len(ordered)) - 1
    return ordered[max(0, min(index, len(ordered) - 1))]


# 与正式 runner 一样，为每个 comparison 派生独立且可复现的 bootstrap seed。
key = f"{CONDITION}:{STRATEGY}:{METRIC}"
digest = sha256(f"20260711:{key}".encode("utf-8")).digest()
rng = Random(int.from_bytes(digest[:8], "big"))
bootstrap_means = [
    fmean(deltas[rng.randrange(len(deltas))] for _ in deltas)
    for _ in range(5_000)
]
bootstrap_ci = (
    nearest_rank(bootstrap_means, 0.025),
    nearest_rank(bootstrap_means, 0.975),
)

print(f"seeds={len(seeds)}")
print(f"mean paired delta={mean_delta:.4f}")
print(f"t interval=({t_ci[0]:.4f}, {t_ci[1]:.4f})")
print(f"percentile bootstrap=({bootstrap_ci[0]:.4f}, {bootstrap_ci[1]:.4f})")

assert mean_delta > 0
assert bootstrap_ci[0] > 0
```

预期输出：

```text
seeds=30
mean paired delta=52.1519
t interval=(20.1387, 84.1651)
percentile bootstrap=(24.3478, 85.3374)
```

这里 `mean_delta > 0` 表示 high-error predicted SJF 的 P99 等待相对 FIFO 更高。两个区间在当前 pilot 中都没有跨 0，但结论仍然只能限定为：在指定合成、单 worker、30-seed pilot 条件下观察到 P99 伤害；它不能证明真实 RAG/Agent 或生产系统中的普遍效果。

代码中的 t 临界值只适用于这个固定的 30-seed 双侧 95% 示例。seed 数改变时，应使用经过验证的统计库计算对应临界值，不能继续复制 `2.045229642`。

### 5.6.10 常见错误

第一个错误：把一个固定 seed 当作重复实验。

固定 seed 只让同一次 workload 可重放。必须有多个预先指定的 seeds，才能观察随机 workload 变化。

第二个错误：分别 bootstrap 两个策略。

RQ01 是配对设计，应先按 seed 计算 `strategy - FIFO`，再重采样完整配对差值。分别重采样会破坏 CRN 带来的对应关系。

第三个错误：把 task 行当独立样本。

同一 run 内任务共享队列状态。用大量 task 行制造极窄 CI 是伪重复，不会让结论更可信。

第四个错误：只看区间是否跨 0。

必须同时报告效应量、单位、主要/次要指标、失败率、公平性伤害和证据边界。

第五个错误：看到哪个切片好就报告哪个。

多策略、多条件、多指标会放大偶然发现。primary comparison 必须预先固定，其余结果按 secondary 或 exploratory 标记。

第六个错误：按结果追加 seeds，直到区间不跨 0。

停止规则和最大 seed 数应在看结论前固定，或按预先声明的 CI 半宽规则决定；不能以“刚好得到想要的方向”为停止条件。

## 第 7 章：失败分析，实验不支持假设也有价值

### 7.1 为什么要学

科研训练不是只寻找好看的结果。

如果实验不支持假设，也要分析原因：

- 负载规模太小。
- 任务分布不合理。
- 指标口径错了。
- 随机性太强。
- baseline 不公平。
- 策略实现有 bug。
- 结论需要分组指标才能看清。

### 7.2 RQ01 常见失败

| 失败现象 | 可能原因 |
|---|---|
| 策略差异不明显 | 低负载下队列不积压 |
| SJF 平均值好但 P99 差 | 长任务被推迟 |
| Priority 整体指标变差 | 高优先级收益没有分组展示 |
| worker 增加后结论变化 | 资源数量改变了瓶颈 |
| Cost-aware 没有效果 | 权重设计不敏感或任务特征不冲突 |

### 7.3 失败怎么写

不要写：

```text

```

要写：

```text
本轮实验在低负载任务流下未观察到明显策略差异，原因可能是队列长度长期接近 0，worker 没有进入饱和状态。下一步应使用高峰任务流或增加并发，使调度策略有发挥空间。
```

第一个错误：把不符合预期的数据删掉。

第二个错误：把失败都归咎于代码。

第三个错误：不提出下一轮实验。

```text
config + workload + per-task results + per-seed summary + paired analysis + errors

报告：
RQ01 初版实验报告
```

### 不允许写成结论的内容

如果还没有真实 P03 压测数据，就不能写：

```text
该方法已经在真实 RAG/Agent 平台证明有效。
```

只能写：

```text
当前结果来自模拟调度实验，说明在该任务流下不同策略存在平均等待和尾延迟取舍；后续需要接入 P03 RAG/Agent 请求和压测数据进一步验证。
```

## 第一轮学习顺序

1. 读第 1-2 章，把 RQ01 写成研究问题和假设。
2. 读第 3-4 章，整理 baseline 和变量表。
3. 读第 5 章，确认指标口径。
4. 读第 5.5 章，区分性能指标与 Rubric/人工评测。
5. 读第 5.6 章，确认推断单位，复算多 seed 配对差值、效应量和 95% CI。
6. 读第 6-7 章，补实验记录和失败分析。
7. 读第 8 章，写 RQ01 初版实验报告结构。
8. 读第 9 章，生成导师沟通版本。
9. 回到 RQ01 文件夹，先复算 E2 reference artifact，再把学习者亲手运行的新证据写入问题、设计、记录和报告。

## 暂时不要深入

- 不写论文大综述。
- 不虚构实验结论。
- 不提前包装成论文成果。
