# 第 9 章：分组分析：谁被牺牲了

## 9.1 本章目标

第 8 章已经讲了 Cost-aware 的成本分数和权重总表。

总表能回答：

```text
哪组权重的平均等待更低？
哪组权重的 P95/P99 更差？
worker utilization 有没有变化？
```

但总表回答不了一个很关键的问题：

```text
到底是谁变快了，谁被推迟了？
```

这就是本章要解决的问题。

学完本章，你应该能做到：

- 解释为什么整体平均值不够。
- 理解 `task_type` 分组的意义。
- 手写一个最小 `summarize_by_task_type`。
- 读懂 Cost-aware 权重实验里的分组表。
- 说明 duration_dominant、token_dominant、priority_dominant 分别改善了谁、牺牲了谁。
- 把“策略好坏”改写成“策略把等待时间重新分配给了哪些任务”。

本章仍然不展开 aging、最大等待保护、多 worker、真实 RAG 或 Kubernetes。

本章只解决一个问题：

```text
为什么只看总表会掩盖某些任务类型被牺牲，以及如何用 task_type 分组分析解释 Cost-aware 的副作用？
```

## 9.2 为什么总表不够

第 8 章里，Cost-aware 权重总表是：

| 权重预设 | 平均等待时间 | 最大等待时间 | P95 | P99 | 最大队列长度 | worker 利用率 |
|---|---:|---:|---:|---:|---:|---:|
| default | 41.97 | 104.40 | 99.60 | 104.40 | 24 | 0.92 |
| duration_dominant | 37.30 | 104.40 | 99.60 | 104.40 | 24 | 0.92 |
| token_dominant | 48.63 | 110.80 | 110.60 | 110.80 | 24 | 0.92 |
| priority_dominant | 46.63 | 104.40 | 99.60 | 104.40 | 27 | 0.92 |

这张表能看出 duration_dominant 的平均等待最低。

但它不能告诉你：

```text
duration_dominant 到底改善了哪些任务？
它牺牲了哪些任务？
token_dominant 为什么尾部更差？
priority_dominant 保护了谁？
```

如果只看总表，你很容易写出这种过度结论：

```text
duration_dominant 最好。
token_dominant 最差。
priority_dominant 没什么用。
```

这三句话都太粗糙。

调度策略不是让所有任务一起变好。

更常见的情况是：

```text
一部分任务变快，另一部分任务变慢。
```

分组分析就是为了看见这种转移。

## 9.3 task_type 是什么

`task_type` 是任务类型。

在 P01 的 Cost-aware 实验里，任务流包含六类任务：

| task_type | 特征 | 想观察的问题 |
|---|---|---|
| `short_high_token` | 执行短，但 token 很高 | token 权重过高时会不会被推迟 |
| `long_low_token` | 执行长，但 token 很低 | duration 权重过高时会不会被推迟 |
| `urgent_medium` | 高优先级，中等耗时 | priority 权重能否保护它 |
| `low_priority_short` | 低优先级，但执行很短 | priority 权重会不会牺牲它 |
| `batch_heavy` | 低优先级，耗时和 token 都高 | 高成本批任务是否被后置 |
| `cheap_medium` | 中等耗时，token 很低 | token 主导时是否受益 |

这些名字不是随便起的。

它们刻意制造了冲突：

```text
短耗时 vs 高 token
低 token vs 长耗时
高优先级 vs 中等成本
低优先级 vs 短耗时
```

只有存在冲突，分组分析才有意义。

如果所有任务都一样，分组表就不会提供新信息。

## 9.4 分组分析的数据流

分组分析的数据流可以画成这样：

```mermaid
flowchart LR
    A["完成任务列表<br/>completed tasks"] --> B["按 task_type 分组"]
    B --> C["每组 waiting_time 列表"]
    C --> D["每组 average / max / P95"]
    D --> E["比较不同权重下的同一类任务"]
    E --> F["解释谁被改善，谁被牺牲"]
```

注意，这里有两个比较方向。

第一个方向是横向看同一组权重：

```text
duration_dominant 里，哪些 task_type 等得久？
```

第二个方向是纵向看同一个任务类型：

```text
short_high_token 在 duration_dominant、token_dominant、priority_dominant 下分别等多久？
```

第二个方向更重要。

因为它能看出权重变化如何影响同一类任务。

## 9.5 手写 summarize_by_task_type

先写一个最小版本。

假设你已经有 `completed`：

```python
def summarize_by_task_type(completed: list[Task]) -> list[dict]:
    groups: dict[str, list[Task]] = {}

    for task in completed:
        groups.setdefault(task.task_type, []).append(task)

    rows = []
    for task_type in sorted(groups):
        tasks = groups[task_type]
        wait_times = [
            task.start_time - task.submit_time
            for task in tasks
            if task.start_time is not None
        ]
        rows.append(
            {
                "task_type": task_type,
                "task_count": len(tasks),
                "average_wait_time": average(wait_times),
                "max_wait_time": max(wait_times) if wait_times else 0.0,
                "p95_wait_time": p95(wait_times),
            }
        )

    return rows
```

这个函数做三件事。

第一，按 `task.task_type` 建组。

第二，对每组任务计算 waiting time。

第三，对每组输出 average、max、P95。

为什么暂时不用 P99？

因为每组当前只有 5 个任务，P95/P99 都会很接近最大值。

第一轮分组分析先看：

```text
average
max
P95
```

这已经足够看出谁被牺牲。

## 9.7 duration_dominant 改善了谁，牺牲了谁

duration_dominant 的分组结果是：

| task_type | count | 平均等待时间 | 最大等待时间 | P95 |
|---|---:|---:|---:|---:|
| batch_heavy | 5 | 94.80 | 104.40 | 104.40 |
| cheap_medium | 5 | 9.60 | 13.20 | 13.20 |
| long_low_token | 5 | 59.40 | 73.00 | 73.00 |
| low_priority_short | 5 | 0.40 | 0.40 | 0.40 |
| short_high_token | 5 | 32.40 | 40.80 | 40.80 |
| urgent_medium | 5 | 27.20 | 32.80 | 32.80 |

duration_dominant 重视执行耗时。

所以它最照顾短任务。

最明显的是：

```text
low_priority_short 平均等待 0.40
```

这说明低优先级短任务几乎立刻被执行。

`cheap_medium` 也比较好：

```text
cheap_medium 平均等待 9.60
```

但它牺牲了长任务和重任务：

```text
batch_heavy 平均等待 94.80
long_low_token 平均等待 59.40
```

这里的结论不能写成：

```text
duration_dominant 全面最好。
```

更准确的写法是：

```text
duration_dominant 显著改善短任务等待，但把高成本批任务和长耗时任务推到队尾。
```

## 9.8 token_dominant 改善了谁，牺牲了谁

token_dominant 的分组结果是：

| task_type | count | 平均等待时间 | 最大等待时间 | P95 |
|---|---:|---:|---:|---:|
| batch_heavy | 5 | 90.80 | 100.40 | 100.40 |
| cheap_medium | 5 | 9.60 | 13.20 | 13.20 |
| long_low_token | 5 | 35.40 | 49.00 | 49.00 |
| low_priority_short | 5 | 0.40 | 0.40 | 0.40 |
| short_high_token | 5 | 88.40 | 110.80 | 110.80 |
| urgent_medium | 5 | 67.20 | 72.80 | 72.80 |

token_dominant 重视 token 成本。

它会照顾 token 少的任务。

例如：

```text
long_low_token 平均等待 35.40
```

对比 duration_dominant 下的 `long_low_token`：

```text
duration_dominant: 59.40
token_dominant: 35.40
```

token_dominant 明显改善了它。

但代价是 `short_high_token` 被严重推迟：

```text
short_high_token 平均等待 88.40
P95 110.80
```

这类任务虽然执行短，但 token 很高。

token_dominant 会认为它成本高，于是把它往后放。

这解释了第 8 章总表里 token_dominant 的 P95/P99 为什么最差。

更准确的结论是：

```text
token_dominant 能保护低 token 任务，但可能严重牺牲高 token 短任务和高优先级中等任务。
```

## 9.9 priority_dominant 改善了谁，牺牲了谁

priority_dominant 的分组结果是：

| task_type | count | 平均等待时间 | 最大等待时间 | P95 |
|---|---:|---:|---:|---:|
| batch_heavy | 5 | 94.80 | 104.40 | 104.40 |
| cheap_medium | 5 | 23.60 | 27.20 | 27.20 |
| long_low_token | 5 | 55.40 | 69.00 | 69.00 |
| low_priority_short | 5 | 35.40 | 35.40 | 35.40 |
| short_high_token | 5 | 64.40 | 80.80 | 80.80 |
| urgent_medium | 5 | 6.20 | 11.80 | 11.80 |

priority_dominant 重视业务优先级。

它最明显地改善了：

```text
urgent_medium 平均等待 6.20
```

对比其他权重：

```text
duration_dominant: 27.20
token_dominant: 67.20
priority_dominant: 6.20
```

这说明 priority_dominant 确实保护了高优先级任务。

但它牺牲了低优先级短任务：

```text
low_priority_short 平均等待 35.40
```

而在 duration_dominant 和 token_dominant 下，这类任务的平均等待都是：

```text
0.40
```

这就是 Priority 类策略的典型副作用：

```text
它保护重要任务，但可能牺牲不重要但很快能完成的任务。
```

这里不能简单说 priority_dominant 差。

如果 `urgent_medium` 代表在线用户请求，priority_dominant 可能是业务上正确的。

但你必须报告它的代价：

```text
low_priority_short 被明显推迟。
```

## 9.10 横向比较：同一类任务在不同权重下怎么变化

分组分析最有价值的方式，是挑同一类任务横向比较。

例如 `short_high_token`：

| 权重 | 平均等待时间 | P95 | 解释 |
|---|---:|---:|---|
| duration_dominant | 32.40 | 40.80 | 执行短，所以还能较早执行 |
| token_dominant | 88.40 | 110.80 | token 高，被严重推迟 |
| priority_dominant | 64.40 | 80.80 | 不如 duration，但比 token 主导好 |

这说明：

```text
高 token 短任务最怕 token 权重过大。
```

再看 `urgent_medium`：

| 权重 | 平均等待时间 | P95 | 解释 |
|---|---:|---:|---|
| duration_dominant | 27.20 | 32.80 | 中等耗时，不是最先被照顾 |
| token_dominant | 67.20 | 72.80 | token 不是最低，且 priority 权重弱 |
| priority_dominant | 6.20 | 11.80 | 高优先级被明显保护 |

这说明：

```text
高优先级任务只有在 priority 权重足够高时才会被明显保护。
```

再看 `low_priority_short`：

| 权重 | 平均等待时间 | P95 | 解释 |
|---|---:|---:|---|
| duration_dominant | 0.40 | 0.40 | 很短，所以几乎立刻执行 |
| token_dominant | 0.40 | 0.40 | token 低，也能很快执行 |
| priority_dominant | 35.40 | 35.40 | 低优先级，被明显推迟 |

这说明：

```text
低优先级短任务最怕 priority 权重过大。
```

## 9.11 把结论写得像工程报告

不要这样写：

```text
duration_dominant 最好。
token_dominant 最差。
priority_dominant 一般。
```

这仍然是总表思维。

应该这样写：

```text
duration_dominant 在当前任务流中平均等待最低，主要因为它明显改善了 low_priority_short 和 cheap_medium 等短任务；代价是 batch_heavy 与 long_low_token 等长任务等待较高。
```

再比如：

```text
token_dominant 强化了 token 成本约束，使 long_low_token 等低 token 任务等待下降；但 short_high_token 被严重推迟，导致 P95/P99 和最大等待时间恶化。
```

再比如：

```text
priority_dominant 显著保护 urgent_medium，高优先级任务平均等待从 token_dominant 下的 67.20 降到 6.20；代价是 low_priority_short 从 0.40 上升到 35.40。
```

这样的表达才说明你真的理解了调度策略。

## 9.12 本章和第 10 章的关系

本章只负责发现问题：

```text
谁被牺牲了？
```

第 10 章才负责尝试解决问题：

```text
如果某类任务长期被推迟，能不能用 aging / 最大等待保护缓解？
```

所以本章不要急着写 aging 公式。

你只需要把问题讲清楚：

- token_dominant 牺牲了 `short_high_token`。
- priority_dominant 牺牲了 `low_priority_short`。
- duration_dominant 牺牲了 `batch_heavy` 和 `long_low_token`。

只有把“谁被牺牲”说清楚，后面的保护机制才有目标。

## 9.14 常见错误

第一个错误：只看总表。

总表告诉你整体指标，分组表告诉你代价落到了谁身上。

第二个错误：只找“最优权重”。

分组分析不是为了给权重排名，而是为了解释权重偏好。

第三个错误：把 task_type 名字当装饰。

`short_high_token`、`long_low_token`、`urgent_medium` 这些名字都在表达任务特征。

第四个错误：只看 average，不看 max 和 P95。

某类任务平均还可以，但 P95 很差，也说明组内有尾部风险。

第五个错误：看到某类任务被牺牲，就马上改算法。

本章先分析问题，第 10 章再讲保护机制。

## 9.17 本章小结

本章把 Cost-aware 从总表分析推进到了分组分析。

你现在应该能理解：

```text
策略不是单纯让系统变快，而是在不同任务类型之间重新分配等待时间。
```

P01 当前结果说明：

- duration_dominant 照顾短任务，但推迟重任务和长任务。
- token_dominant 照顾低 token 任务，但严重推迟高 token 短任务。
- priority_dominant 照顾高优先级任务，但牺牲低优先级短任务。
- 分组表能解释总表里 P95/P99 为什么变差。

本章最重要的能力是：

```text
不要只说指标变好了，而要说清楚谁变好了、谁变差了。
```

如果这些内容能说清楚，就可以进入第 10 章：Aging / 最大等待保护。
