# 第 12 章：项目总结与复盘

## 12.1 本章目标

前面 11 章已经把 M05 的核心内容走完：

```text
任务建模 -> 调度策略 -> 指标统计 -> 高峰负载 -> 成本感知 -> 分组分析 -> 等待保护 -> 多 worker
```

这一章不再引入新的调度算法。

它的任务是把已经学过的东西收束成三类成果：

```text
你真正理解了什么？
你亲手复现了什么？
哪些内容可以继续项目化表达？
```

学完本章，你应该能做到：

- 用自己的话复述 M05 的完整主线。
- 区分“P01 参考答案”和“自己亲手完成的项目成果”。
- 写出 5 条稳定的项目结论。
- 说明项目局限，不夸大当前能力。
- 知道 README、简历表达和科研问题分别应该怎么转化。

本章仍然不扩展 FastAPI、Redis、真实 RAG、Kubernetes、GPU 调度或生产系统。

本章只解决一个问题：

```text
如何把 M05 学到的调度模型、策略、指标和实验结论整理成可复盘、可继续项目化的成果？
```

## 12.2 先确认 P01 的正确定位

这件事要再强调一次。

P01 Mini Scheduler 当前在 M05 里不是直接拿来当成果展示的项目。

它的正确定位是：

```text
M05 教材配套参考答案 / 实验样板 / 后续亲手复现的对照材料
```

也就是说，P01 里的 README、简历表达、图表、实验记录都可以参考。

但你不能在没有亲手复现和解释之前，把它直接当作自己的项目成果。

正确顺序是：

```text
先按章节亲手写核心逻辑
再跑实验
再解释结果
再对照 P01
最后改写成自己的 README / 简历表达
```

如果跳过前面几步，项目表达会提前透支。

别人一问：

```text
为什么 token_dominant 的 P99 更差？
为什么 strong_aging 有效但不是免费优化？
为什么 8 workers 的 P95 最低但不一定最好？
```

你就会卡住。

本章就是为了避免这种情况。

## 12.3 M05 到底证明了什么

M05 不是证明“某个调度策略最好”。

它证明的是：

```text
调度系统必须同时看策略、负载、指标、公平性和资源成本。
```

更具体地说，M05 证明了五件事。

第一，调度不是简单排序。

排序函数只决定候选任务的顺序，真正的调度还要处理：

- 任务什么时候到达。
- worker 什么时候空闲。
- 哪些任务当前可选。
- 任务完成后指标如何变化。

第二，平均等待时间不够。

平均值可能掩盖尾部任务的糟糕体验。

所以要同时看：

```text
average wait
max wait
P95
P99
queue length
worker utilization
```

第三，高峰负载会放大策略差异。

低负载下，策略差异可能很小。

高峰负载下，队列积压，策略才真正体现取舍。

第四，Cost-aware 权重会改变谁受益、谁被牺牲。

例如：

- duration_dominant 照顾短任务。
- token_dominant 可能推迟高 token 短任务。
- priority_dominant 保护高优先级任务，但可能牺牲低优先级短任务。

第五，保护机制和扩容都不是免费午餐。

Aging 能降低极端等待，但会改变原本排序。

增加 worker 能降低 P95/P99，但会降低 utilization。

## 12.4 你应该已经掌握的能力

学完 M05，你不需要声称自己会做完整生产级调度器。

但你应该能说清楚自己具备下面这些能力。

### 建模能力

你应该能解释：

- Task 是什么。
- Worker 是什么。
- Queue / pending / available_tasks 有什么区别。
- `submit_time`、`start_time`、`finish_time` 分别表示什么。
- 为什么不能调度未来还没到达的任务。

### 策略能力

你应该能手写和解释：

- FIFO
- Priority
- SJF
- Cost-aware
- Aging / 最大等待保护

你不需要说它们谁绝对最好。

你要能说：

```text
每种策略优化什么，牺牲什么，适合什么场景。
```

### 指标能力

你应该能计算和解释：

- waiting time
- turnaround time
- average
- P95 / P99
- max wait
- max queue length
- worker utilization

尤其要能解释：

```text
为什么 P95/P99 和 utilization 必须一起看。
```

### 实验能力

你应该能跑和解释：

- 小样例策略对比。
- 高峰负载实验。
- Cost-aware 权重实验。
- task_type 分组分析。
- aging / 最大等待保护实验。
- worker 数量实验。

重点不是会运行命令，而是能解释结果。

## 12.5 五条稳定项目结论

你可以先把 M05 的结论写成下面 5 条。

第一条：

```text
低负载下，调度策略差异不明显；高峰负载下，队列积压会放大策略取舍。
```

第二条：

```text
SJF / Cost-aware 可以降低平均等待时间，但不一定降低 P95/P99 或最大等待。
```

第三条：

```text
Cost-aware 权重不是唯一最优公式，不同权重会改变不同 task_type 的等待时间分布。
```

第四条：

```text
分组分析能暴露“整体指标变好时，某类任务被牺牲”的问题。
```

第五条：

```text
Aging 和多 worker 都能缓解尾部等待，但前者改变任务顺序，后者增加资源成本。
```

这些结论是稳妥的。

它们没有夸大项目，也没有声称生产级能力。

## 12.6 哪些能写进 README

README 可以写：

- 项目定位：最小调度模拟器。
- 核心模型：Task / Worker / Queue。
- 支持策略：FIFO / Priority / SJF / Cost-aware / Aging。
- 支持实验：高峰负载、权重敏感、分组分析、worker 数量。
- 支持指标：average、P95/P99、max wait、queue length、utilization。
- 稳定实验结论。

README 不应该写：

- 已经是生产级调度系统。
- 已经支持真实 Kubernetes 调度。
- 已经接入真实 RAG / Agent 平台。
- 已经解决所有尾延迟问题。
- 找到了最优调度算法。

更好的写法是：

```text
这是一个用于理解 AI workload 调度取舍的最小模拟项目。
```

而不是：

```text
这是一个完整 AI Infra 调度平台。
```

## 12.7 哪些能进入简历表达

简历表达必须建立在你亲手复现和解释的基础上。

如果你已经完成复现，可以写：

```text
实现 Mini Scheduler 调度模拟器，支持 FIFO、Priority、SJF、Cost-aware、aging / 最大等待保护和多 worker 模拟，统计平均等待时间、P95/P99、最大队列长度和 worker 利用率。
```

也可以写实验分析版：

```text
设计高峰负载、Cost-aware 权重和 worker 数量实验，分析不同策略在平均等待、P95/P99 尾延迟和资源利用率上的取舍；实验显示 FIFO 下 worker 从 1 增至 8 可将 P95 从 97.70 降至 3.50，但利用率从 0.97 降至 0.30。
```

但如果你还没亲手复现，就只能写成学习复盘，不适合写进正式简历。

可以先写：

```text
正在复现 Mini Scheduler 调度模拟项目，重点理解任务队列、调度策略、尾延迟指标和资源利用率之间的关系。
```

这更诚实，也更安全。

## 12.8 项目局限必须写清楚

一个成熟的项目表达必须有局限。

P01 当前局限包括：

- 任务是模拟任务，不是真实 RAG / Agent 请求。
- `estimated_duration` 默认已知，真实系统里通常只能估计。
- 当前没有真实失败、重试、超时、取消逻辑。
- 当前没有数据库、消息队列、API 服务。
- 当前不是 Kubernetes 调度器。
- 当前不处理真实 GPU、显存、模型并发、batching 等推理系统问题。

这些不是缺点。

它们是边界。

你能说清楚边界，反而说明你理解项目。

## 12.12 常见错误

第一个错误：把参考答案当成自己的成果。

P01 是参考答案，必须先复现再表达。

第二个错误：夸大成生产系统。

当前只是调度模拟项目。

第三个错误：只写功能，不写实验结论。

M05 的价值在于能解释策略取舍，而不是列一堆函数。

第四个错误：只写结果，不写局限。

没有局限的项目表达往往不可信。

第五个错误：过早接真实系统。

如果调度内核还讲不清楚，接 API、Redis、Kubernetes 只会让问题更乱。
