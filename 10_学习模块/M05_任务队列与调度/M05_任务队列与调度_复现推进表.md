# M05 任务队列与调度复现推进表

## 这份文档的定位

M05 第 0-12 章已有连续的主体草稿和可用 reference，但内容成熟度仍为 `draft`、教学就绪仍为
`partial`。本阶段先暂停横向扩展新技术点，把已写内容亲手复现，再把复现中暴露的解释缺口反馈到
教材审校；这不表示整本教材已经完成。

这份推进表只解决一个问题：

```text
我接下来每一轮应该读哪几章、亲手写什么、跑什么结果、留下什么复盘？
```

它不是项目台账，也不是新的教材正文。真正学习时仍以 [[10_学习模块/M05_任务队列与调度/M05_任务队列与调度_章节教材|M05 章节教材]] 为主。

## 总原则

每一轮都按同一个闭环执行：

```text
读概念 -> 手写最小代码 -> 跑小样例 -> 观察结果 -> 对照 P01 -> 写复盘
```

其中最重要的是“先手写，再对照”。P01 Mini Scheduler 当前仍然是：

```text
M05 教材配套参考答案 / 实验样板 / 后续亲手复现的对照材料
```

不要先复制 P01 代码，也不要提前把 README 或简历表达当成自己的成果。

## 当前不推进的内容

第一轮复现期间暂时不做：

- FastAPI 服务化
- Redis / Celery / RQ
- 数据库持久化
- Docker 部署
- Kubernetes Scheduler 源码
- 真实 RAG / Agent 请求
- GPU 调度
- 强化学习调度或论文复现

这些不是永远不做，而是等 M05 调度内核能够亲手复现、解释和修改之后，再放到 M02 / M06 / M08 / M09 / P03 里展开。

## 什么时候可以转成项目成果

只有满足下面条件后，P01 才适合从“参考答案”转成“你的项目成果”：

- 能不看 P01 代码，写出 Task / Worker / FIFO / Priority / SJF 的核心逻辑。
- 能解释 waiting time、turnaround time、P95/P99、worker utilization 的计算口径。
- 能用同一组任务跑不同策略，并解释指标变化。
- 能说清 Cost-aware、aging、多 worker 分别解决什么问题，又带来什么代价。
- 能主动说明项目局限：模拟任务、非真实 RAG、非 Kubernetes、没有数据库和服务化。

达到这个状态后，再进入 README、实验报告和简历表达会更稳。
