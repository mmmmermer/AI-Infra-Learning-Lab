# M06 数据库缓存与异步任务学习地图

## 怎么读这个模块

把 M06 当成“一个 task_id 的一生”来读，不要当成数据库、Redis、Celery/RQ 工具大全。

阅读主线是：API 创建 task，写入 tasks 表，入队，worker 读取 task_id，状态变成 running，执行后回写 result 或 error，用户查询状态。SQL、Redis、重试、幂等都服务这条线。

第一轮只要能讲清 `pending -> queued -> running -> succeeded/failed`，就比背很多数据库概念更重要。

## 在总路线中的位置

M06 负责把 M02 的 API、M03 的 RAG 请求、M05 的调度任务，从内存脚本推进到可保存、可恢复、可异步执行的工程系统。

在 P03 中，它承接这条链路：

```text
API 提交请求
-> PostgreSQL 同事务写 task + outbox
-> dispatcher 发布 task_id 到 Redis Stream
-> consumer-group worker reserve 并用数据库 CAS/fencing claim
-> worker 异步执行
-> 状态和结果回写
-> metrics 可查询
```

## 要解决的问题

- RAG 请求和调度任务如何从内存对象变成数据库记录？
- API 如何快速返回 task_id，而不是同步等待长任务？
- worker 如何异步执行任务并回写状态？
- 失败、重试、幂等和状态流转如何设计？
- Redis 缓存如何降低 embedding 和重复查询成本？
- M06 如何给 M05 调度和 M08 监控提供工程数据？

## 学习目标

- [ ] 能设计 P03 第一版 `tasks` 表。
- [ ] 能保存 pending/queued/running/succeeded/failed/retrying/cancelled 状态。
- [ ] 能把 `POST /tasks`（`task_type=rag_retrieval`）变成“task + outbox 同事务入库、异步入队、返回 task”。
- [ ] 能用 Redis 做 embedding 缓存或任务队列支撑。
- [ ] 能解释 Redis Streams consumer-group worker 如何 reserve、claim、回写并在数据库终态后 ack。
- [ ] 能设计失败重试、幂等键和基础 metrics 字段。

## 核心内容

| 内容 | 学到什么程度 | 落地点 |

## 推荐学习顺序

1. 读 [[M06_数据库缓存与异步任务_适配教材|M06 适配教材]] 第 1-2 章，理解为什么任务必须持久化。
2. 完成 E06-01：设计 tasks 表并实现状态更新。
3. 对照 M02，把 `POST /tasks` 改造成 repository + transactional outbox + queue 流程。
4. 学 Redis Streams、consumer groups 和 worker claim/ack，完成 E06-02/E06-03；RQ/Celery 只作框架对比。
5. 加入失败重试、幂等键和 metrics 字段。
6. 最后把结果接回 P03 的任务状态页和 metrics API。

## E06 练习路线

```text
任务入库
-> 状态更新
-> 异步执行
-> 失败重试
-> 查询状态
```

第一轮可以先用 SQLite 和伪 worker 做通，再对照 P03 接 PostgreSQL transactional outbox、dispatcher
与 Redis Streams consumer group。RQ/Celery 只作为进阶对比，不作为当前 reference 主线。
