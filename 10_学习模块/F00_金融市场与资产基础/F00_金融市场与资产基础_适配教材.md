## 本模块最值得理解的难点

F00 的难点不是名词多，而是同一个“金融数据”可能来自完全不同的世界。

| 难点 | 为什么难 | 工程痛点 | 后续影响 |
|---|---|---|---|
| 资产类别不同 | 股票、债券、期权的字段和风险完全不同 | 同一套代码不能随便套所有资产 | F02 数据字段和 F03 指标会变 |
| 价格不是收益率 | 价格水平不能直接比较资产表现 | 需要转换成 return 才能做风险指标 | F02/F03 的基础 |
| 原始价格和复权价格不同 | 分红、拆股会影响价格序列 | 用错字段会让收益率断裂 | 回测和风险指标失真 |
| 利率既是价格也是输入 | 利率影响债券、折现和宏观环境 | FRED 等宏观序列和股票价格频率不同 | F05/Q01/P03 |
| 财报和市场数据不同 | 财报是公司披露，市场数据是交易结果 | 文档 RAG 和时间序列处理方式不同 | M12/M03 与 F02 边界 |
| 风险不是坏事本身 | 风险是结果不确定和损失可能性 | 指标必须带样本范围和口径 | F03/Q02 表达边界 |

学习 F00 的核心习惯是：

```text
看到一个字段，先问它属于哪类数据、来自哪里、代表什么、能不能用于当前实验。
```

## 一条贯通学习线

F00 最容易被学成“金融名词表”，所以这里先给一条贯通线。你后面每次看到金融数据，都按这条线处理：

```text
金融对象
-> 数据字段
-> 字段口径
-> 可计算指标
-> 实验记录
-> P03 task 输入
```

例如看到一列 `adjusted_close`：

| 步骤 | 你要问的问题 | 合格回答 |
|---|---|---|
| 金融对象 | 它属于什么资产？ | 股票/ETF 的复权价格序列 |
| 数据字段 | 它代表什么？ | 分红、拆股调整后的收盘价 |
| 字段口径 | 它和 `close` 有什么区别？ | `close` 是原始收盘价，长期收益率通常更适合用复权价 |
| 可计算指标 | 能算什么？ | 日收益率、滚动波动率、最大回撤 |
| 实验记录 | 必须记录什么？ | 来源、时间范围、频率、字段说明、是否复权 |
| P03 task | 进入什么任务？ | `data_quality_task` 或 `risk_metric_task` |

这个流程的价值是：你不是先背金融概念，而是先学会判断一个字段能不能被安全地用于实验。

## 可迁移的原则

1. 先识别数据类型，再决定处理方法。市场价格、财报文本、宏观序列和公告文件不能混成同一种“金融数据”，否则后续 F02 的清洗、F03 的风险指标、M12 的 RAG 引用都会失去边界。
2. 先记录字段口径，再计算指标。`close`、`adjusted_close`、`rate`、`filing_date` 这类字段看起来简单，但只要含义不清，收益率、波动率、检索证据和项目表达都会被污染。
3. 先写清不能说明什么，再写能说明什么。F00 的结论只能说明数据来源、字段含义和任务边界，不能自然推出投资建议、收益预测或资产配置观点。

## 踩坑现场

> 你下载了一段股票价格，用 `close` 直接算收益率，然后发现某天收益率异常大。代码没有报错，但原因可能是分红、拆股或数据供应商字段口径不同。正确做法不是马上删掉异常值，而是先查字段说明、确认是否需要 `adjusted_close`，再把处理选择写进实验记录。

## 第 1 章：金融市场在做什么

### 1.1 本章解决什么问题

金融市场不是一个抽象名词。它至少包含：

- 资金需求方：公司、政府、金融机构。
- 资金供给方：个人、基金、银行、保险等。
- 交易场所和中介：交易所、券商、清算机构。
- 信息披露和监管：财报、公告、监管文件。
- 数据记录：价格、成交量、利率、财务指标、宏观数据。

### 1.2 金融直觉

金融市场的一个基本作用是把资金、风险和信息连接起来。

例如，公司发行股票或债券获得资金；投资者承担不确定性，希望获得回报；市场价格不断反映交易者对信息和风险的判断。

### 1.3 工程场景

在本学习库里，金融市场不是用来做投资判断，而是提供任务场景：

```text
公开数据
-> F02 清洗时间序列
-> F03 计算风险指标
-> M12 做金融文档 RAG
-> P03 执行 finance workload
```

### 1.4 常见错误

- 把金融市场学习等同于炒股。
- 只看价格涨跌，不看数据来源和字段口径。
- 把财报文本、市场价格和宏观序列混成一种数据。
- 在还没有实验记录时写项目成果。

### 1.5 小练习

列出三类数据：

| 数据 | 来源 | 未来用途 |
|---|---|---|
| 股票价格 | 市场数据 API | F02/F03 |
| 公司财报 | SEC EDGAR | M12/M03 |
| 利率序列 | FRED | F05/Q01 |

## 第 2 章：资产类别和数据字段

### 2.1 本章解决什么问题

不同资产的数据字段不同，风险也不同。F00 第一轮只需要建立基本区分。

### 2.2 股票

股票代表公司所有权的一部分。第一轮关注字段：

| 字段 | 含义 | 后续用途 |
|---|---|---|
| `open/high/low/close` | 交易价格 | F02 时间序列 |
| `adjusted_close` | 复权价格 | 收益率和风险指标 |
| `volume` | 成交量 | 第一轮只记录 |
| `ticker` | 股票代码 | 数据标识 |

股票风险包括价格波动、公司经营风险、市场风险和流动性风险。

### 2.3 债券

债券更像借款凭证。第一轮关注：

| 概念 | 含义 |
|---|---|
| face value | 到期偿还本金 |
| coupon | 票息 |
| maturity | 到期日 |
| yield | 收益率/到期收益率 |
| duration | 利率敏感性 |

债券风险包括利率风险、信用风险和流动性风险。

### 2.4 基金和指数

基金和指数常用于表示一组资产。

第一轮只需要知道：

- 指数不是一个单独公司。
- ETF/基金可能跟踪某个指数或策略。
- 指数价格可以用于市场基准，但不能当成个股。

### 2.5 期货和期权

期货和期权属于衍生品。第一轮只建立直觉：

- 期货：约定未来按某价格买卖。
- 期权：拥有未来买入或卖出的权利。

F00 不深入定价，后续交给 F04。

### 2.6 工程痛点

很多错误来自“字段看起来像价格，就直接拿来算收益率”。

例如：

- 股票 `adjusted_close` 适合长期收益率。
- 债券 yield 不是债券价格。
- 宏观利率 series 不是可交易资产价格。
- 财报 revenue 是季度/年度披露，不是日频市场数据。

跨资产比较前还要问“现金流去了哪里”：

| 资产 | 仅看价格会漏掉什么 | 第一轮记录字段 |
|---|---|---|
| 股票 | dividend、split 等公司行动 | raw/adjusted 字段和调整说明 |
| 债券 | coupon、本金、应计利息 | clean/dirty price、cash-flow date |
| 基金/ETF | distribution、费用和 NAV/市场价差异 | NAV/price、distribution、fee basis |
| 期货 | 到期、保证金损益和换月 | contract、expiry、roll rule、adjustment |
| 期权 | premium、行权/到期结算 | strike、expiry、option type、settlement |

因此不能把不同资产的某个“价格”列统一 `pct_change()` 后就宣布可比。连续期货序列尤其
必须保存换月和回溯调整规则；回溯调整后的图形连续，不等于真实可交易组合没有换月损益。

### 2.7 字段判断流程

拿到一个新字段时，不要直接进 pandas。先做五个判断：

```text
1. 这是市场价格、宏观序列、财报字段，还是公告文本？
2. 它的时间频率是什么？
3. 它是原始值、复权值、比例，还是年化值？
4. 它能不能和当前任务的其他字段对齐？
5. 它的来源和限制能不能写进实验记录？
```

一个很小的例子：

| 字段 | 不能直接做什么 | 可以先做什么 |
|---|---|---|
| `close` | 直接当长期收益率基准 | 查是否存在 `adjusted_close` |
| `DGS10` | 当股票价格 | 作为 10 年期国债收益率序列记录 |
| `revenue` | 当日频交易信号 | 作为财报字段进入 M12/RAG 或低频分析 |

这一步看起来慢，但它是在保护后面的 F02、F03 和 F06。字段错了，代码越漂亮，错误越隐蔽。

## 第 3 章：价格、收益率和风险

### 3.1 本章解决什么问题

价格是某个时点的水平，收益率是两个时点之间的相对变化。风险描述收益或价值的不确定性。

### 3.2 金融直觉

两个资产都涨 1 元，不代表表现一样。100 元涨到 101 元是 1%，10 元涨到 11 元是 10%。

所以 F02/F03 通常用收益率而不是价格差。

### 3.3 最小公式

简单收益率：

```text
r_t = P_t / P_{t-1} - 1
```

若区间内还有可归属现金流 `C_t`，最小 total return 口径是：

```text
total_return_t = (P_t - P_{t-1} + C_t) / P_{t-1}
```

`adjusted_close` 常用于把部分公司行动和分配折回价格序列，但各数据商的调整范围不同。
字段名不能替代数据商定义；必须记录 vendor、endpoint、adjustment 和下载时间。

波动率：

```text
volatility = std(returns)
```

最大回撤：

```text
drawdown = wealth / peak - 1
```

### 3.4 最小代码

第一轮不需要复杂金融库，先用最小列表理解“价格到收益率”的转换：

```python
prices = [100, 102, 101]
returns = []

for i in range(1, len(prices)):
    r = prices[i] / prices[i - 1] - 1
    returns.append(r)

print(returns)
```

这段代码背后的关键不是语法，而是口径：`prices` 到底是 `close` 还是 `adjusted_close`？日期是否连续？缺失值是否处理过？如果这些问题没有回答，`returns` 就只是一个看起来正确的数字列表。

## 第 4 章：利率、期限和信用风险

### 4.1 本章解决什么问题

金融工程不只看股票。利率和期限结构是债券、固定收益、折现和宏观分析的基础。

学完本章，你应该能在一个简化现金流例子中解释并计算：

1. 为什么未来现金流需要折现。
2. 为什么要求收益率上升时固定现金流现值下降。
3. 为什么到期时间更长时，价格通常对同样的收益率变化更敏感。
4. 为什么信用利差是风险补偿的一部分，而不是已经实现的违约概率。

### 4.2 金融直觉

利率可以理解为资金的时间价格。未来 100 元今天值多少，取决于折现率。

债券价格和利率通常反向变化：利率上升，已有固定票息债券吸引力下降，价格可能下降。

先做一个预测，不要立即计算：同样承诺一年后支付 100 元，市场要求收益率从 5% 上升到
6% 后，今天愿意支付的价格应该上升还是下降？如果现金流改为五年后才支付，价格变化会
更小还是更大？把判断写下来，再看下面的 worked example。

### 4.3 Worked example：一笔固定现金流为什么会随利率变化

第一轮使用零息现金流隔离最核心的折现关系：

```text
PV = CF / (1 + y)^T
```

其中 `CF` 是未来现金流，`y` 是每年的要求收益率，`T` 是年数。假设一年后确定支付 100：

```text
y = 5%: PV = 100 / 1.05 = 95.238095
y = 6%: PV = 100 / 1.06 = 94.339623
```

要求收益率提高后，现值下降 `0.898472`。原因不是未来现金流从 100 变了，而是同样的未来
现金流现在用更高的机会成本折现。

再把期限改成五年：

```text
y = 5%: PV = 100 / 1.05^5 = 78.352617
y = 6%: PV = 100 / 1.06^5 = 74.725817
```

五年现金流下降 `3.626800`，比一年现金流的变化更大。这个例子只证明在固定现金流、单一
年复利收益率和零息结构下的期限效应；完整票息债券、久期、凸性和收益率曲线留给 F05。

### 4.4 信用风险：承诺支付不等于无条件收到

上面的计算把 100 当作确定现金流。公司债还要考虑发行人不能按约支付的风险。教学上可以
先把要求收益率拆成：

```text
required yield approximately
  risk-free benchmark
  + credit spread
  + liquidity and other compensation
```

例如，一年后都承诺支付 100：

```text
按 4% 折现的简化基准现值 = 100 / 1.04 = 96.153846
按 6% 折现的风险债券现值 = 100 / 1.06 = 94.339623
```

较高要求收益率对应较低现值，但这 2 个百分点不能直接解释为“违约概率等于 2%”。信用
利差还会混合流动性、风险厌恶、税收和市场供需等因素。F00 只建立边界，不做信用模型。

### 4.5 数据口径

FRED 中常见的是宏观和利率序列：

| 概念 | 含义 |
|---|---|
| `series` | 一条时间序列 |
| `observation` | 某个日期的观测值 |
| `release` | 数据发布来源 |
| `frequency` | 日、周、月、季等频率 |

同一个“利率”字段还必须说明：

```text
series_id
rate_type: yield / spot rate / policy rate / spread
unit: decimal / percent / basis points
maturity
frequency
observation_date
release_or_availability_time
```

`5`、`0.05` 和 `500 bp` 都可能表达 5%，但不能在代码中混用。一个十年期国债收益率也不能
在没有说明模型的情况下直接当作所有期限现金流的统一折现率。

### 4.6 可复现反例：只看 observation date

错误做法是把月度宏观数据按 `observation_date` 直接连接到当日市场价格，并假设当时已经
知道该数值。宏观统计通常在观察期结束后发布，有时还会修订。如果回测在发布日期之前使用
最终修订值，就引入了未来信息。

修复时至少区分：

```text
observation_date -> 数值描述哪个时期
release_time     -> 首次何时发布
availability_time -> 你的系统何时真正可见
vintage          -> 使用哪一次发布或修订版本
```

回归检查不是“日期列能 join”，而是断言每一行 `availability_time <= decision_time`。

### 4.7 独立练习

一张教学零息债券两年后支付 100。分别使用 4% 和 5% 年复利收益率：

1. 计算两个现值，保留 6 位小数。
2. 写出价格较低的是哪一种情形以及原因。
3. 再计算一年期同样现金流的价格变化，比较期限敏感性。
4. 说明为什么这个结果不能直接推广到带票息、可赎回或信用恶化的债券。

验收不直接给完整数值答案，但必须同时满足：两个现值都低于 100；5% 对应价格低于 4%；
两年期相对价格变化的绝对值大于一年期；公式、单位和限制说明均保存在实验记录中。

### 4.8 工程痛点

利率数据和股票价格数据常常频率不同。不能随便把月频宏观数据和日频股票收益率拼接到一起。

必须记录：

```text
series_id
frequency
release_source
observation_date
revision_note
```

- [ ] 能从折现机制解释价格与要求收益率的反向关系。
- [ ] 能用一组数值说明期限为何影响敏感性。
- [ ] 能区分信用利差和已经实现的违约概率。

- [ ] 能完成独立练习，但没有从 worked example 复制同一组输入。

### 4.11 本章依据

- [MIT 15.401 Present Value Relations](https://ocw.mit.edu/courses/15-401-finance-theory-i-fall-2008/pages/video-lectures-and-slides/present-value-relations/)
  支持固定现金流、折现率和期限的现值关系。
- [FINRA Bond Yield and Return](https://www.finra.org/investors/insights/bond-yield-return)

## 第 5 章：金融数据来源和数据口径

### 5.1 本章解决什么问题

后续所有实验都依赖数据来源。F00 要先让你知道常见公开数据入口分别适合什么。

本章不是网址清单。你要学会根据研究对象选择来源，记录“当时能看到什么”，并判断一个
数据集能否被复现、共享和用于给定结论。

### 5.2 常见来源

| 来源 | 适合数据 | 后续用途 |
|---|---|---|
| SEC EDGAR | 公司 filings、submissions、companyfacts | M12/M03 金融文档 RAG |
| FRED | 宏观、利率、经济时间序列 | F05/Q01 |
| 市场数据 API | 股票、ETF、指数价格 | F02/F03 |
| 公司公告/财报 PDF | 文本和表格 | M12/RAG |

### 5.3 Worked example：为一个问题选择来源

问题是：“比较某公司股票月收益与失业率变化”，至少需要两类输入：

| 输入 | 合适来源 | 不能省略的口径 |
|---|---|---|
| 股票价格 | 明确许可和字段定义的市场数据 API | symbol、交易日、复权口径、币种、下载时间 |
| 失业率 | FRED series/observations/release | series_id、观察期、发布日期、vintage、单位 |

不能从 SEC companyfacts 取得日频股价，也不能把 FRED observation 当作可交易价格。来源
“权威”只说明发布主体，不自动说明两个数据集的时间粒度、使用许可和信息可用时点兼容。

先写数据卡，再下载数据。一个最小记录可以是：

```yaml
dataset_id: unemployment_and_equity_demo_v1
decision_time: 2024-02-02T21:00:00Z
inputs:
  - source: FRED
    series_id: UNRATE
    observation_period: 2024-01
    release_time: recorded-from-release-calendar
    vintage: recorded
    unit: percent
  - source: market_data_provider
    symbol: TEST
    price_field: adjusted_close
    currency: USD
    downloaded_at: recorded
license_or_access_note: recorded
```

这里的 `recorded` 是学习者必须填写的待办，不是已经满足的事实。

### 5.4 数据字段说明模板

后续看到数据表，至少写：

```yaml
data_source:
source_url:
asset_type:
symbol_or_series_id:
date_range:
frequency:
fields:
adjusted_or_raw:
license_or_access_note:
point_in_time_note:
not_investment_advice: true
```

### 5.5 可复现反例：来源正确，时间仍然错误

假设一条一月份宏观 observation 在二月份才发布。错误代码按一月份 observation date 向前
填充到一月每个交易日，回测就会在发布前“知道”数值。这个错误不会产生异常，图表甚至会
很好看。

诊断时为每个样本打印：

```text
decision_time
observation_date
release_time
availability_time
vintage
```

修复后加入断言：

```python
assert (dataset["availability_time"] <= dataset["decision_time"]).all()
```

这段代码只是验收不变量；具体字段构造在 F02 的 point-in-time 章节完成。

### 5.6 工程痛点

数据来源不清会直接导致：

- 实验无法复现。
- 指标口径无法解释。
- 简历或项目表达没有证据。
- P03 task 无法追踪输入。

所以 F00 的“字段说明”不是文档装饰，而是工程能力的一部分。

### 5.7 从字段说明到实验记录

后续 GF 实验里，数据说明至少要能回答：

| 问题 | 为什么重要 |
|---|---|
| 数据是否自造、公开来源还是商业接口？ | 决定能否复现和分享 |
| 日期范围是什么？ | 决定样本是否足够、是否存在未来信息 |
| 字段是否复权、年化或百分数？ | 决定收益率、波动率、利率计算是否正确 |
| 缺失值怎么处理？ | 决定结果是否被清洗规则影响 |
| 结果不能说明什么？ | 防止把教学实验写成投资建议 |

你可以把这张表理解成金融工程版的“README”。代码能跑只是第一步，别人能看懂输入和边界，才算进入工程化。

### 5.8 独立练习：来源选择和数据卡

从下面两个问题中选一个：

1. “统计某公司 10-K 中风险因素章节的变化”，选择 SEC filing 来源。
2. “比较教学债券收益率与股票月收益”，选择 FRED 与一个可说明字段/许可的市场数据来源。

提交一份数据卡和一张时间线，至少包含 source URL、标识符、字段、单位、频率、观察期、
release/availability/download 时间、vintage、许可或访问限制、不能推出什么。再故意删除
`availability_time`，写出这会使哪类结论失去可复核性。

### 5.9 本章检查标准

- [ ] 能根据问题选择 SEC、FRED、市场数据或文档来源，而不是只凭域名权威性。
- [ ] 能区分 observation、release、availability、download 和 vintage。
- [ ] 能解释为什么来源正确仍可能发生未来信息泄漏。
- [ ] 能提交可复现数据卡，并明确许可、访问和结论边界。
- [ ] 能用时间不变量检查 point-in-time 数据，而不只检查 join 是否成功。

### 5.10 本章依据

- [SEC EDGAR APIs](https://www.sec.gov/search-filings/edgar-application-programming-interfaces)
  用于核对 submissions、companyfacts 和 filing 来源边界。
- [FRED API](https://fred.stlouisfed.org/docs/api/fred/) 与
  [FRED series observations](https://fred.stlouisfed.org/docs/api/fred/series_observations.html)
