## 本模块最值得理解的难点

F04 的难点不是公式本身，而是“模型价格”很容易被误读成真实市场价格或交易建议。

| 难点 | 为什么难 | 工程痛点 | 后续影响 |
|---|---|---|---|
| payoff 和 price 不同 | payoff 是到期结果，price 是今天估值 | 初学者容易混在一起 | GF04-01/GF04-02 会错位 |
| 参数是假设 | sigma、r、T 不是随便填 | 参数变一点，价格可能变很多 | Q03 必须记录参数 |
| Black-Scholes 有前提 | 欧式、无分红、连续交易、常数波动等 | 不写假设会误导 | 不能当真实报价 |
| Monte Carlo 有抽样误差 | seed、simulation_count 会影响估计 | 只给点估计无法判断误差尺度 | P03 要记录标准误、区间、runtime 和 seed |
| 数值任务有成本 | simulation_count 越大越慢 | 适合进入队列和监控 | M05/M08 连接 |
| 模型输出不是建议 | 价格只是模型估计 | 容易被包装成交易结论 | 必须有 disclaimer |

学习 F04 的核心习惯是：

```text
每个定价结果都必须和参数、模型假设、随机种子、模拟次数、标准误、区间、解析基线和非投资建议边界一起出现。
```

## 一条贯通学习线

F04 不按“先背公式”来学，而按“合约到 workload”来学：

```text
合约条款
-> 到期 payoff
-> 今天的模型价格
-> 参数和假设
-> 数值方法
-> 运行成本
-> pricing_task 记录
```

这条线能防止两个常见偏移：

- 只会背 Black-Scholes 公式，但说不清输入字段和假设。
- 只会跑 Monte Carlo 代码，但不记录随机种子、模拟次数和耗时。

第一轮你只需要把一个欧式期权讲清楚：它到期怎么给钱、今天怎么估值、模型为什么可能错、为什么它适合作为 P03 计算型任务。

## 可迁移的原则

1. payoff、price 和 profit 是三件事。payoff 是到期现金流，price 是模型估值，profit 还要考虑买入价格和交易成本。
2. 定价模型输出必须绑定假设。Black-Scholes 或 Monte Carlo 的结果离不开 `S/K/r/sigma/T`、随机种子、模拟次数和误差说明。
3. 数值实验天然适合进入 workload。模拟次数越大，耗时越长，失败和超时概率越高，因此它能自然连接 M05 调度和 M08 监控。

## 踩坑现场

> 你把 Monte Carlo 模拟次数从 1,000 提到 1,000,000，价格更稳定了，但运行时间暴涨。这个实验的重点不是“价格终于对了”，而是记录 `simulation_count`、`runtime_ms`、`random_seed` 和误差变化，理解为什么定价任务需要队列和监控。

## 第 1 章：衍生品和 payoff

### 1.1 本章解决什么问题

衍生品的价值来自某个基础资产。第一轮只理解三类：

| 工具 | 直觉 |
|---|---|
| forward | 约定未来按某价格买卖 |
| futures | 标准化、交易所交易的远期合约 |
| option | 拥有权利但不一定执行 |

F04 第一轮重点是 option，尤其是欧式 call/put。

### 1.2 基本术语

| 术语 | 含义 |
|---|---|
| underlying | 标的资产 |
| strike / K | 行权价 |
| maturity / T | 到期时间 |
| call | 买入标的的权利 |
| put | 卖出标的的权利 |
| volatility / sigma | 标的波动率参数 |
| risk-free rate / r | 无风险利率参数 |

### 1.3 payoff 公式

欧式 call 到期 payoff：

```text
max(S_T - K, 0)
```

欧式 put 到期 payoff：

```text
max(K - S_T, 0)
```

payoff 是到期时的结果，不是今天的价格。

一个手算例子：

| `S_T` | `K` | call payoff | put payoff |
|---:|---:|---:|---:|
| 80 | 100 | 0 | 20 |
| 100 | 100 | 0 | 0 |
| 120 | 100 | 20 | 0 |

这里的 `S_T` 是到期标的价格。表里的数字只说明“到期时权利值多少钱”，还没有回答“今天这份权利值多少钱”。今天的价格需要考虑时间、利率、波动率和不确定性。

### 1.4 最小代码

```python
def call_payoff(s_t, k):
    return max(s_t - k, 0)

def put_payoff(s_t, k):
    return max(k - s_t, 0)
```

### 1.5 工程痛点

payoff 图是理解期权的入口。不要一开始背复杂公式。先能回答：

- 标的价格高于 strike 时，call 为什么有价值？
- 标的价格低于 strike 时，put 为什么有价值？
- payoff 为什么不能直接当 price？

如果把 payoff 当 price，P03 里的 `pricing_task` 会出现严重口径错误：`result_json.price` 其实是某个到期场景下的 payoff，而不是模型估值。后续比较 Black-Scholes 和 Monte Carlo 时会完全错位。

### 1.6 检查标准

- [ ] 能解释 call/put payoff。
- [ ] 能画出 payoff 形状。
- [ ] 能说明 payoff 和 price 的区别。

## 第 2 章：Black-Scholes 输入输出

### 2.1 本章解决什么问题

Black-Scholes 是最经典的欧式期权定价模型之一。第一轮不推导公式，只理解输入、输出和假设。

### 2.2 输入参数

| 参数 | 含义 |
|---|---|
| `S` | 当前标的价格 |
| `K` | 行权价 |
| `r` | 无风险利率 |
| `q` | 连续分红率；本教材固定为 0 |
| `sigma` | 波动率 |
| `T` | 到期时间，通常按年 |
| `option_type` | call 或 put |

这些参数的教学读法是：

| 参数 | 初学者应该怎么理解 | 常见口径错误 |
|---|---|---|
| `S` | 今天标的价格 | 用了不同日期的价格却没记录 |
| `K` | 合约约定的行权价 | 和 `S` 混淆 |
| `r` | 连续复利的年化无风险利率参数 | 把 3% 写成 3；或误写成真实预期收益 |
| `q` | 连续分红率 | 当前公式必须记录 `q=0`，不能直接套到分红标的 |
| `sigma` | 标的波动率参数 | 把 20% 写成 20；或者不说明年化 |
| `T` | 距到期的时间，通常按年 | 把 30 天写成 30 而不是约 30/365 |

### 2.3 主要假设

第一轮至少记住：

- 欧式期权。
- 无分红，即连续分红率 `q=0`。
- 在风险中性测度下，标的价格按几何布朗运动建模。
- 波动率和无风险利率为常数。
- `r` 使用连续复利，折现因子为 `exp(-rT)`。
- 忽略交易成本。

这些假设不是装饰。它们决定模型价格的适用边界。这里的 `q=0` 不是“忽略一个不重要字段”，而是解析公式和 Monte Carlo 路径必须共同遵守的合同；有连续分红率时漂移和公式都要改成包含 `r-q` 的版本。

### 2.4 最小代码

```python
from math import erf, exp, isfinite, log, sqrt

def norm_cdf(x):
    return 0.5 * (1 + erf(x / sqrt(2)))


def validate_option_inputs(S, K, r, sigma, T, option_type):
    if not all(isfinite(value) for value in (S, K, r, sigma, T)):
        raise ValueError("option inputs must be finite")
    if S <= 0 or K <= 0:
        raise ValueError("S and K must be positive")
    if sigma <= 0 or T <= 0:
        raise ValueError("sigma and T must be positive")
    if option_type not in {"call", "put"}:
        raise ValueError("option_type must be call or put")


def black_scholes_price(S, K, r, sigma, T, option_type="call"):
    validate_option_inputs(S, K, r, sigma, T, option_type)
    d1 = (log(S / K) + (r + 0.5 * sigma ** 2) * T) / (sigma * sqrt(T))
    d2 = d1 - sigma * sqrt(T)

    if option_type == "call":
        return S * norm_cdf(d1) - K * exp(-r * T) * norm_cdf(d2)
    return K * exp(-r * T) * norm_cdf(-d2) - S * norm_cdf(-d1)
```

这段代码故意只实现 `q=0`。`r` 可以是有限的负数；当前边界要求 `S/K/sigma/T` 为正且所有输入有限。`T=0` 时应直接回到第 1 章的到期 payoff，本函数选择显式拒绝，避免在 `sqrt(T)` 和 `sigma * sqrt(T)` 处分母为 0。

### 2.5 解析价作为教学 oracle

使用固定 fixture：

```text
S=100, K=100, r=0.03, q=0, sigma=0.20, T=1
continuous compounding, European option
```

Python 3.13 双精度计算的已知答案为：

| 输出 | 已知答案 |
|---|---:|
| call price | 9.413403383853 |
| put price | 6.457956738704 |

还可以用无分红欧式期权的 put-call parity 做第二条独立检查：

```text
call - put = S - K * exp(-rT)
```

这两个解析价是当前同模型 Monte Carlo 的教学 oracle，即用于检测实现和抽样误差，不是市场真价、报价或投资信号。

### 2.6 工程痛点

这个函数很短，但真实记录不能只保存 price。必须保存：

```text
S
K
r
q: 0
r_compounding: continuous
sigma
T
option_type
model_assumptions
oracle_method: Black-Scholes under the same assumptions
```

否则几周后你不知道这个价格是怎么来的。

### 2.7 最小观察任务

先固定 `S=100, K=100, r=0.03, T=1`，只改变 `sigma`：

```text
sigma = 0.15
sigma = 0.20
sigma = 0.25
```

你要观察的不是“哪个价格更适合买”，而是：

- 波动率参数变大时，模型价格如何变化？
- 这个变化是否符合“期权是权利，不是义务”的直觉？
- 记录里是否写清 `sigma` 是教学假设，而不是市场隐含波动率校准结果？

### 2.8 常见错误

- `T` 的单位不清楚。
- `sigma` 用百分数 20 而不是 0.20。
- 不说明 call 还是 put。
- 没有记录 `q=0` 和连续复利口径。
- 把风险中性参数 `r` 写成标的真实预期收益。
- 不写模型假设。
- 把模型价格写成真实报价。

## 第 3 章：Monte Carlo 定价

### 3.1 本章解决什么问题

Monte Carlo 定价用随机样本估计同一模型下的折现 payoff 期望。它非常适合连接 P03，因为估计误差和计算成本都可以记录：

```text
simulation_count 越大
-> standard_error 通常约按 1 / sqrt(N) 缩小
-> runtime_ms 更高
-> 更像 workload
```

### 3.2 风险中性定价不是市场预测

本教材在风险中性测度下模拟无分红标的：

```text
S_T = S * exp((r - 0.5 * sigma^2) * T + sigma * sqrt(T) * Z)
Z ~ standard normal
```

这里漂移使用 `r`，是为了在当前模型假设下做无套利定价。它不表示：

- 真实世界里标的预期收益等于无风险利率。
- 模拟路径是对未来市场价格的概率预测。
- 落在置信区间里的值就是未来报价范围。

风险中性路径只是计算模型价格的工具。真实世界预测需要另一套概率测度、参数估计和验证问题，不在 F04 第一轮范围内。

### 3.3 估计量、标准误和近似区间

我们模拟很多个到期价格 `S_T`，计算每条路径的 payoff，再折现取平均：

```text
X_i = exp(-rT) * payoff_i
monte_carlo_price = average(X_i)
standard_error = sample_std(X_i) / sqrt(N)
approx_95_ci = price +/- 1.96 * standard_error
```

可以把它想成四个动作：

| 动作 | 含义 | 工程记录 |
|---|---|---|
| 生成样本 | 在风险中性模型下模拟 `S_T` | `random_seed`、`simulation_count`、`measure` |
| 计算 payoff | 每条路径按 call/put 公式算到期现金流 | `option_type`、`K` |
| 折现平均 | 把 payoff 折回今天并取均值 | `r`、`T`、`price` |
| 估计误差 | 用折现 payoff 的样本标准差计算标准误 | `standard_error`、`confidence_interval` |

`1.96` 区间是基于大样本正态近似的 Monte Carlo 误差区间，不是有限样本精确保证，更不是市场价格预测区间。不要为了让下界非负而静默截断区间；那会掩盖样本量不足或近似较差的问题。

### 3.4 最小代码

```python
import math
import random
import statistics
import time


def monte_carlo_option_price(
    S,
    K,
    r,
    sigma,
    T,
    option_type="call",
    simulation_count=10_000,
    seed=42,
):
    validate_option_inputs(S, K, r, sigma, T, option_type)
    if not isinstance(simulation_count, int) or simulation_count < 2:
        raise ValueError("simulation_count must be an integer >= 2")
    if not isinstance(seed, int):
        raise ValueError("seed must be an integer")

    rng = random.Random(seed)
    start = time.perf_counter()
    discounted_payoffs = []
    discount_factor = math.exp(-r * T)
    drift = (r - 0.5 * sigma ** 2) * T
    diffusion_scale = sigma * math.sqrt(T)

    for _ in range(simulation_count):
        z = rng.gauss(0, 1)
        s_t = S * math.exp(drift + diffusion_scale * z)
        if option_type == "call":
            payoff = max(s_t - K, 0.0)
        else:
            payoff = max(K - s_t, 0.0)
        discounted_payoffs.append(discount_factor * payoff)

    price = statistics.fmean(discounted_payoffs)
    standard_error = (
        statistics.stdev(discounted_payoffs) / math.sqrt(simulation_count)
    )
    confidence_interval_95 = (
        price - 1.96 * standard_error,
        price + 1.96 * standard_error,
    )
    runtime_ms = (time.perf_counter() - start) * 1000
    return {
        "price": price,
        "standard_error": standard_error,
        "confidence_interval_95": confidence_interval_95,
        "runtime_ms": runtime_ms,
        "simulation_count": simulation_count,
        "seed": seed,
    }
```

`random.Random(seed)` 创建局部 RNG，不会重置或消耗进程里其他代码使用的全局随机状态。相同实现、Python 版本、参数和 seed 用于复现；跨语言或更换 RNG 算法时，不应要求逐路径完全相同，而应比较解析 oracle、标准误和统计性质。

### 3.5 单次运行的已知答案

仍使用第 2 章 fixture，并设置 `simulation_count=10_000, seed=42`：

| 输出 | Python 3.13 确定性结果 |
|---|---:|
| Black-Scholes oracle | 9.413403384 |
| Monte Carlo price | 9.283616693 |
| standard error | 0.140315677 |
| approximate 95% CI | [9.008597967, 9.558635419] |

这一次区间包含解析价，只能说明该固定运行与声明的 Monte Carlo 误差尺度相容。一次“包含”不能证明 95% 覆盖率，一次“不包含”也不能单独证明实现错误；实现错误要结合已知答案、多 seed 和输入不变量判断。

### 3.6 多 seed 和 `1/sqrt(N)` 检查

只跑一个 seed 无法观察估计量的重复波动。用固定 `seeds=range(20)`，分别运行三个样本量，并对解析价计算跨 seed RMSE：

```python
oracle = black_scholes_price(100, 100, 0.03, 0.20, 1.0, "call")
seeds = range(20)

for simulation_count in (1_000, 10_000, 100_000):
    runs = [
        monte_carlo_option_price(
            100,
            100,
            0.03,
            0.20,
            1.0,
            "call",
            simulation_count,
            seed,
        )
        for seed in seeds
    ]
    mean_standard_error = statistics.fmean(
        run["standard_error"] for run in runs
    )
    rmse_to_oracle = math.sqrt(
        statistics.fmean(
            (run["price"] - oracle) ** 2 for run in runs
        )
    )
    print(simulation_count, mean_standard_error, rmse_to_oracle)
```

该固定 fixture 的结果为：

| `N` | 20 个 seed 的平均标准误 | 对解析价 RMSE | `mean_SE * sqrt(N)` |
|---:|---:|---:|---:|
| 1,000 | 0.449436 | 0.391258 | 14.2124 |
| 10,000 | 0.141146 | 0.126068 | 14.1146 |
| 100,000 | 0.044597 | 0.040766 | 14.1029 |

`N` 每增大 10 倍，平均标准误约缩小 `sqrt(10)` 倍，而 `SE * sqrt(N)` 大致稳定。这是 `1/sqrt(N)` 的教学观察，不代表任何一次随机结果会单调靠近解析价。

### 3.7 可判定练习

1. 对第 2 章 fixture，断言 call/put 解析价分别在已知答案的 `1e-9` 绝对误差内，并断言 put-call parity 残差小于 `1e-10`。
2. 复现 `simulation_count=10_000, seed=42` 的 Monte Carlo 点估计、标准误和区间；价格与表格绝对误差不超过 `1e-9`。若 Python/RNG 实现不同，则改为记录版本并执行第 3 项的统计验收。
3. 对 `seeds=range(20)` 重跑三个 `N`，确认相邻平均标准误之比位于 `[2.8, 3.5]`，并解释目标值为什么接近 `sqrt(10)`。
4. 把代码中的风险中性漂移 `r` 误写成任意真实收益假设，说明为什么即使代码能运行，也已经不再与 Black-Scholes oracle 属于同一教学模型。不得把这一步解释为预测准确率比较。

### 3.8 工程痛点

Monte Carlo 的输出必须带：

```text
seed
simulation_count
measure: risk_neutral
q: 0
r_compounding: continuous
price
standard_error
confidence_interval_method: normal_approximation_1.96
confidence_interval_95
oracle_method
oracle_price
error_to_oracle
runtime_ms
pricing_method
model_assumptions
```

如果不记录 seed，每次结果可能不同；如果不记录标准误，只增加 `simulation_count` 仍无法量化精度。`oracle_price` 只在当前欧式、`q=0`、同参数模型中可用，不能把解析价和市场报价互换。

这个实验连接 M05/M08：当 `simulation_count` 或重复 seed 数量变大，任务更可能排队、更需要超时限制，也更值得监控 `P95/P99` 和 worker utilization。

### 3.9 常见错误

- 调用 `random.seed()` 污染进程全局随机状态。
- 用很小 simulation_count 就做结论。
- 只返回 price，不返回 standard error 和区间方法。
- 把风险中性路径当成真实价格预测路径。
- 不记录 runtime。
- 把 Monte Carlo 价格和 Black-Scholes 价格差异写成市场套利。

## 第 4 章：参数敏感性和 Greeks 直觉

### 4.1 本章目标与前置

前置：已经完成第 2 章的 `black_scholes_price()`，并能说明当前模型固定 `q=0`、连续复利、
欧式期权和常数参数。

第一轮不深入 Greeks 推导，但需要知道定价结果会对参数敏感。

| Greek | 第一轮直觉 |
|---|---|
| delta | 标的价格变化时，期权价格大概怎么变 |
| gamma | delta 自身变化得多快 |
| vega | 波动率变化时，期权价格怎么变 |
| theta | 时间流逝对期权价格的影响 |
| rho | 利率变化对期权价格的影响 |

### 4.2 Worked example：一次只改变一个参数

<!-- textbook-code: role=fragment env=python-3.13 network=off -->
```python
base = black_scholes_price(100, 100, 0.03, 0.20, 1.0, "call")
higher_sigma = black_scholes_price(100, 100, 0.03, 0.25, 1.0, "call")
lower_sigma = black_scholes_price(100, 100, 0.03, 0.15, 1.0, "call")
shorter_T = black_scholes_price(100, 100, 0.03, 0.20, 0.5, "call")
```

该片段继承第 2 章函数，固定其他输入后得到：

| 场景 | `sigma` | `T` | call price | 相对基准的解释 |
|---|---:|---:|---:|---|
| 基准 | 0.20 | 1.0 | 9.413403 | 当前比较起点 |
| 提高波动率 | 0.25 | 1.0 | 11.348477 | 在同一模型内价格提高 |
| 降低波动率 | 0.15 | 1.0 | 7.485088 | 在同一模型内价格降低 |
| 缩短到期时间 | 0.20 | 0.5 | 6.371028 | 本 fixture 中剩余时间减少，call 价格降低 |

这里能观察的是有限参数变化，不是 Greek 的无穷小导数估计。要近似 vega、theta 等局部导数，
还必须声明冲击步长、差分方向和单位。

### 4.3 怎么读 Greeks

Greeks 第一轮只当“模型敏感性语言”，不要当交易策略。

| 问题 | 对应直觉 |
|---|---|
| 标的价格多涨一点，期权价格大概变多少？ | delta |
| 波动率参数多一点，期权价格怎么变？ | vega |
| 时间减少一点，价格怎么变？ | theta |
| 利率参数变化，价格怎么变？ | rho |

这类观察适合写进 `result_json.sensitivity`，但不能写成“模型预测市场会怎样”。

### 4.4 工程痛点

参数敏感性不能写成“预测”。它只是说明在模型假设下，输入参数变化会改变输出。

记录：

```text
base_parameters
changed_parameter
price_before
price_after
model_assumption_note
```

若提高 `sigma` 后价格反而下降，先检查是否误换了 `option_type`、参数位置或单位；修复后重跑
第 2 章解析价和 put-call parity，再恢复敏感性比较。不能用“市场就是这样”解释代码回归。

### 4.5 独立练习

使用不同 fixture `S=95, K=100, r=0.02, q=0, sigma=0.18, T=0.75`，一次只改变
`sigma=0.28` 或 `T=0.5`，填写：

| changed_parameter | before | after | price_before | price_after | explanation |
|---|---:|---:|---:|---:|---|
| sigma | 0.18 | 0.28 | 代码输出 | 代码输出 | 只说明模型敏感性 |
| T | 0.75 | 0.5 | 代码输出 | 代码输出 | 注意 T 的单位 |

### 4.6 验收标准

- 基准、提高波动率和缩短期限的价格应分别约为 `4.417280`、`7.685362`、`3.176090`，
  绝对误差不超过 `1e-6`。
- 每一行只能改变一个参数，记录中必须保留完整基准参数和 `q=0`/连续复利假设。
- 结论只能写“当前模型与 fixture 下的敏感性”，不得写成真实报价、未来预测或交易策略。


### 5.4 和 M05/M08 的连接

Monte Carlo 定价任务适合做 workload，因为它可以改变：

```text
simulation_count
-> runtime_ms
-> queue_wait_ms
-> worker utilization
```

这能连接 M05 调度和 M08 监控。

更完整的任务链路是：

```text
POST pricing_task
-> queued
-> worker 读取参数、simulation_count 和 seed
-> running
-> 写入 price、standard_error、CI、oracle error、runtime_ms、model_assumptions
-> succeeded 或 failed
-> M08 汇总 P95/P99、error_rate、worker_utilization
```

如果参数非法，例如 `sigma <= 0` 或 `T <= 0`，不要让 worker 静默返回空结果，而要写入 `failed` 和明确的 `error_type`。
